# briefing — 道場中間発表スライド

AWS spec駆動手法の第1フェーズ。聞き手・伝えたいこと・どうなってほしいかを定義する。talk-draft を下敷きに再定義。

## 聞き手は誰か

- 道場生（同じく個人開発・AIコーディングを実践中）とメンターエンジニア。
- 技術リテラシーは高い。宣伝・完成自慢には冷める。**実践者どうしの正直な共有**を期待している。
- 関心：AIをどう使って詰まりを越えたか、どこで失敗したか、何が再現可能な知見か。

## 何を伝えたいか（1メッセージ）

**「サブスク見直しくらい作れる」と甘く見ていた。判定方式の考慮漏れと実現可能性の見積もりの甘さで何度も壁にぶつかったが、AIに調べさせ・反論させ・設計を助けてもらって越えてきた——という1ヶ月（折り返し）の途中経過。**

補助メッセージ:
- AIは壁を越える相棒。ただし**決める・確かめる・責任を持つのは自分**。
- 完成報告ではなく**中間報告**。詰まった所も正直に出す。

## 聞き手にどうなってほしいか

- 「自分も“軽く見ていた前提”で壁にぶつかっている／ぶつかりうる」と自分ごと化する。
- AIの使い方として「飛びつく前に調べさせ・反論させ・方式を設計させる」を持ち帰る。
- SubBuddy の独自地点（利用量を根拠に見直しを提案、ローカルファースト）に納得する。
- 宣伝と受け取らず、**実践者の共有**として信頼する。

## 制約・トーン

- 10分厳守（質疑除く）。9枚・合計約9分40秒。山場は壁①（S6）・壁②（S8）。
- トーン：静謐・誠実・中立。煽らない・大げさにしない（[[web-ui-design-direction]] と一致）。
- 内部項番（P1等）を表に出さない（[[no-internal-codes-in-docs]]）。内容語で表現。
- PII・実データを載せない。画面例は合成データ。
- しきい値・定義は実装値を正とする（7日・月¥2,000・12ヶ月、判定5種、型6つ＋未マッチは継続）。

## 各スライドの「自分が甘かった→AIに助けられた→越えた」の一貫構造

山場（S6・S8）はこの順で語る。聞き手の自分ごと化と、AI活用の再現性が同時に立つ。
