## S2 SubBuddyとは（図版向き：3点の関係図）

```
自動更新に流されないための、手元で動く「見直しの相棒」

解決したい課題: 契約中のサブスクを把握できているか／使ってないのに払い続けていないか
やること: 料金・更新日・使い方の性質から「継続／様子見／プラン見直し／解約検討」を根拠つきで提案
こだわり1: 判断するのは人、アプリは材料を出すだけ（自動解約しない・煽らない）
こだわり2: 機微な支出情報は手元(Mac)に置く（ローカルファースト）
```

## S4 iPhone→SubBuddy 連携（図版向き：2経路フロー）★中核

```
iPhoneの利用データを「集計値だけ」MacのSubBuddyへ送る2つの方法

1 Screen Time(DeviceActivity): 対象アプリを利用 → しきい値イベント → 拡張 → App Group → 本体アプリ
2 Shortcuts(iPhone標準): アプリを開く → オートメーション発火
両経路は合流 → HTTP POST(集計値だけ＋トークン認証) → Mac: SubBuddy → Web画面に反映

1は精密だが専用の仕組みが要る。
2は手軽だが粗い
```

## S6 壁① 判定方式の考慮漏れ（図版向き：左→右の転換フロー）★中核

```
「どのサブスクを見直すべきか」の判定を、軽く考えすぎていた

当初の想定: 利用時間を測れば、不要なサブスクを特定できると考えていた
AIによる検証で判明した穴: 利用時間で解約を判断する前例がない／iPhoneはどのアプリを使ったかを公開しないため測れない／測れるのは画面を操作した時間だけで、バックグラウンドでの利用を取りこぼす
見直した方針: 利用形態ごとに判定を切り替える方式へ
```

## S7-a 「利用」の定義（図版向き：3分類）

```
「利用」の定義 ＝ ここが壁①の肝

能動利用: 前面で操作／裏で再生(音楽)／別端末で使う(PC・TV)
受動利用: 保管・同期・常時動いているだけ(バックアップ等)
権利保有: 特典として持つだけ(プライム等)／容量で見るもの: iCloud+ 等
→「非アクティブ」と判定してよいのは能動利用(前面・背景)だけ。受動・権利保有には言わない
```

---

## S7-b 判定の6つの型（図版向き：型→判定の対応表）

```
点数ではなく、状況の「型」に当てはめて理由を出す（判定は5種：継続／様子見／ダウングレード検討／解約検討／強い解約候補）

非アクティブ(能動利用なのに) → 様子見（長期間ずっと非アクティブなら強い解約候補）
同カテゴリに複数あって割高 → 解約検討
同じサービスに安い有料プランがある → ダウングレード検討
同カテゴリに安い有料競合がある → 解約検討（理由：安い競合がある）
年額更新が近い(7日以内) → 様子見（更新前に見直し）
高額で長く継続(月¥2,000以上・12ヶ月以上) → 様子見（確認をうながす）
どれにも当てはまらない → 継続
```
