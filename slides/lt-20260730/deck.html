The key finding

要望は、技術を決める材料になった

ユーザー目線の要望（要約）Codexとの対話で具体化
iPhoneだけで使いたい→iOSを主製品にする構成、SwiftUI
他の人にも試してもらいたい→TestFlight、クラウド、認証、データ分離
勝手に解約を決めてほしくない→判断材料と理由を示し、断定しない画面
個人データを集めすぎたくない→ローカル優先、集計値だけ同期、削除設計

要望は技術仕様ではない。
でも、技術仕様を決める材料になる。

3:30–5:10

ここが今日いちばん伝えたい点です。自分が作り方を指定できなくても、何を実現したいか、何を避けたいかは技術判断の材料になります。表示した言葉は会話の要約で、原文引用ではありません。

A useful setback

「動く」と「使える」は違った

最初：技術検証として動く

  • API URLを入力
  • 契約IDを入力
  • 計測・同期を手動操作
  • 正常な場合だけ確認
→

見直し後：利用者が使える

  • 接続先を意識しない
  • 契約を画面から選ぶ
  • 必要な処理は自動化
  • 拒否・失敗・再試行も案内

「正しく動くか」は検証できる。
「使いたいか・迷わないか」は、ユーザーとして判断する。

5:10–6:40

最初のiOS版は動きましたが、開発者向けの道具でした。「iPhoneだけで迷わず使えるか」を基準に、画面構成、拒否、通信失敗、再試行まで設計し直しました。

8:00–9:20

技術仕様を準備し切らなくても、この4点から始められます。最初から正解を言う必要もありません。「これは違う」「ここは不安」という返答も重要な入力です。

Takeaway

作り方はCodexと考える。
最後はユーザーとして判断する。

技術仕様が書けるまで待たず、小さな試作から始めてみる。
9:20–10:00

自分が作り方を全部決めたのではありません。欲しい体験と制約を伝え、Codexに技術判断へ具体化してもらい、その良し悪さを判断しました。まず自分の困りごとから、小さな試作を始めてみてください。
