# 投票結果

作成日: 2026-06-28

## 投票ルール

- 3チーム、各3名、合計9名で投票する想定。
- 各メンバーは、次の評価軸で最も良いと思う案へ1票を投じる。
- 自チームへの投票は許可する。LT本番に採用する案を選ぶため、忖度より発表としての強さを優先する。

## 評価軸

| 評価軸 | 見る点 |
|---|---|
| 10分適性 | 10分で一言のテーマが残るか |
| 具体性 | docs、apps/web、Prisma、iOS Spike の実態に根ざしているか |
| 聞き手価値 | 道場生・メンターが自分の開発に持ち帰れるか |
| SubBuddyらしさ | ローカルファースト、PII最小化、人が判断する思想が伝わるか |
| 誠実さ | できること・できないこと・リスクを隠していないか |

## 採用案

**チームA「自動更新に流されないための、ローカルな見直し相棒を作る」**を採用する。

## 採用理由

- 既存の `talk-draft-ai-coding-dojo.md`、`slides/spec/outline.md`、`slides/spec/slide-spec.md` と最も整合する。
- 「SubBuddyとは何か」から入り、判定方式とiPhone連携の2つの壁へ自然につながる。
- チームBの「AIを制御する」視点と、チームCの「信頼設計」視点を部分的に吸収できる。
- 完成自慢ではなく、詰めの甘さ・調査・反証・作り直しを正直に話せる。

## 採用案への統合メモ

チームAを土台にしつつ、次の要素を統合すると強くなる。

| 取り込む元 | 統合する要素 | 入れる場所 |
|---|---|---|
| チームB | AIに調査・反証・設計を助けてもらったが、判断は仕様に戻した | 壁1・壁2の台本 |
| チームB | Spikeを go/no-go ゲートとして扱った | iPhone連携の山場 |
| チームC | 自動解約しない、煽らない、判断は人に残す | SubBuddy紹介・まとめ |
| チームC | 詳細ログではなく集計値だけ送ることを「信頼設計」として語る | iPhone連携スライド |
