# チームA: ローカル判断支援班

## 役割

| 役割 | 担当方針 |
|---|---|
| スライド作成担当 | 1枚1メッセージで、図解と画面例を整理 |
| 台本作成担当 | 「甘い仮説 → 調査/反証 → 作り直し」の流れを10分に圧縮 |
| テーマ/タイムキーパー担当 | 10分厳守、山場を2つに絞る |

## 10分LTテーマ

**自動更新に流されないための、ローカルな見直し相棒を作る**

副題: **サブスク管理を“台帳”で終わらせず、根拠つきの判断支援にするまで**

## 10分構成

| 時間 | 内容 |
|---:|---|
| 0:00-0:40 | SubBuddyとは何か。自動解約ではなく、判断材料を出すアプリ |
| 0:40-1:40 | 課題。サブスクは忘れる、重複する、更新日に流される |
| 1:40-2:50 | 方針。支出・利用情報はMacローカル、iPhoneは集計値センサー |
| 2:50-4:10 | 壁1。利用時間だけでは解約判断できない |
| 4:10-5:40 | 解決1。状況の型で、理由つき判定にする |
| 5:40-7:00 | 壁2。iPhone利用量の取得は簡単ではない |
| 7:00-8:10 | 解決2。Screen Time/Shortcutsから集計値だけ同期 |
| 8:10-9:10 | いま動くもの。Web、Prisma、API、レコメンド、iOS Spike |
| 9:10-10:00 | 学び。動くから役立つへ。判断権はユーザーに残す |

## スライド案

| # | 見出し | 内容 |
|---:|---|---|
| 1 | 自動更新に流されないための、ローカルな見直し相棒 | 表紙 |
| 2 | SubBuddyとは | 料金・更新日・使い方から、継続/様子見/プラン見直し/解約検討を根拠つきで出す |
| 3 | なぜ作るか | サブスクは少額に見えて積み上がる。忘れる、重複する、更新日に気づかない |
| 4 | 設計のこだわり | MacローカルでDB/API/Webを動かす。iPhoneから送るのは詳細ログではなく集計値のみ |
| 5 | 壁1: 利用時間だけでは判断できない | 音楽、容量、権利保有、別端末利用は「非アクティブ」と断定できない |
| 6 | 解決1: 状況の型で理由を出す | 未利用、重複、安いプラン、安い競合、更新間近、高額長期などを判定根拠にする |
| 7 | 壁2: iPhone利用量取得は難しい | Apple権限、実機検証、暗号化トークン、ローカルMacへの同期、認証が必要 |
| 8 | 解決2: 集計値だけ同期する | DeviceActivity/Shortcuts → API → Prisma upsert。詳細ログは送らない |
| 9 | まとめ | SubBuddyの本質は「解約させるアプリ」ではなく「後悔しない更新判断を支える道具」 |

## 台本方針

- 宣伝ではなく、実践共有として話す。
- 山場は2つに絞る。
- 1つ目は「判定ロジックの甘さ」、2つ目は「iPhone連携の実現性」。
- 失敗談は短く、必ず「どう設計に反映したか」まで言う。
- 実データは出さず、画面例は合成データ前提。
- 内部コード名や実装記号ではなく、「非アクティブ」「重複」「更新間近」など日本語の理由で話す。

## 強み

- 聞き手が自分ごと化しやすい。サブスクの放置は身近。
- 技術の見せ場がある。Next.js、Prisma、Zod、ルール判定、iOS Screen Time連携。
- プライバシー方針が明確。支出・利用情報をクラウド前提にしない。
- 「自動化しすぎない」という思想が差別化になる。
- 動くWeb MVPとiOS Spikeがあり、机上案で終わらない。

## 弱点/リスク

- 10分では詰め込みすぎになりやすい。
- iOS Screen Timeの話は深掘りすると技術説明に寄りすぎる。
- 「利用量で判断」から「見直し支援」へ軸が変わった経緯を短く説明しないと、価値がぼやける。
- 手動登録の入力負荷はプロダクト上の大きなリスク。
- 商用価値はまだ検証途中。完成報告ではなく中間報告として出すべき。

## 投票時の評価軸

- 10分で一言のテーマが残るか。
- リポジトリ固有の設計・コードに根拠があるか。
- 技術とプロダクト価値の両方が伝わるか。
- 弱点を隠さず、次の打ち手まで示せているか。
- 聞き手が「自分の開発にも使える」と思える学びがあるか。
