## 1. この作業で何をするか

iCloud+（利用の性質＝`capacity`／容量ベース）に対するレコメンド判定方針を確定し、実装可能な形に落とす。

中心となる決定（2者議論で収束済み）：

- **使用容量（＝今どれだけ使っているか。例：200GBプランで38GB使用）は、全サブスク共通の必須情報ではない。**
- ただし iCloud+ の「**安いプランがある（＝下位プランへのダウングレード）**」判定（既存エンジンの P3）を**安全に**出すための**ゲート条件（＝安全確認の関門）**として必要。
- **新しい判定パターンは作らない。** 既存の「安いプランがある」判定に、iCloud+ のときだけ容量チェックという安全弁を足す。
- 容量入力は**オンボーディング必須にしない**。下位プラン候補があり、ユーザーの動機が立った瞬間（＝just-in-time）に取得を促す。

---

## 2. 判定の振る舞い（容量の有無・鮮度で出し分け）

iCloud+ で下位プラン候補がある場合、使用容量データの状態によって提示を3段階で出し分ける。

| 容量データの状態 | 提示 | 強さ |
|---|---|---|
| **なし** | 「もっと安い下位プランがあります。**使用容量を確認すると、下げられるか判定できます**」 | 断定しない発見プロンプト（要確認） |
| **あり＋鮮度OK＋安全に収まる** | 「現在の使用容量なら**下位プランで足ります**。月額／年換算で◯円の見直し余地」 | ダウングレード提案（断定） |
| **あり＋鮮度OK＋収まらない** | ダウングレードは提案しない | 提案なし |
| **あり＋古い（鮮度切れ）** | 「前回確認時点では足りそうでした。**今の使用容量を再確認しましょう**」 | 弱める（要再確認） |

「安全に収まる」の判定式（しきい値は設定外出し）：

```
使用容量 + 安全バッファ ≤ 下位プラン容量
安全バッファ = max(固定GB, 下位プラン容量 × 一定割合)
```

---

## 3. 正確性のルール（誤った断定をしない）

判定・文言で必ず守る（調査で確定した罠）。

| ルール | 理由 |
|---|---|
| **容量から価格を一意に決めつけない** | 値上げ前後・家族共有の割り勘・Apple One 内包・既存契約で実効額が変わる。プラン名だけで月額を断定しない |
| **年額は「月額×12の目安」と明示** | iCloud+ は月額課金のみ。実在の年額プランのように見せない |
| **「データが消える」と断定しない** | 公式は「同期・バックアップ・新規保存が止まる可能性」。変更前に Apple 画面で確認、と案内する |
| **無料枠5GBは定数で保持し入力させない** | 誤入力源を断つ |

---

## 4. データの取得方式（前提・既決）

- **主：手動入力**（プラン容量＝選択式、使用容量＝数値）。
- **補助：スクリーンショット OCR**（候補→ユーザー確認→保存の3段、自動確定しない）。**本ステアリングの前提条件ではなく別スパイク**。
- **不採用：非正規スクレイピング**（資格情報保持・ToS/BAN・保守不能）。
- 自動取得できる正規 API は存在しない（調査で確定）。ゆえに判定品質は**信頼度（ユーザー確認済みか）・鮮度（いつの値か）・スコープ（個人か家族全体か）**に依存する。

---

## 5. 実装の優先順位（意義の議論より）

| 優先 | 観点 | 本ステアリングでの扱い |
|---|---|---|
| 最優先 | **重複クラウド費用の発見** — iCloud+ を Google One / Dropbox 等と横並びにする | 既存の「重複で割高／安い競合がある」判定にカタログ登録するだけ。容量不要。**最初に出荷** |
| 本命 | **ダウングレード可否の安全判断**（本ステアリングの中心） | 容量ゲートを実装 |
| 付随 | **「怖くて放置」を縮小可能な支出に変える** | 上記の文言・UX として乗せる（独立機能ではない） |
| 後回し | 値上げ・更新伴走／家族共有スコープ | 別スパイク。月額契約のため既存の更新判定は発火しない点に注意 |

## 6. ユーザーストーリー

| # | ストーリー |
|---|---|
| US-1 | ユーザーとして、iCloud+ を他のクラウドサービスと横並びで見られるので、二重に払っているクラウド支出に気づける |
| US-2 | ユーザーとして、使用容量を入れていなくても「安い下位プランがある」と気づけ、確認を促されるので、見直しのきっかけを得られる |
| US-3 | ユーザーとして、使用容量を入れると「下位プランで足りる／足りない」を安全に判定してもらえるので、安心してダウングレードを判断できる |
| US-4 | ユーザーとして、容量入力を最初に強制されず、必要になった場面で求められるので、登録の手間で離脱しない |
| US-5 | ユーザーとして、「データが消える」と脅されず、変更の安全条件（同期影響・次回請求から・確認方法）を示されるので、信頼できる |

## 7. 受け入れ条件

| # | 条件 | 検証方法 |
|---|---|---|
| AC-1 | iCloud+ が他クラウドと同カテゴリで「重複／安い競合」判定の対象になる | カタログに iCloud+ と Google One 等を登録し、重複・競合の根拠が出ることを確認 |
| AC-2 | 使用容量なしの iCloud+ では、ダウングレードを断定せず「容量確認を促す」提示になる | 容量未入力で断定提案が出ないことを確認 |
| AC-3 | 使用容量あり＋鮮度OK＋安全バッファ込みで収まる場合のみ、ダウングレード提案が出る | 収まる／収まらないの境界ケースで出し分けを確認 |
| AC-4 | 鮮度切れ（しきい値超過）の容量では、提案を弱め「再確認を促す」提示になる | `capacityCheckedAt` を古く設定し弱める挙動を確認 |
| AC-5 | しきい値（鮮度日数・安全バッファ）が設定で差し替えられる | config 差し替えのユニットテストで境界を検証 |
| AC-6 | 年額表示が「月額×12の目安」と明示される／「データが消える」と断定しない | 文言を確認 |
| AC-7 | 既存の判定パターン（能動利用の使っていない判定等）に影響しない | 既存テスト・E2E が全件パス |
| AC-8 | 新しい判定パターンを追加していない（既存判定への安全弁のみ） | 設計レビューで確認 |

---

## 8. 制約事項

| # | 制約 | 理由 |
|---|---|---|
| C-1 | 新しい判定パターンを追加しない。既存「安いプランがある」判定への安全ゲートとして実装する | 分類体系の維持・複雑化回避（2者議論で収束） |
| C-2 | 使用容量を全サブスク共通の必須入力にしない | 取得摩擦が高く、登録離脱を招く |
| C-3 | 容量ベース判定は能動利用の時間軸判定に干渉しない | 利用の性質ごとに判定方法は独立 |
| C-4 | しきい値（鮮度日数・安全バッファ）は設定外出しにする | 既存 `config/scoring.ts` 方針との整合 |
| C-5 | 実データ（実在の容量・契約）を扱わない。テストは合成データのみ | CLAUDE.md PII 方針 |
| C-6 | OCR 実装・家族共有スコープ・値上げアラートは本ステアリング外 | 別スパイク。スコープを絞る |

---

## 9. スコープの線引き

### 含まれるもの
- iCloud+ を含むクラウドカテゴリの「重複／安い競合」判定（カタログ登録）
- 使用容量・プラン容量・容量確認日時の任意フィールド追加
- iCloud+ プラン容量表と「1つ下のプラン」解決
- 「安いプランがある」判定への容量ゲート（断定／確認プロンプト／弱めるの出し分け）
- しきい値の設定外出し
- 容量ゲートのユニットテスト
- 容量入力 UI（just-in-time）と安全条件の文言
- docs（PRD 6.3・functional-design 判定ルール・glossary）の改訂

### 含まれないもの（別スパイク／ポストMVP）
- スクリーンショット OCR の実装
- 家族共有スコープ（個人／家族全体）の本格モデリング
- 値上げアラート（外部データ依存）
- 容量の時系列スナップショットテーブル（当面は任意フィールドで足りる。必要時に移行）
