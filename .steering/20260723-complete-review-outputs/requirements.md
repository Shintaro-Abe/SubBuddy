# 要求内容 — 見直し機能の全出力と安全な確認優先度

> 作成日: 2026-07-23
> 区分: フル
> 親タスク: `.steering/20260711-release-roadmap-rebaseline/` の見直し修復、確認優先度、重大情報エラー、情報鮮度

## 背景・目的

現行実装は、利用実績、1利用日あたり単価、判定根拠、注意点を保存してWebとiPhoneへ表示できる。一方で、Webには「継続」「解約検討」等の古い最終判断表現が残り、代替プラン・代替サービス・節約額・不足情報・選択肢はスナップショットへ構造化保存されていない。

また、カタログ料金は180日を超えると金額へ係数を掛けて使う実装になっている。これは「公式確認から90日を超えた情報を節約額や確認優先度の根拠に使わない」という確定済み方針と矛盾する。

本作業では、見直しを最終判断ではなく、利用者が自分で判断するための安全な情報として完成させる。

## 変更・追加する機能

### やること

- 利用者向け表示を次の4つの確認優先度へ統一する。
  - 今確認したい
  - 更新前に確認したい
  - 情報が不足している
  - 現時点では急いで確認する材料が少ない
- 事実、対象期間、根拠、分からないこと、注意点、選択肢を構造化して保存し、WebとiPhoneで同じ意味に表示する。
- 下位プラン、代替サービス、解約した場合の年間差額を、対象名、比較金額、計算条件、情報源、確認日とともに保存する。
- 料金・プラン・代替候補・公式導線は、確認日から90日以内で情報源がある場合だけ、節約額と確認優先度の根拠に使う。
- 契約登録・編集、利用量同期、容量更新後に、影響する見直し結果を安全に再計算する。再計算できない場合は古い結果を表示しない。
- 重大な見直し情報エラーを表示前に検査し、該当する見直し情報を停止する。
- 既存スナップショットを安全に扱う移行と、Web・iPhone・API・ドメイン・DBの回帰試験を行う。
- `docs/` とiPhone機能移植表を実装結果へ合わせて更新する。

### やらないこと

- SubBuddyが継続、変更、解約を最終決定する機能
- 自動解約、解約代行、外部サイトへの自動入力
- AIによる自然文生成
- 料金情報を外部サイトから自動収集するクローラー
- ユーザーが登録した契約金額をカタログ料金で上書きする処理
- 金額未入力、外貨、オフラインキャッシュ、通知の実装
- 見直し結果に対する利用者の判断記録

## ユーザーストーリー

- 利用者として、どの契約を先に確認するとよいか知りたい。継続や解約をアプリに決められず、自分で判断したいから。
- 利用者として、表示された差額の比較対象と計算条件を確認したい。誤った金額を信じたくないから。
- 利用者として、分からない情報や計測できない範囲を知りたい。利用記録がないことを未使用と誤解したくないから。
- 利用者として、古い料金や非公式な手続き先を最新・公式として案内されたくない。
- iPhone利用者として、小さい画面でも最初に要点を理解し、必要な時だけ詳細を読めるようにしたい。

## 受け入れ条件

- [ ] AC-1: WebとiPhoneの見直し一覧・詳細は4つの確認優先度だけを利用者向けに表示し、「継続」「解約候補」等を最終判断として表示しない。
- [ ] AC-2: 利用実績が0日という事実だけでは「今確認したい」または「更新前に確認したい」へ上がらず、計測外の可能性を不足情報として表示する。
- [ ] AC-3: 見直しスナップショットに、事実、根拠、注意点、不足情報、選択肢、確認優先度を構造化保存し、API・Web・iPhoneで同じ内容を扱える。
- [ ] AC-4: 下位プラン、代替サービス、解約の各選択肢は、比較対象名、現在月額、比較月額、年間差額、計算条件を混同せずに保存・表示する。
- [ ] AC-5: カタログ情報は情報源URLと確認日を持ち、90日を超えた情報または情報源のない情報を、確認優先度、候補名、比較金額、節約額の根拠に使わない。
- [ ] AC-6: カタログ同期や再配備だけで確認日が現在日に更新されず、ユーザーが登録した契約金額も変更されない。
- [ ] AC-7: 別ユーザー情報、登録事実との不一致、期限切れ情報、利用0日だけの優先度上昇、情報不足の隠蔽、誤った公式導線、節約額の取り違え、継続推奨への誤変換を合成データで検出し、該当する見直し表示を停止できる。
- [ ] AC-8: 利用者が登録したURLは「登録した手続きページ」、確認済みカタログ由来URLだけを「公式ページ」として区別し、http/https以外を開かない。
- [ ] AC-9: 契約登録・編集、利用量同期、容量更新後は影響する見直しを再計算し、失敗時または入力更新後の古いスナップショットを表示しない。
- [ ] AC-10: 新しい構造を持たない既存スナップショットは最終判断表示へ変換せず、「再計算が必要」と案内する。
- [ ] AC-11: APIは認証済み利用者本人の結果だけを返し、表示停止中の結果本文・金額・根拠をクライアントへ返さない。
- [ ] AC-12: iPhoneでは一覧に確認優先度、契約名、短い根拠だけを表示し、事実・不足情報・選択肢は詳細で段階的に表示する。Dynamic TypeとVoiceOverで意味と順序を理解できる。
- [ ] AC-13: Webの320px以上の表示とPC表示を維持し、見直し一覧・詳細にページ全体の横スクロールを発生させない。
- [ ] AC-14: ドメイン・永続化・API・Web・iPhoneの合成データ試験、lint、型チェック、buildを実施し、結果と未確認範囲を`.audit/test-status.md`へ記録する。
- [ ] AC-15: `docs/product-requirements.md`、`docs/functional-design.md`、`docs/architecture.md`、`docs/glossary.md`、iPhone機能移植表を実装結果と一致させる。

## 制約事項

- 実在の契約、支出、利用量、メールアドレス、認証情報、実データベースを参照しない。試験は合成データだけを使う。
- iPhoneから受け取る利用情報は日別集計値だけとし、詳細な利用ログを追加しない。
- ユーザーの契約金額を本人の事実として扱い、カタログから自動変更しない。
- 料金鮮度90日、容量鮮度、安全バッファ等のしきい値は設定へ集約する。
- 古い表示との互換性より、誤った見直し情報を表示しないことを優先する。
- 内部互換用の`decision`は当面維持できるが、利用者向けの文言・並び順・色へ直接使わない。
