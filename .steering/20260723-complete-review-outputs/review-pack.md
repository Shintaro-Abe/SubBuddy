## 今回の結論

- 既存の内部`decision`は互換用に残すが、利用者向け画面では使わない。
- 利用者向けには4つの`reviewPriority`だけを表示する。
- 利用0日だけでは優先度を上げない。
- 代替候補、各差額、不足情報、選択肢をスナップショットへ構造化保存する。
- カタログ情報は出典付き・確認後90日以内だけを比較に使う。古い料金を係数で補正する現行方式は廃止する。
- 保存時と読取時の二段階で重大情報エラーを検査し、失敗した結果本文はクライアントへ返さない。
- iPhoneの一覧は情報を増やしすぎず、詳細画面で段階的に表示する。

## 前提・未決事項

### 要ユーザー判断

- なし。

### 設計上の前提

- 4つの確認優先度、90日鮮度、重大情報エラー8種類は既存の承認済みロードマップとADRを継承する。
- 出典不明または期限切れの候補が一時的に表示されなくても、安全性を優先する。
- iPhoneの画面占有を避け、一覧では要点、詳細では全出力という段階表示にする。
- カタログの公式情報確認は実データやユーザー情報に触れず、公開されている公式情報だけを使う。
- 実装中に確認優先度の意味または表示停止条件を変更する必要が出た場合は再承認する。

## セルフレビュー結果

| 観点 | 指摘 | 対応 |
|---|---|---|
| プロダクト | 全出力を一覧へ載せるとiPhoneで見づらい | 一覧は優先度・契約名・短い根拠だけとし、詳細で段階表示する |
| UX/UI | 「急いで確認する材料が少ない」が継続推奨に見える可能性 | 固定説明で「継続推奨ではない」と示し、成功色やチェックマークを使わない |
| アーキテクチャ | JSONだけではクライアント間で型がずれる | Zod、`schemaVersion`、Swiftの明示型、契約試験を追加する |
| データベース | 非null列追加で既存行を誤って新仕様へ変換する危険 | 新列をnullableで追加し、旧行は再計算まで表示しない |
| セキュリティ | 表示停止してもAPIが本文を返すとクライアント不具合で露出する | サーバーで本文を除外し、`blockedItems`には契約IDと固定文だけを返す |
| プライバシー | エラー調査ログへ契約名・金額が入る危険 | 内部エラーコードだけを扱い、契約内容をログへ出さない |
| QA | 作成時検査だけでは、後から契約や料金が変わった時に古い結果が出る | `sourceSubscriptionUpdatedAt`と読取時90日検査を追加する |
| 運用 | 配備のたびに`verifiedAt`を現在へ更新すると鮮度が偽装される | 確認日はカタログ資材へ固定し、同期処理の現在日時既定値を廃止する |
| 安全性 | 期限切れ料金へ係数を掛けると、実在しない安い価格が作られる | 疑似価格補正を廃止し、候補と節約額から完全に除外する |
| 保守性 | 契約更新ごとに全件再計算すると200件時に遅い | 単件再計算を追加し、手動操作だけ全件再計算を維持する |

### 第二意見

- 手段: テックリード、QA、セキュリティ・プライバシー、UXの別ペルソナによるアドバーサリアル・セルフレビュー
- 主な反証:
  - 古い料金を値引き補正して使う現行方式は、信頼度を下げるのではなく誤った価格を新しく作っている。
  - 作成時に正しくても、契約編集後や91日目には表示が不正になるため、読取時検査が必要。
  - UIで隠すだけでは別ユーザー・不整合結果がAPIから漏れるため、サーバー側で本文を除外すべき。
  - 全出力をiPhone一覧へ載せると、中心価値より情報量が勝つ。
- 反映:
  - 期限切れ候補の完全除外、読取時検査、安全なAPI DTO、一覧と詳細の段階表示を設計へ追加した。
