## 0. このガイドの使い方

### 前提と方針

- **方針（2026-06-16）：まずローカルファーストで構築する。** 配布は後回し（段階7は机上完了・当面オフ）。
  そのため、ここで検証するのは「自分の iPhone で動かして、取り込む数字が信頼できるか」を確かめる段階3・4・5と、段階1・6 の残項目。
- 検証は Spike アプリ（`spikes/ios-screen-time/`）で行う。確認手段は主に次の3つ。
  1. **本体アプリの画面**（レコード数・読み取った内容の表示）
  2. **Console.app のログ**（拡張は `os.Logger` で出力。`print` は出ない）
  3. **Mac の DB**（`ios_usage_daily_summaries` テーブル）

### 進め方の推奨（並行で時間を縮める）

- **段階4の「三点照合（7日間）」は時間がかかるので、いちばん先に開始**し、毎日少しずつ記録する。
  その裏で段階1残・段階3・段階6残・段階5を順に進めると、全体の待ち時間が縮む。
- 1つでも MUST がはっきり失敗（no-go）したら、その時点で記録して立ち止まり、方針を相談する。

### 記録の扱い（PII 注意）

- 記録テンプレの**空欄の表はリポジトリに置いてよい**が、**実際の利用傾向（どのアプリを何分使ったか等）が入った結果や実機スクショは Git 管理しない**。
  必要ならマスキング・合成化する（`design.md` §6・PII 方針）。手元メモや別管理に置くのが安全。

## 段階1の残項目：受け渡しの堅牢性（MUST-1 の未検証ぶん）

> 段階1は「単発の受け渡し」までは実機で OK（暫定 go）。ここで残りの**多重到達・再起動耐性**を確認し、MUST-1 を確定する。

### 1-4. 複数同時到達で取りこぼさない

**目的（MUST-1）**：複数の閾値が短時間に到達しても、全件が App Group 経由で本体に読めること。

**手順**
1. 1つの対象アプリで複数閾値（15/30/60/120分）を登録した状態にする。
2. 対象アプリを連続使用し、短時間に複数閾値（例：15分→30分→…）を立て続けに到達させる。
   - 短時間で詰めにくい場合は、閾値を小さめに一時設定して再現しやすくしてよい（検証後に戻す）。
3. 到達のたびに Console.app で `reached its threshold` を確認。
4. 本体アプリで読み取り、**到達した全段階が欠けずに反映**されているか（最上位段階まで上がっているか）確認。

**合格基準**：短時間の複数到達でも、本体が全到達を取りこぼさず読める（最上位段階まで反映）。
**NG 例**：同時/連続到達のうち一部が欠ける、最上位まで上がらない。

**記録テンプレ**

| 試行 | 連続到達させた閾値 | Console で確認した到達 | 本体で読めた最上位段階 | 取りこぼし | 結果(OK/NG) |
|---|---|---|---|---|---|
| 1 | | | | 有/無 | |
| 2 | | | | 有/無 | |

### 1-5. 本体アプリ再起動後もデータが残る

**目的（MUST-1）**：本体を kill→再起動しても、App Group のデータが残っていること。

**手順**
1. 到達済みのデータがある状態にする（レコード数 ≥ 1）。
2. 本体アプリを**スワイプで kill** する。
3. 本体アプリを再起動し、読み取り → 前回のレコードが残っているか確認。

**合格基準**：再起動後も同じレコード（日付・段階）が読める。
**NG 例**：再起動でレコードが消える／件数が減る。

**記録テンプレ**

| kill 前のレコード数・最上位段階 | 再起動後のレコード数・最上位段階 | 一致 | 結果(OK/NG) |
|---|---|---|---|
| | | 一致/不一致 | |

### 1-6. 端末（iPhone）再起動後もデータが残る

**目的（MUST-1）**：iPhone 本体を再起動しても、App Group のデータが残っていること。

**手順**
1. 到達済みのデータがある状態にする。
2. iPhone を**電源 off→on で再起動**（初回はパスコード解除が要る＝Data Protection の都合）。
3. 本体アプリを起動し、読み取り → 前回のレコードが残っているか確認。

**合格基準**：端末再起動後も同じレコードが読める。
**NG 例**：端末再起動でレコードが消える。
**つまずきヒント**：端末再起動直後はロック解除前だと保護領域が読めないことがある。**一度パスコードで解除してから**アプリを開く。

**記録テンプレ**

| 再起動前のレコード数・最上位段階 | 解除→起動後のレコード数・最上位段階 | 一致 | 結果(OK/NG) |
|---|---|---|---|
| | | 一致/不一致 | |

> **段階1 の確定**：1-4／1-5／1-6 がすべて OK なら MUST-1 を「実機 go」で確定。1つでも NG なら no-go 候補として記録し相談。

## 段階3：復旧・許可失効（MUST-2・MUST-8）

### 3-1. iPhone 再起動後に監視を立て直せる

**目的（MUST-2）**：端末再起動の後、監視状態を確認し、必要なら再アーム（仕掛け直し）で復旧できること。当日値を壊さないこと。

**手順**
1. 監視中（startMonitoring 済み）の状態で、当日すでに到達データがある状態にする。
2. iPhone を再起動 → 解除 → 本体アプリを起動。
3. 監視が生きているか確認（その後の使用で到達通知が来るか）。来なければ**再アーム**（再度 startMonitoring）。
4. 再アーム後、**当日の既存データ（段階）が下がっていない**ことを確認。

**合格基準**：状態が分かり、再アームで監視が復旧し、当日値が壊れない（段階が下がらない・重複しない）。
**NG 例**：再アームのたびに当日値がリセット／同じ日が二重登録される。

**記録テンプレ**

| 再起動後の監視 | 再アーム要否 | 再アーム後の当日段階 | 当日値の保全 | 結果(OK/NG) |
|---|---|---|---|---|
| 生存/停止 | 要/不要 | | 保たれた/壊れた | |

### 3-2. アプリ kill では監視が止まらない（拡張は独立）

**目的（MUST-2 補強）**：本体を kill しても、裏の拡張は独立プロセスなので監視が続くこと。

**手順**
1. 監視中に本体アプリを kill。
2. 本体を起動せずに対象アプリを使い、閾値に到達させる。
3. Console.app で拡張の到達ログが出るか確認。その後で本体を起動し、データが入っているか確認。

**合格基準**：本体 kill 中でも拡張が到達を検知し、App Group にデータが残る。
**NG 例**：本体を閉じると到達が記録されない。

**記録テンプレ**

| 本体 kill 中の到達ログ | 本体起動後に読めたか | 結果(OK/NG) |
|---|---|---|
| 出た/出ない | 読めた/読めない | |

### 3-3. 許可の取り消し → 再許可 → 再対応づけ

**目的（MUST-8）**：設定で許可を取り消したのを検知し、再許可 → 対象を選び直し → 既存サブスクと結び直す導線が成立すること（取消後は選択値＝トークンが無効になる）。

**手順**
1. 監視中に、iPhone の設定（スクリーンタイム関連／このアプリの許可）から**許可を取り消す**。
2. 本体アプリを開き、**許可が失効したことを検知**できるか確認（認可状態の表示・エラー）。
3. アプリ内で**再許可**（再度 requestAuthorization）。
4. **対象を選び直し**（FamilyActivityPicker）、**既存サブスクと結び直す**（対応表を復元）。
5. 再び監視が機能するか（到達 → 読取）確認。

**合格基準**：取消を検知でき、再許可 → 再選択 → 再対応づけで監視と紐付けが復旧する。
**NG 例**：再許可しても既存サブスクの監視/対応が復元できない。

**記録テンプレ**

| 取消検知 | 再許可 | 再選択→再対応づけ | 復旧後の到達→読取 | 結果(OK/NG) |
|---|---|---|---|---|
| 可/不可 | 可/不可 | 可/不可 | 可/不可 | |

> **段階3 の確定**：3-1/3-2 で MUST-2、3-3 で MUST-8 を判定。結果と再現手順を記録。

## 段階4：測定精度・日付境界（MUST-3・MUST-4・MUST-5）

> ⚠️ **4-1 の三点照合は7日かかる**。**このガイドの最初に開始**し、他の段階と並行で毎日記録する。

### 4-1. 三点照合を7日間（MUST-3）

**目的（MUST-3）**：取り込む段階が実際の利用とどれだけ合うかを、3つの基準を突き合わせて確かめる。

**毎日やること**
1. 対象アプリを、その日狙った時間だけ使う（**Controlled use**：例 0／16／31／61／121 分のいずれか）。
2. 3つを記録：
   - **①手計測**：自分でストップウォッチ等で測った実利用分。
   - **②設定アプリの Screen Time 表示**：iPhone 設定 → スクリーンタイムのそのアプリの時間。
   - **③通知ログから推定した段階**：拡張の到達ログ／本体の読取段階（例 `m120_plus`）。
3. ①と③が**1段階以内**に収まっているか確認。境目（例：ちょうど閾値付近）でズレたら理由を書く。

**合格基準**：有効7日のうち**6日以上が1段階以内**。境目日のズレは理由を説明できる。初日・途中開始日は除外/partial 扱い。
**NG 例**：ズレが大きく、理由も説明できない。

**記録テンプレ（三点照合表）**

| 日 | 狙った分 | ①手計測(分) | ②Screen Time(分) | ③推定段階 | ①と③のズレ | 1段階以内? | 備考(境目/初日等) |
|---|---|---|---|---|---|---|---|
| 1 | | | | | | はい/いいえ | |
| 2 | | | | | | はい/いいえ | |
| 3 | | | | | | はい/いいえ | |
| 4 | | | | | | はい/いいえ | |
| 5 | | | | | | はい/いいえ | |
| 6 | | | | | | はい/いいえ | |
| 7 | | | | | | はい/いいえ | |

集計：有効日数＝＿／7、1段階以内＝＿日 → 判定(OK/NG)＿

### 4-3. 日跨ぎ前後で正しい日に入る（MUST-4）

**目的（MUST-4）**：23:50〜0:10 のような日跨ぎや、タイムゾーン変更でも、利用が iPhone 現地日付で正しい1日に入ること。

**手順**
1. **日跨ぎ**：夜 23:50 ごろから 0:10 ごろまで対象アプリを使う。到達した段階が**どの日付で記録されたか**を本体読取／DB で確認。
2. （任意）**タイムゾーン変更**：設定で TZ を変えて、同じ利用が別日にずれないか確認。

**合格基準**：日跨ぎ・TZ 変更でも、利用が iPhone 現地日付の正しい日に入る（別日にずれない）。
**NG 例**：同じ利用が別日に保存される／二重に入る。

**記録テンプレ**

| ケース | 使用した時間帯 | 記録された日付 | 期待した日付 | 一致 | 結果(OK/NG) |
|---|---|---|---|---|---|
| 日跨ぎ | 23:50〜0:10 | | | 一致/不一致 | |
| TZ変更(任意) | | | | 一致/不一致 | |

### 4-4. 二重通知・順不同でも段階が下がらない（MUST-5）

**目的（MUST-5）**：到達通知が二重／順不同で来ても、端末側は上がる方向のみ、DB 側は段階が下がらない・重複行が出ないこと。

**手順**
1. 同じ閾値の到達を意図的に二重に起こす／順不同で起こす（例：高い段階の後に低い段階の通知が来る状況を作る）。
2. 同じ内容の送信を複数回行う。
3. 本体の読取段階と、Mac DB の `(subscriptionId, usageDate)` 行を確認。

**合格基準**：端末側は単調更新（上がるだけ）、DB 側は段階が下がらず、重複行が出ない。
**NG 例**：段階が下がる／重複行ができる／過大計上。

**記録テンプレ**

| 起こした状況 | 端末側の段階推移 | DB の最終段階 | 重複行 | 結果(OK/NG) |
|---|---|---|---|---|
| 二重通知 | | | 有/無 | |
| 順不同 | | | 有/無 | |
| 同一再送 | | | 有/無 | |

### 4-5. 「ゼロの日」「未同期で欠けた日」「監視停止」を判別できる

**目的（MUST-5 関連／ゼロ日判別）**：閾値方式では0分の日は通知が来ない。そのため次の3状態を**技術的に区別**できるか確認する。
- **none（利用ゼロ）**：監視は生きているが、その日まったく使わなかった。
- **missing（未同期で欠け）**：iPhone がまだ送っていない／Mac に届いていない。
- **監視停止**：許可失効・再起動未復旧などで、そもそも測れていない。

**手順**
1. 対象を**まったく使わない日**を作る（none）。
2. **送信前**の状態を作る（missing）。
3. **許可失効/監視停止**の状態を作る（停止）。
4. それぞれを、本体アプリ／DB のどのシグナルで見分けられるか整理する（例：監視生存フラグの有無、最終同期時刻、当日レコードの有無）。

**合格基準**：3状態を見分ける方法が確定し、記録できる。
**NG 例**：「使ってないだけ」と「届いてないだけ」が区別できない（節約アプリでは致命的）。

**記録テンプレ**

| 状態 | 見分けるシグナル（何を見れば分かるか） | 本Spikeで判別可否 |
|---|---|---|
| none（利用ゼロ） | | 可/不可 |
| missing（未同期） | | 可/不可 |
| 監視停止 | | 可/不可 |

> **段階4 の確定**：4-1=MUST-3、4-3=MUST-4、4-4=MUST-5、4-5=ゼロ日判別。各結果を記録。

## 段階6の残項目：同期堅牢性（MUST-7）

> 段階6は冪等保存（6-4）と AC-3 着弾は実機で OK 済み。残りの**再送・Mac停止耐性**を確認し MUST-7 を確定する。

### 6-2. 不通 → 復旧で再送できる

**手順**
1. iPhone を**機内モード**等で通信不通にする。

3. 通信を復旧 → 本体アプリを開く（フォアグラウンド送信）→ 未送信が再送され DB に入るか確認。

**合格基準**：不通中に貯めた未送信が、復旧後に全件 DB に入る（冪等で重複なし）。
**NG 例**：欠損が気づかず貯まる／復旧しても送られない。

| 不通中に作ったデータ | 復旧後に DB に入った件数 | 重複 | 結果(OK/NG) |
|---|---|---|---|
| | | 有/無 | |

### 6-3. Mac 停止 → 再起動を挟んでも帳尻が合う

**手順**

2. Mac を復旧（`sudo service postgresql start` → `npx next dev --hostname 0.0.0.0`）。
3. iPhone 側で再送 → DB に全件入るか確認。

**合格基準**：Mac 停止中の送信失敗が、復旧後の再送で全件保存される（冪等）。
**NG 例**：Mac 停止中のデータが失われる。
