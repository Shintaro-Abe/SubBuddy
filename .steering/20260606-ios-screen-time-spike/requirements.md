## 1. 今回の作業の目的

これは **Spike（＝本格的に作り始める前に「技術的に本当に実現できるか」を小さく試す検証）** です。PRD（プロダクト要求定義）§10.3 が「**実装前に必ず実施**」と定めている、**iPhone の Screen Time（利用時間計測のしくみ。Apple の DeviceActivity / Family Controls という機能群で実現）と連携するための技術検証**にあたります。

この Spike の目的は「動く製品を作ること」ではなく、**実際の iPhone/Mac で試して「実現できそうか」を確かめ、本格実装に進むか（go）／いったん中止するか（no-go）を判断する材料を集めること**です。

- **P1（使っていない）パターンの判定手段**である「iPhone の利用量を DeviceActivity で自動取得し → Mac に毎日まとめて送る」しくみが、Apple の制約や実機の挙動のもとで**技術的に実現できるか**を見極める。レコメンドエンジンはパターン判定方式（P1〜P7）に転換済みで、**利用量は P1 の判定手段であり MVP の中核ではない**。P2〜P6 は利用量なしで機能するため、本 Spike が no-go でもレコメンド機能自体は成立する。対象は `usage_type = active_foreground`（iPhone 前面で操作するサブスク）に限定される。
- 採用予定の **アーキテクチャ（＝部品構成・つくりの全体像。`investigation.md` §4）の、P1 判定を支える技術的前提**を、最小の手間で先に確かめておく（成立しないなら早く気づけるように）。最優先の検証項目（後述の A1）が成立しなかった場合は、**P1 判定を見送り、P2〜P6 のみで運用する**判断を、データに基づいて下せるようにする。なお、Shortcuts 起動記録（`source: ios_shortcut`、実装済み）が DeviceActivity の部分的な代替として `active_background` サブスクの P1 判定に利用できる。
- 不確実な箇所（公式ドキュメントで答えが確認できなかった A1〜A4、開発者コミュニティで「変な動きをする」と報告されている挙動）を実機で確かめ、結果を `investigation.md` の検証欄／本ステアリングに記録する。
- **スコープの線引き（重要）**：本ステアリングは「**技術的に実現できるか（＝iPhone の数字を Mac まで運ぶしくみ）**」だけを扱う。「**どのサブスクに利用量判定を適用するか**」という適性問題は、パターン判定方式（`.steering/20260609-quantitative-recommendation-engine/`）で**解決済み**：`usage_type` フィールドで P1 の適用可否を制御し、`active_foreground`（前面操作）と `active_background`（背景再生）のみに適用する設計が確定・実装済みである。**本ステアリングの go は技術成立のみで判断**し、go 判定が出れば本格実装に進む。

> ⚠️ 実行環境の制約：このリポジトリは Linux コンテナ上で動いており、**Xcode でのビルド（＝Swift コードを iPhone アプリに組み立てる作業）や iPhone 実機での動作確認は、この環境では実行できません**。Spike の実機作業はユーザーが Mac/iPhone（有料の Apple Developer Program アカウント＋実機）で行います。Claude Code は設計・段取り・記録・判断材料の整理を担当し、`verify`/`run`（＝実アプリを起動して動作確認するスキル）は使いません。

## 2. スコープ（＝今回の作業の範囲）

### 2.1 今回やること（In Scope）

1. **合格ライン（go/no-go の判断基準）を先に決めて検証する（最優先）**：§4.1 の合格ライン（MUST 9項目／SHOULD）を、検証を始める前に固定する。最重要は **MUST-1 = A1「裏で動く計測プログラム（DeviceActivityMonitor 拡張）が集めた数字を、共有の置き場（App Group）経由で本体アプリが読み取れるか」が、実機で安定して動くか**。
   - DeviceActivityMonitor 拡張（＝アプリ本体とは別に、アプリを開いていなくても iOS が裏で動かす小さなプログラム）
   - App Group（＝本体アプリと拡張がデータを受け渡しできる共有の保管棚）
   - 検証では、保管棚の2方式（`UserDefaults(suiteName:)`＝簡易設定の保存／App Group 内のファイル）両方、原子的書込（＝途中で壊れず「全部書けるか／まったく書かないか」になる安全な書き込み）、Data Protection（＝iOS の保存データ暗号化）の状態、複数同時到達、再起動後を含める。
   - **A1 が no-go（成立しない）なら、以降の段階は止めて、中止判断の根拠を記録する。**

2. **最小縦割りの試作アプリ（＝入口から出口まで一本だけ通す、いちばん小さい試作）**：以下を一通り通すだけの最小構成。
   `計測の許可取り（認可 .individual）→ 対象アプリ選択（FamilyActivityPicker）→ 選んだ対象の名前確認（Label(token) で表示）→「○分使った」を判定する設定（DeviceActivityEvent）を1つ登録（startMonitoring）→ 到達通知（eventDidReachThreshold）をログ出力 → 共有置き場（App Group）から本体が読み取り → 本体画面に表示`
   - 認可 `.individual`（＝iOS に計測を許可させる設定。本人の端末を本人が許可する方式。iOS 16 以降）
   - FamilyActivityPicker（＝対象アプリ/サイトをユーザーが選ぶ、Apple 標準の選択画面）
   - Label(token)（＝中身=アプリ名を読めない値から、画面に名前だけを表示できるしくみ）
   - DeviceActivityEvent と「閾値（しきいち）」（＝「○分使ったら通知」という判定設定）／startMonitoring（＝計測の開始登録）／eventDidReachThreshold（＝「○分に到達した」と知らせる通知）

3. **まだ答えが出ていない疑問を実機で確かめる（A2〜A4）**：
   - **A2**：iPhone やアプリを再起動しても計測が続くか／仕掛け直し（再アーム＝止まった計測をもう一度仕掛けること）が要るか。1日の計測の区切り（`intervalStart`/`intervalEnd`＝計測期間の開始/終了）の挙動。
   - **A3**：Apple に提出するプライバシー説明ファイル（`PrivacyInfo.xcprivacy`）に何を書く必要があるか、App Store 審査（＝Apple による公開前チェック）で通る用途説明の方向性（配布はポストMVP だが、方針は先に把握しておく）。
   - **A4**：対象選択画面・裏の計測プログラム・到達通知が、本物の iPhone でないと動かないのか、シミュレータ（＝PC 上の擬似 iPhone）でどこまで動くか。

4. **あらかじめ知られているおかしな挙動を実機で確かめる**：計測のしくみ（DAM＝DeviceActivity の監視機能）には約15分より細かく測れない下限がある。そのため、**「15分以上使った」の通知（15m_plus）が確実に鳴るか**（1分・5分の判定が無意味にならないか）、1日の終わりの区切り通知（`intervalEnd`）が鳴らないことがある問題、到達通知が二重に鳴る問題、計測開始前の使用を含めるか（`includesPastActivity`）と初日（計測を始めた当日）の扱いを実測する。あわせて、**「利用ゼロの日（none）」「未同期で欠けた日（missing）」「監視停止」を技術的に判別できるか**を確認する（しきい値方式では0分の日は通知が来ないため、別シグナルが要る）。※ここは「判別できるか」という技術的な確認まで。「ゼロだから解約候補にする/しない」という判定ルールは別ステアリング（適性）の担当。

5. **どこまで増やせるかの上限を試す（限界試験）**：「監視対象の数 × しきい値の数（`N activities × M events`）」をどこまで登録できるか（同時に監視できる数の上限は Apple 非公開、裏の計測プログラムには ~6MB 級というメモリ制限の報告あり）を実機で確認。MVP で安全に扱える**自動監視の対象数の上限**（暫定 3〜5 件）を実測で決める。

6. **送信経路の最小確認**：本体アプリが共有置き場（App Group）の数字を読み、既存の送信窓口 `POST /api/usage/daily` が求めるデータの形（`usageDailyBatchSchema`＝送るデータの形式の決まり）に合った JSON を送れることを確認する。同じ家庭内ネットワーク（LAN）での到達性と、通信相手が本物だと確かめる方式（証明書方式。QR コードで初回に安全に紐づける「QR ペアリング＋証明書ピン留め」案 vs 検証用の中継トンネル）を**比較して選ぶ**（実装は最小限）。

7. **判断材料を記録する**：各検証項目の結果（OK／NG／要追加検証）、go/no-go の判定、MVP の確定スコープ（単一アプリ・15分以上・iPhone の日付基準・アプリを開いた時に送る方式を主経路にする）を、本ステアリング／`investigation.md` に追記する。

### 2.2 今回やらないこと（Out of Scope）

- **本格実装（`apps/ios/` の製品レベルの作り込み）** — Spike が終わり go 判定が出た後に、別ステアリング（`.steering/[YYYYMMDD]-ios-screen-time`）で行う。
- **配布用の Apple 許可（distribution entitlement）の申請・App Store/TestFlight への配布** — ポストMVP（後回し）。Spike では開発検証用の許可（dev entitlement＝自分で付けられる開発用の許可）＋実機で確認する。ただし申請の手順・要件を**把握すること**は今回やる（A3）。
- **複数アプリを1サブスクにまとめる合算・カテゴリ／Web サイト（WebDomain）対象** — MVP から除外（中身を読めない値〔不透明トークン〕では合算できず、Web サイト対象は二重に数える報告あり）。Spike では1つのアプリ対象（ApplicationToken）だけにする。
- **1分・5分バケットの正式採用** — 計測の15分下限により、MVP では「15分以上」から扱う。1分・5分は「鳴るか実測する」までで、製品には採用しない。
- **裏で自動送信するしくみ（BackgroundTasks）を送信の主役にする作り込み** — アプリを開いた時に送る方式を主役にし、裏送信は将来の補助とする。
- **送信窓口 `/api/usage/daily` のトークン認証** — `USAGE_SYNC_TOKEN` による Bearer トークン認証は RE-11.3 で**実装済み**（コミット `937cfe9`）。Spike では既存の認証に合わせて Bearer ヘッダで送信する。
- **`apps/web`（Mac 側の既存コード）の機能変更** — Spike は Mac 側の決まりに「合わせる」側。Mac 側の改修は原則しない。
- **サブスクの適性分類・iPhone 外利用による誤判定の回避** — パターン判定方式（`.steering/20260609-quantitative-recommendation-engine/`）で**解決済み**。`usage_type` によって P1 の適用可否を制御し、受動利用（`passive`）・権利保有（`entitlement`）には P1 を適用しない設計が確定・実装済みである。本ステアリングでは技術検証のみを扱う。

## 3. ユーザーストーリー（＝誰が・何をしたいか）

- **US-1（ユーザー）**：iPhone で「このサブスクはこのアプリ」と一度だけ対応づければ、あとは自分の利用量が自動で SubBuddy に反映されてほしい（毎日手で利用記録を付けたくない）。
- **US-2（ユーザー）**：アプリ名や細かい利用ログがどこかに送られず、**まとめた数字（集計値）だけ**が手元の Mac に渡る、という安心感がほしい（プライバシー）。
- **US-3（開発者）**：本格実装に多くの時間をかける前に、**この計測のしくみが本当に実用になるか**を最小の手間で確かめ、ダメなら早めに方針転換したい。
- **US-4（開発者）**：再起動・日付の境目・しきい値通知など**壊れやすい箇所**の実機の動きを把握し、本実装の設計を固めたい。

### 4.1 合格ライン（go/no-go の判断基準・改訂版／Codex の反証＋Apple 公式ドキュメントを反映）

> **go＝本実装に進む。no-go＝自動取得をあきらめ「手動記録」へ切り替える。** 合格ラインは検証を始める前に固定する。**MUST（必須）が1つでもはっきり失敗すれば no-go**。SHOULD（推奨）は満たさなくても go にできるが、その場合はリスクを明記する。手動記録への切り替えはあくまで保険であり、「自動取得が惜しいから go」を救う条件にはしない（手動記録の価値は別途・独立に判断する）。

> ### 方針更新（2026-06-16）：まずローカルファーストで構築する
>
> 開発方針として、**まず自分の iPhone で動かすローカルファースト構成から作る**ことを確認した
> （配布＝他人に配ることは後回し）。これに伴い、go/no-go の見方を次のとおり整流する。
> 下の MUST 定義そのものは変えない（将来の配布判断で再び全 MUST が効く）。当面の「ローカルファーストで go」の判断では、各 MUST を次の重みで見る。
>
> | MUST | ローカルファースト構成での扱い | 理由 |
> |---|---|---|
> | MUST-1（受け渡し堅牢） | **必須ゲート（変更なし）** | 配管が成立しないと何も始まらない |
> | MUST-2（計測の立て直し） | **必須** | 自分用でも再起動・更新は日常的に起きる |
> | MUST-3（測定誤差が許容内） | **必須** | 取り込む数字の信頼性に直結 |
> | MUST-4（日付の境目） | **必須** | 日付がずれると見直し材料が壊れる |
> | MUST-5（二重通知でもズレない） | **必須** | 過大計上を防ぐ |
> | MUST-6（想定件数で安定） | **優先度↓（必須から緩和）** | 自分用は監視対象が数件。上限は実測で把握するが、20件級の合格は必須としない。安全上限を MVP スコープに明記すれば足りる |
> | MUST-7（開く運用で送れる） | **必須** | 欠損が気づかず貯まると節約判断が壊れる |
> | MUST-8（許可失効から立て直す） | **必須** | 自分用でも許可は切れる |
> | MUST-9（配布できる見込み） | **充足・当面オフ** | 2026-06-16 に机上で「配布見込みあり」を確認済み（`stage7-distribution-guide.md`）。ローカルファーストでは Apple 申請は不要のため、当面この MUST は判定対象から外す。配布判断時に再度この MUST を満たす |
>
> **当面の「ローカルファーストで go」の合格条件**：MUST-1〜5・7・8 が実機で OK（MUST-6 は安全上限を実測し MVP スコープに記録、MUST-9 は充足済み）。
> 残る実機検証は段階3・4・5 と段階1・6 の残項目（→ `field-test-guide.md` に手順と記録テンプレ）。

**MUST（必須。1つでも失敗すれば即 no-go）**

| ID | 項目 | 合格と言える状態 | no-go の例 |
|---|---|---|---|
| MUST-1 | 受け渡しが堅牢（A1） | 裏の計測→共有置き場→本体で、同じ日・同じサブスクの複数段階を全件記録でき、本体・端末の再起動後も読める。共有置き場に「どの監視/どの通知/日付/段階/生成時刻/連番（`activityId/eventId/date/bucket/generatedAt/sequence`）」を保存し、複数日・複数対象で取りこぼさない | 単発でしか読めない／同時到達で取りこぼす |
| MUST-2 | 計測状態を立て直せる | iPhone 再起動・強制終了・アプリ更新の後、状態を確認でき、仕掛け直し（再アーム）で復旧する（`startMonitoring` は同名監視を上書きする点に注意） | 仕掛け直しで当日値を頻繁にリセット/二重登録する |
| MUST-3 | 測定誤差が許容内 | 三点照合（対象を 0/16/31/61/121 分使う Controlled use ＋ Screen Time 表示 ＋ 通知ログ）で、有効7日中6日以上が1段階以内。境目日のズレは理由を説明できる。初日・途中開始日は除外/partial | ズレが大きく理由も説明できない |
| MUST-4 | 日付の境目が決めてある | 日付（`usageDate`）は iPhone 現地日付基準で確定し Mac で再解釈しない。日跨ぎ・TZ 変更・DST 相当でも同じ利用が別日に保存されない（`startMonitoring` は計測開始時の TZ で期間計算） | TZ/DST/日跨ぎで同じ利用が別日に保存される |
| MUST-5 | 二重通知でもズレない | 到達通知が即時/二重/順不同でも、端末側は単調更新、DB 側は `(subscriptionId, usageDate)` で段階を下げない（`max(段階)` か版数/観測時刻ルール） | 重複行・過大計上が起きる |
| MUST-6 | 想定件数でも安定 | MVP 上限（例 20 サブスク × 4 しきい値）で `startMonitoring` がエラー（throw）にならず、密集到達でも取りこぼし・共有置き場の競合が起きない（同時監視上限は Apple 非公開＝実測） | 想定件数の半分未満で失敗＋失敗検知/縮退UXなし |
| MUST-7 | 開く運用だけで送れる | ネットワーク不通・Mac 停止・アプリ再起動をはさんでも、未送信分を貯めて再送し、重複しない保存で最終的に数が正しく揃う | 欠けたデータが気づかないうちにたまる |
| MUST-8 | 許可失効から立て直せる | 許可の取消/失効を検知 → 再許可 → 対象を選び直し → 既存サブスクと結び直す画面操作（再対応づけ導線）がある（取消後は選択値=token が無効=void になる） | 再許可しても既存サブスクの監視/対応を復元できない |
| MUST-9 | 配布できる見込み | 配布用 Apple 許可の申請説明・`PrivacyInfo.xcprivacy`（使用理由の申告が要る API・集める情報・家庭内ネットへ送る情報）・App Store プライバシー表示・用途説明が矛盾せず、「節約」用途を説明しきれる見込みが立つ（配布自体は後回し） | parental-controls 文脈とのズレを説明しきれない |

**SHOULD（満たせば望ましい。go は可能だが、満たさない場合はリスクを明記）**

| ID | 項目 | 内容 |
|---|---|---|
| S-1 | 未送信データの保持上限 | 貯めておける上限を決める（最低30日／できれば90日）。超えたら「未同期で欠けた」と分かるように見せる |
| S-2 | 欠けの種類を区別表示 | 「iPhone がまだ送っていない」「Mac に届いていない」「許可が切れた」を見分けて見せる（気づかない欠けは節約アプリで致命的） |
| S-3 | 証明書ピン留めの更新手順 | QR で再ペアリング・複数の証明書を許容・期限切れ表示を検証（ピン留めは証明書を更新するとかえって通信できなくなることがある） |
| S-4 | 対応づけの分かりやすさ | 名前表示（`Label(token)`）で「このサブスク＝どの選択か」を後から理解・維持できるか確認 |
| S-5 | 対象アプリ削除/再インストール | その時の通知挙動を記録し、復旧の導線／FAQ に反映 |

**NICE-TO-HAVE（あれば尚良い）**

| ID | 項目 | 内容 |
|---|---|---|
| N-1 | iOS 大型更新後の回帰 | 壊れていないか再確認。Spike 中に難しければリリース前チェックリスト化 |
| N-2 | 監視数の安全マージン測定 | 上限ギリギリでの壊れ方を把握 |
| N-3 | 手動記録 MVP の価値評価 | 自動取得が no-go の時の、別プロダクト判断として切り離して評価 |

> 出典（Codex の反証が参照した Apple 公式）：`startMonitoring(_:during:events:)`（同名監視の上書き・多すぎ/密すぎでエラー・タイムゾーンは計測開始基準）、`DeviceActivityEvent`（計測開始後からの利用を見る）、`DeviceActivityReport`（データ持ち出し禁止の隔離）、`AuthorizationCenter`/`FamilyActivitySelection`（許可失効で選択値が無効化）、`Configuring Family Controls`（配布時は許可の承認が必要）、Privacy manifest / TN3183（不備のあるプライバシー説明は審査で却下対象）。

## 5. 制約事項（＝守るべき前提・ルール）

- **実行環境**：この Linux コンテナでは Xcode ビルド・実機検証ができない。実機作業はユーザーが Mac/iPhone で実施。Claude Code は `verify`/`run` を行わない（設計・記録・判断支援のみ）。
- **個人情報・秘密を扱わない**：試験用データのみ。細かいログ・全アプリ一覧は送らず、**まとめた数字（集計値）だけ**送る（アーキ文書の制約 TC-3）。アプリ名や中身を読めない値の内容は Mac へ送らない。
- **変えない前提（不変条件）**：自動で解約しない／外部サービスの ID・パスワードを保存しない／自動ログインやスクレイピング（＝サイトを機械的に巡回して情報を抜くこと）をしない／手元の Mac だけで完結する（ローカルファースト。Mac の localhost 単体で起動）。
- **既存の決まりに合わせる**：送信窓口 `POST /api/usage/daily` の形式（`usageDailyBatchSchema`）・段階の表し方（`none/1m_plus..120m_plus`）・保存キー `(subscriptionId, usageDate)` に iPhone 側を合わせる。Mac 側は原則改修しない。
- **薄く始める（rule of three＝同じものが3回現れてから共通化する考え方。早すぎる汎用化をしない）**：Mac 側の取り込み共通部品（`src/ingestion/connectors/screen-time/`）は今回作らない。Spike は最小の一本通しに留め、先回りの抽象化をしない。
- **バージョンの具体値**：iOS の最低要件・Swift/Xcode のバージョンは、実機/プロジェクト設定を一次情報とし、この文書には固定値を書かない（`.individual` 認可は iOS 16 以降が前提）。

- **(D1 ← Q2) 分担**：Spike の実機検証は**ユーザーが Mac/iPhone で実施**。Claude は手順書・チェックリスト・記録テンプレート・Swift コードの設計を用意し、**ビルド・実行はしない**（`verify`/`run` 不可）。
