iOS 実機テスト 同時進行ガイド

Screen Time（利用時間の取得）と Shortcuts（前面起動の検知）を、1台の iPhone で並行して検証する手順。初めて iOS を触る前提で記載。

1全体像

2つのテストは別々の iOS のしくみを使うため、同じ端末・同じ期間で同時に走らせて問題ない。互いに干渉しない。

AScreen TimeBShortcuts
取得するもの利用時間（◯分以上のバケット）前面でアプリを開いたという検知のみ
しくみDeviceActivity（＝iOS の利用時間計測）＋専用アプリ個人オートメーション（＝決めた条件で自動実行する設定）
送信アプリ起動時にまとめて送信アプリを開くたびに即送信
所要長い（精度確認に7日）短い（疎通確認は数時間）
注意：送信先・認証トークン・保存テーブルは両者で共通。テスト対象のサブスクは A 用・B 用で必ず分けること。同じサブスク・同じ日に両方が書くと1行にまとまり、どちらの結果か切り分けられなくなる。

2共通の前提準備 共通・最初に1回

A・B どちらの前にも、ここを通す。Mac 側を LAN に公開し、iPhone から届くことを確認する。

2-1. Mac の IP アドレスを調べる

ipconfig getifaddr en0

表示された IP（例 192.168.1.100）を後で使う。

2-2. SubBuddy を LAN から届くように起動する

Mac 上で直接起動する場合：

cd apps/web
npx next dev --hostname 0.0.0.0

devcontainer 内で起動している場合は、VS Code のユーザー設定（Cmd+Shift+P → Open User Settings (JSON)）に次を追加してウィンドウを再読み込みする：

"remote.localPortHost": "allInterfaces"
注意：127.0.0.1（localhost）は使えない。iPhone にとって 127.0.0.1 は iPhone 自身を指すため Mac に届かない。必ず上記の IP を使い、プロトコルは http（https の自己署名は失敗する）。

2-3. 公開できているか確認する

lsof -nP -iTCP:3000 -sTCP:LISTEN

*:3000 または 0.0.0.0:3000 なら公開済み。127.0.0.1:3000 なら未公開（2-2 を見直す）。

2-4. iPhone の Safari で疎通を確認する

iPhone の Safari で http://<IP>:3000 を開く。SubBuddy が表示されれば経路は正常。

注意：ここが表示されないうちはアプリや Shortcut から送っても無駄。必ず Safari で先に通すこと。以降のエラーを「経路」と「アプリ」に切り分けられる。
Safari でも開けないときの確認順：①iPhone と Mac が同じ Wi-Fiか。②Mac のファイアウォールがポート 3000 を塞いでいないか（システム設定 → ネットワーク → ファイアウォール。オンなら一時的にオフか、Node の通信を許可）。③一部のルーターは端末同士の通信を遮断する「クライアント分離（AP アイソレーション）」が有効なことがある。ゲスト用 Wi-Fi を避け、通常の Wi-Fi を使う。

2-5. 認証トークンを控える

Mac の apps/web/.env の USAGE_SYNC_TOKEN= の値を控える。A・B 両方で同じ値を使う。一致しないと送信が 401 になる。

注意（PII）：実際の IP・トークン・利用傾向は Git に置かない。このガイドの値はすべて例。

3推奨スケジュール

時間のかかる A の精度確認を先に始め、その裏で B を立ち上げると待ち時間が縮む。

順作業目安
1共通準備（第2章）30分
2Aアプリを組んで最小動作まで通す（第4章）半日〜1日
3A7日間の精度確認を開始し、毎日記録する7日

4トラックA：Screen Time A

専用の検証アプリ（spikes/ios-screen-time/）を Xcode でビルドして iPhone に入れる。詳細手順は同フォルダの README.md に沿う。要点とつまずきやすい所のみ記載する。

4-1. 準備（一度だけ）

  • Apple Developer Program（有料・年額 $99）に加入し、規約に同意しておく。
  • Xcode を入れた Mac、iOS 16 以降の iPhone、両者が同じ Wi-Fi。
注意：加入直後は規約反映に数時間〜24時間かかる。Xcode で PLA Update available 等のエラーが出たら未反映が原因。developer.apple.com/account で同意して待つ。

4-2. Xcode 設定の要点

  1. 本体アプリと Monitor 拡張（＝裏で動く小プログラム）の両方に、Family Controls と App Groups を付ける。
  2. App Group ID は本体と拡張で完全に同じにする（例 group.com.yourname.SubBuddySpike）。違うとデータを共有できない。
  3. Shared フォルダの4ファイルは 両ターゲットに所属させる（Target Membership）。漏れるとビルドエラー。
  4. ATS 設定で Allow Arbitrary Loads = YES。これがないとローカル Mac への HTTP 送信が全部失敗する。

4-3. 定数ファイルを用意する

cp spikes/ios-screen-time/SubBuddySpike/Shared/Constants.swift.example \
   spikes/ios-screen-time/SubBuddySpike/Shared/Constants.swift

Constants.swift の3箇所を自分の環境に書き換える（App Group ID／http://<IP>:3000／トークン）。

4-4. 最小動作の確認

  1. アプリで「Screen Time の許可」→「アプリを選ぶ」を実行。
  2. サブスク ID を入力（Mac の SubBuddy 詳細画面の URL subscriptions/xxx の xxx）。
  3. 「監視を開始」→ 対象アプリを 20分以上 使う → 「読み取り更新」でレコードが出る → 「未送信データを送信」。
注意：サブスク ID はいま送信先になっている DB に実在する IDを使う。存在しない ID は外部キー制約で HTTP 500 になる（再シードで ID が変わる点に注意）。
計測下限は約15分。15分ぴったりでは反応しないので20分使う。

DB への着弾確認（created_at では絞らない。冪等更新で時刻が変わらないため）：

SELECT s.name, u.usage_date, u.usage_bucket, u.estimated_minutes_min, u.source
FROM ios_usage_daily_summaries u JOIN subscriptions s ON s.id = u.subscription_id
WHERE u.subscription_id = '<送ったサブスクID>' AND u.usage_date = '<送った日付>';

4-5. 7日間の精度確認（先に開始する本題）

毎日、対象アプリを狙った時間だけ使い、3つを突き合わせる。field-test-guide.md の記録表に記入する。

  • ①手計測（ストップウォッチの実利用分）
  • ②iPhone 設定 → スクリーンタイムの表示分
  • ③アプリが取り込んだ段階（例 m120_plus）

合格目安：有効7日のうち6日以上で①と③が1段階以内。

注意（途中で監視が止まる）：7日間の途中でiPhone を再起動すると監視が止まることがある。再起動後はアプリを開き、ステータスが「監視中」かを確認する。止まっていたら「監視を開始」を押し直す（押し直しても当日の段階は下がらない）。気づかないと数日分が空欄になる。
注意（0分の日）：まったく使わなかった日は通知が来ないためレコードが無いのが正常。「使っていない日（none）」と「まだ送れていない日（missing）」は別物。記録表ではその日を「0分・利用なし」として埋め、欠損と区別する。

5トラックB：Shortcuts B

iPhone だけで完結する。アプリのビルドは不要。

5-1. オートメーションを作る

まずトリガを決め、その後でアクションを上から4つ並べる。アクションは候補一覧には出ないので、検索窓に名前を打ち込んで探す。

  1. ショートカットアプリ →「オートメーション」→ 個人用を新規作成。
  2. トリガ：「App」→ 対象アプリ →「開いている（Is Opened）」。
  3. 「次へ」→「新規の空のオートメーション」を選び、アクション編集画面に入る。

ここから、次の4アクションを上から順に追加する。

アクション① 日付（Date）

検索窓で 日付 →「日付」を追加。設定不要（「現在の日付」になる）。

アクション② 日付を整形（Format Date）

検索窓で 日付を整形 → 追加。日付の入力欄に①の変数が入っていることを確認し、形式 → カスタムを選んでフォーマットに yyyy-MM-dd と入力。→ 2026-06-20 のような端末ローカル日付の文字列ができる。

アクション③ テキスト（Text）— ここに JSON を書く

検索窓で テキスト →「テキスト」を追加。下の JSON をそのまま貼り付け、date の値の中身だけを消して②の整形済み日付を変数として差し込む（手打ち文字ではなく変数）。ダブルクォートは残したまま中身だけ変数にする。

{"items":[{"subscriptionId":"<B用サブスクのUUID>","date":"<ここに②の変数>","used":true,"usageBucket":"none","source":"ios_shortcut"}]}

アクション④ URL の内容を取得（Get Contents of URL）

「表示を増やす（Show More）」を開いて設定する。

項目設定値
URLhttp://<MacのIP>:3000/api/usage/daily
方法（Method）POST
ヘッダ（Headers）Authorization = Bearer <トークン>
Content-Type = application/json
要求の本文（Request Body）初期値の「フォーム」から「ファイル」に切り替える
本文の中身③のテキスト変数を選ぶ
注意（本文の入れ方＝最も詰まりやすい）：本文の種類を「フォーム」→「ファイル」に変え、そこに③のテキスト（JSON 文字列）変数を渡す。フォームのままだと「キー＝値」の別形式になり、受信側が JSON として読めず弾かれる（request body must be valid JSON）。JSON を④のフォームで手組みせず、③で文字列を完成させてから送るのが確実。
注意（サブスク ID）：subscriptionId は送信先 DB に実在する IDを使う（A の 4-4 と同じ。ダッシュボードのサブスク詳細 URL の subscriptions/xxx の xxx をコピペ）。実在しない ID は外部キー制約で HTTP 500 になる。A とは別のサブスクを割り当てる。
注意（日付の作り方）：date は①②で動的に作る。固定で書くと毎回同じ日になる。端末ローカルの時間帯で作り、Mac 側で解釈し直さない。日跨ぎで別日にならないか後で確認する。

5-2. 無音で自動実行する設定

  • 「実行前に尋ねる（Ask Before Running）」を OFF。
  • 「実行時に通知（Notify When Run）」を OFF。
  • iOS 17 以降は「すぐ実行（Run Immediately）」を選ぶ。
  • 別途、オートメーションのサマリ通知も OFF にする。

5-3. 疎通ゲート（ここが必須の関門）

対象アプリを開いて、次をすべて満たすか確認する。

  • 確認タップやバナーなしで発火する。
  • POST が通り 200／201 が返る（初回は「ローカルネットワークの許可」ダイアログが出るので許可）。
  • DB の対象行で daysSinceLastUse 相当（最終利用日）が更新される。
注意（意味の限界）：このシグナルは「前面で開いた」時しか出ない。背景再生（ロック画面のみの音楽再生など）では発火しない。「検知あり＝最近触った」には使えるが、「検知なし＝使っていない」とは断定しない。

6ミスしやすい箇所（共通）

症状原因対処

sudo service postgresql start

復旧後に iPhone 側で再送すれば、止まっていた間のデータも冪等に入る（重複しない）。
