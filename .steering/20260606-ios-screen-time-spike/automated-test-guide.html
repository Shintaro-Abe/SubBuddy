iOS 実機テスト 自動化ガイド

Screen Time（利用時間の取得）と Shortcuts（前面起動の検知）の配線を、Xcode の GUI 操作なしで MacBook 上のコマンド一発で通す。手動版（同時進行ガイド）の自動化版。配線（パイプライン）の結合スモークが対象で、利用量の「精度」は別途の手動フィールドテストが担う。

1このガイドの対象範囲

自動化が確かめるのは「iPhone アプリ → Mac → Web 管理画面の配線が実機で end-to-end に動くか」。閾値到達に 15 分以上かかる利用量の精度や、OS のプライバシーシート操作は対象外。

AScreen TimeBShortcuts
取得するもの利用時間（◯分以上のバケット）前面でアプリを開いた検知のみ（時間は測らない）
しくみDeviceActivity ＋専用アプリ（自動ビルド・自動インストール・Appium で UI 操作）個人オートメーション（前面起動で POST /api/usage/daily を即送信）
DB の sourceios_device_activityios_shortcut
自動化の度合いビルド〜操作〜Web 確認まで一括 スクリプト送信自体が自動。疎通は Safari／curl で確認
重要な環境前提：Claude Code / devcontainer は Linux。iOS ビルド・実機・Appium は Mac のネイティブターミナルでしか動かない（xcodebuild/xcrun/USB が無い）。Web アプリと PostgreSQL はコンテナ側で起動し、VS Code のポート転送で Mac/LAN から到達させる。

2スクリプト構成 spikes/ios-screen-time/scripts

スクリプト役割
setup.sh初回セットアップ。e2e/.env.local と Constants.swift を生成し、UDID・LAN IP・Team ID・サブスク ID・トークンを自動取得。--run で実行まで。

3一度きりの手動準備（自動化対象外）

以下は人手でしか実施できない。最初に 1 回だけ。

  • iPhone：Developer Mode 有効化／このコンピュータを信頼／アプリ内で Screen Time 許可＋対象アプリ選択（認可は保持される）。
  • Mac（ツール導入）：下記。Node は 22 LTS（24 系は undici で webdriverio が UND_ERR_INVALID_ARG になる）。
brew install xcodegen

4実行手順 Mac のネイティブターミナル

4-1. Web アプリと DB を起動（コンテナ側）

devcontainer 内で PostgreSQL と Web を起動し、LAN へ公開する。

sudo pg_ctlcluster 15 main start          # PostgreSQL
cd apps/web && npm run dev -- --hostname 0.0.0.0   # Next.js

VS Code 転送を LAN に出すため、ユーザー設定（Cmd+Shift+P → Open User Settings (JSON)）に追記してウィンドウ再読込：

"remote.localPortHost": "allInterfaces"
注意：iPhone にとって 127.0.0.1 は iPhone 自身。必ず Mac の LAN IP を使い、プロトコルは http。PORTS タブの Local Address が 0.0.0.0:3000 になっていれば公開済み。

4-2. 初回はセットアップ

cd spikes/ios-screen-time
scripts/setup.sh            # 生成のみ（全値を自動取得）
scripts/setup.sh --run      # 生成 → preflight → 実機 E2E まで

4-3. 通常実行（コード変更を反映）

cd spikes/ios-screen-time
scripts/run-e2e-real-device.sh

4-4. 高速反復（再ビルドせず・認可を保持）

SKIP_BUILD=1 scripts/run-e2e-real-device.sh

成功すると [e2e] PASSED / ✅ E2E 成功。失敗時は e2e/artifacts/<日時>/ に appium.log・xcodebuild.log・appium-failure.png・playwright-trace.zip。

5LAN IP の自動同期（再起動対策）refresh-ip.sh

Mac の LAN IP は再起動・DHCP で変わる。変わると iPhone から届かず request time out になる。run-e2e は実行時に refresh-ip.sh を自動で呼び、現 IP を検出して Constants.swift の apiBaseURL を更新する。単体実行も可：

scripts/refresh-ip.sh          # 検出 → 必要なら Constants.swift 更新 → 案内
scripts/refresh-ip.sh --check  # 変更せず差分の有無だけ判定（preflight 用）

IP 検出は デフォルトルートのインターフェースを優先する（Wi-Fi が en0 以外＝Ethernet/Thunderbolt 併用時 en1 等でも正しく拾える）。

IP が変わった回に必要な手当て（焼き込みは 2 箇所）：
  1. アプリの再ビルド・再インストール。apiBaseURL はアプリにコンパイルされるため、SKIP_BUILD=1 を使わず run-e2e を回す。
  2. ショートカットの送信先 URL を変更。ショートカット App で「URL の内容を取得」アクションの http://<旧IP>:3000/api/usage/daily を新 IP に直すか、refresh-ip.sh が生成する e2e/.generated/subbuddy-usage.shortcut を取り込み直す。
最後に iPhone(Wi-Fi) の Safari で http://<新IP>:3000 が開けるか確認。

6トラックA：Screen Time の自動 E2E A

run-e2e が XcodeGen → xcodebuild → devicectl インストール → Appium(XCUITest) でアプリ操作 → Playwright で Web 確認まで通す。利用時間の取得そのものは、対象アプリを前面で使って閾値を跨ぐことで発火する。

6-1. 閾値とバケットの対応

監視閾値は Constants.swift の thresholdMinutes = [15, 30, 60, 120]。到達した最上位段階が記録される。

フォアグラウンド累積記録される bucket
〜15分未満none
15分到達m15_plus
30分到達m30_plus
60分到達m60_plus
120分到達（最上位）m120_plus

識別できる上限は 120 分。配線確認だけなら 15／30／60 分のいずれかで止めても、その段階の bucket で経路は証明できる。

6-2. 計測の前提

  • アプリの「監視を開始」で DeviceActivity を登録して初めて閾値が発火する。ボタンが「監視を停止」表示なら既にアーム済み。
  • 監視イベントは includesPastActivity: true（iOS 17.4+）。視聴後に「監視を開始」を押しても当日分はさかのぼって計上される。だから project.yml は deploymentTarget iOS 17.4。
  • スケジュールは midnight〜midnight・repeats: true で毎日自動再アーム。ただし iPhone 再起動で監視が落ちることがあるので、再起動後はアプリを開いて状態を確認。
  • 計測外：ロック/スリープ中の背景音声、別端末での視聴、他アプリに切り替え中。

7トラックB：Shortcuts の起動シグナル B

個人オートメーション（対象アプリを開いたら実行）が POST /api/usage/daily を即送信する。送信自体が自動なので、確認は受信側で行う。記録は「今日使った」という事実だけ（used=true / usage_bucket=none / source=ios_shortcut）。

意味の限界：このシグナルは「前面で開いた」時しか出ない。背景再生では発火しない。「検知あり＝最近触った」には使えるが「検知なし＝使っていない」とは断定しない。

8結果の確認（受信側）

送信後、Web の next dev ログと DB を見る。

8-1. 送信が認証を通って届いたか（ログ）

grep "POST /api/usage/daily" <next dev のログ> | tail

200＝成功。401＝トークン不一致、400/422＝本文不正、ログに何も出ない＝そもそも届いていない（経路＝LAN IP/到達性の問題）。

8-2. DB に入ったか

psql "$DATABASE_URL" -c "SELECT usage_date, used, usage_bucket, source
  FROM ios_usage_daily_summaries
  WHERE subscription_id='<対象サブスクID>'
  ORDER BY usage_date DESC LIMIT 5;"

送信先は subscription_id × usage_date で冪等 upsert。同じ日に複数回送っても行は増えず、最後の送信値で上書きされる（同日内は時間が増える一方なので値は下がらない）。

合格例（Screen Time・60 分テスト）：used=true / usage_bucket=m60_plus / source=ios_device_activity の行が当日分として 1 行入り、ログに POST /api/usage/daily 200 が出ていれば、IP 修正 → 到達 → 認証通過 → 冪等 upsert まで配線が end-to-end で通ったことになる。
