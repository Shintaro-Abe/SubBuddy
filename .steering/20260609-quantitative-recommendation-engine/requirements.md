## 1. レコメンドエンジンのパターン判定方式への転換

SubBuddy のレコメンドエンジンを、**利用量ベースの判定**から**パターン判定方式**に転換する。

現行の判定ロジックは「未使用日数」を中核に据えているが、以下の調査結果から方式を変更する。

| 調査で確定した事実 | 出典 |
|---|---|
| 利用量ベースの解約判定は業界前例ゼロ | `research/20260607-subscription-app-market-research/` |
| Screen Time の計測は「画面の最前面にある時間」だけ。背景音声・別端末は計測外 | `research/20260606-usage-based-recommendation-validity/` |
| サブスクの種類によって「利用している」の意味が異なる。受動利用や権利保有には「使っていない＝解約」が成り立たない | 本ステアリングでの具体化検討 |

### 変更後の判定方式

抽象的なスコアではなく、**具体的な状況（パターン）ごとにレコメンドを定義**する。サブスクの「利用の性質」に応じて適用するパターンを切り替える。

## 2. 「利用している」の定義（サブスクの性質ごと）

サブスクリプションの種類によって「利用している」の意味が異なる。この分類を間違えると、受動利用のサブスクに「使っていないから解約」と誤ったレコメンドを出してしまう。

| 利用の性質 | 意味 | 例 | SubBuddy での検出 |
|---|---|---|---|
| **能動利用（前面）** | ユーザーがアプリを画面で操作する | Netflix視聴・ゲーム・Todo入力 | **検出可能**（DeviceActivity で前面利用時間を計測） |
| **能動利用（背景）** | アプリは裏で動いているが、ユーザーが意図的に使っている | Spotify再生・ポッドキャスト | **起動だけ検出可能**（Shortcuts）。再生時間は不明 |
| **能動利用（別端末）** | iPhone以外で使っている | PCでAdobe CC・TVでNetflix | **検出できない** |
| **受動利用** | ユーザーが何もしなくても価値を提供し続けている | Dropbox同期・1Password保管・VPN常時接続 | **検出不要**（稼働している限り「利用中」） |
| **権利保有** | 使う頻度に関係なく、権利を持っていることが価値 | Amazon Prime送料無料 | **検出できない** |

## 4. レコメンドパターン（具体的な状況 → 具体的なレコメンド）

| # | パターン | 具体的な状況 | レコメンド | 適用条件 |
|---|---|---|---|---|
| P1 | **使っていない** | 判定スパン内の利用日数＋最終利用からの経過日数の両方で判定（§4.1 参照） | 解約検討 | 能動利用（前面）のサブスクのみ。受動利用・権利保有には適用しない |
| P2 | **重複で割高** | 同カテゴリに複数あり、こちらの方が高い | 解約検討（安い方を残す） | 全サブスク共通 |
| P3 | **安いプランがある** | 同じサービスにもっと安い有料プランがある | ダウングレード検討 | 知識ベースにプラン情報があるサブスク。無料プランは比較対象外 |
| P4 | **安い競合がある** | 同カテゴリにもっと安い有料の競合がある | 乗り換え検討 | 知識ベースに代替情報があるサブスク。無料プランは比較対象外 |
| P5 | **更新が近い** | 年額更新が7日以内に迫っている | 更新前に見直し | 年額契約のサブスク |
| P6 | **高額で長期継続** | 月額¥2,000以上で12ヶ月以上継続 | 本当に必要か確認 | 全サブスク共通 |
| P7 | **該当なし** | 上記のどれにも該当しない | 継続 | — |

### パターンの適用可否マトリクス

| パターン | 能動（前面） | 能動（背景） | 能動（別端末） | 受動 | 権利保有 |
|---|---|---|---|---|---|
| P1 使っていない | **適用** | 補助的に適用（起動のみ） | 適用不可 | **適用不可** | **適用不可** |
| P2 重複で割高 | 適用 | 適用 | 適用 | 適用 | 適用 |
| P3 安いプランがある | 適用 | 適用 | 適用 | 適用 | 適用 |
| P4 安い競合がある | 適用 | 適用 | 適用 | 適用 | 適用 |
| P5 更新が近い | 適用 | 適用 | 適用 | 適用 | 適用 |
| P6 高額で長期継続 | 適用 | 適用 | 適用 | 適用 | 適用 |

### 各パターンの具体的な判定条件

#### 4.1 P1：使っていない（方式C＝スパン内の利用日数＋最終利用からの経過日数の両方で判定）

P1 は「スパン内の利用日数」と「最終利用からの経過日数」の**両方**を見て判定する。
片方だけだと以下の誤判定が起きるため、両方を組み合わせることで揺れに強い判定にする。

| 片方だけの問題 | 具体例 |
|---|---|
| スパン固定のみだと、年に1回しか使わないサービスを誤判定する | 確定申告ソフト：30日間利用なし → 解約検討（実際は年1回で正常） |
| 最終利用日のみだと、たまたま1回開いただけでリセットされる | 3ヶ月使っていないサービスを昨日1回だけ開いた → 最終利用1日前 → 非該当（実態は使っていない） |

**判定スパンの決め方**

| 契約タイプ | 判定スパン |
|---|---|
| 月額契約 | 直近30日 |
| 年額契約 | 直近365日 |

**判定フロー**

```
usage_type を確認する
  ├ active_foreground の場合
  │   → 判定スパン内の利用日数と最終利用からの経過日数を見る
  │
  │   ① スパン内利用日数 = 0 かつ 最終利用から61日以上
  │     → P1 該当（解約検討）
  │     → evidence="最後に使ったのはX日前です。直近N日間の利用が0日です"
  │
  │   ② スパン内利用日数 = 0 かつ 最終利用から31〜60日
  │     → P1 該当（様子見）
  │     → evidence="最後に使ったのはX日前です"
  │
  │   ③ スパン内利用日数 ≧ 1 かつ 最終利用から61日以上
  │     → P1 該当（様子見）
  │     → evidence="直近N日間に利用がありますが、最後に使ったのはX日前です"
  │     （過去に使ったが最近は使っていない状態）
  │
  │   ④ スパン内利用日数 ≧ 1 かつ 最終利用から30日以内
  │     → P1 非該当（使っている）
  │
  ├ active_background の場合
  │   → 上記と同じロジックだが、Shortcuts の起動記録を使用
  │   → caveat="背景再生は計測外です"
  │
  ├ active_other_device → P1 適用不可
  ├ passive → P1 適用不可
  ├ entitlement → P1 適用不可
  └ capacity → P1 適用不可
```

**具体例**

| サブスク | 契約 | スパン内利用日数 | 最終利用 | 判定 |
|---|---|---|---|---|
| Netflix | 月額 | 0日/30日 | 90日前 | **解約検討**（利用ゼロ＋長期未利用） |
| Netflix | 月額 | 0日/30日 | 45日前 | **様子見**（利用ゼロだが経過日数は中程度） |
| Netflix | 月額 | 2日/30日 | 3日前 | **非該当**（使っている） |
| Netflix | 月額 | 1日/30日 | 75日前 | **様子見**（スパン内に利用あるが最終利用が古い） |
| 確定申告ソフト | 年額 | 3日/365日 | 300日前 | **非該当**（年間スパン内に利用あり＝年1回の正常利用） |
| 確定申告ソフト | 年額 | 0日/365日 | 400日前 | **解約検討**（1年間利用ゼロ） |
| 某アプリ | 月額 | 1日/30日 | 1日前 | **非該当**（たまたま昨日開いたが、スパン内の利用日数も1日ある＝ここだけ見れば使っている。ただし P6 等の他パターンで検出される可能性あり） |

**P2：重複で割高**

```
同カテゴリに他のサブスクがあるか？
  ├ はい → 自分の月額は同カテゴリ内で最安か？
  │   ├ 最安でない → P2 該当
  │   │   evidence="同カテゴリに¥XXX/月のサービスがあります"
  │   └ 最安 → P2 非該当
  └ いいえ → P2 非該当
```

**P3：安いプランがある**

```
カタログに紐付いているか？（matched_service_id があるか）
  ├ はい → ServicePlan に自分より安い有料プランがあるか？
  │   （is_free_tier = true のプランは除外する）
  │   ├ はい → P3 該当
  │   │   evidence="○○プラン（¥XXX/月）に変更できます"
  │   └ いいえ → P3 非該当
  └ いいえ → P3 非該当（カタログ未登録のため判定できない）
```

**P4：安い競合がある**

```
カタログに紐付いているか？（matched_service_id があるか）
  ├ はい → ServiceAlternative に自分より安い有料の競合があるか？
  │   （is_free_tier = true は除外する）
  │   ├ はい → P4 該当
  │   │   evidence="○○（¥XXX/月）があります"
  │   └ いいえ → P4 非該当
  └ いいえ → P4 非該当
```

**P5：更新が近い**

```
年額契約か？
  ├ はい → 更新日まで7日以内か？
  │   ├ はい → P5 該当
  │   │   evidence="年額更新まで残りX日（更新額¥XXX）"
  │   └ いいえ → P5 非該当
  └ いいえ → P5 非該当
```

**P6：高額で長期継続**

```
月額¥2,000以上か？
  ├ はい → 契約12ヶ月以上か？
  │   ├ はい → P6 該当
  │   │   evidence="Xヶ月継続中（累計¥XXX）"
  │   └ いいえ → P6 非該当
  └ いいえ → P6 非該当
```

### レコメンド（Decision）の決定ルール

P1〜P6 の判定結果から、最終的なレコメンドを1つ決定する。

```
該当したパターンを集める
  ↓
各パターンを Decision に変換する
  P1（解約検討：スパン内利用0日＋最終利用61日以上） → strong_cancel_candidate
  P1（様子見：スパン内利用0日＋最終利用31〜60日、またはスパン内利用あり＋最終利用61日以上） → review
  P2 → consider_cancel
  P4 → consider_cancel
  P3 → consider_downgrade
  P5 → review
  P6 → review
  ↓
複数該当した場合は最も強い Decision を採用する
  強い順：strong_cancel > consider_cancel > consider_downgrade > review > keep
  ↓
何も該当しない場合 → keep
```

該当した**全パターンの根拠（evidence）**をレコメンドと一緒に表示する。

## 5. 判定に使うデータソース

| # | データソース | 取得方法 | 対応するパターン |
|---|---|---|---|
| 1 | 契約事実（料金・更新日・カテゴリ） | ユーザーが登録時に入力 | P2, P5, P6 |
| 2 | iOS 利用量（前面利用時間） | DeviceActivity で計測（iOS Spike で検証済みの方式） | P1 |
| 3 | Shortcuts 起動記録 | iPhone のショートカットから API サーバに HTTP POST（§5.1 参照） | P1（補助） |
| 4 | サービス知識ベース（代替・プラン・料金） | 手動カタログ（主要50件）＋陳腐化管理 | P3, P4 |
| 5 | 契約期間（登録日から自動算出） | 自動算出 | P6 |
| 6 | 初回登録時の1問（「なくなったら困るか」） | ユーザーが登録時に回答 | P1〜P6 の信頼度補強 |

### 5.1 Shortcuts 設定のユーザー工数

Shortcuts の設定はユーザーが**サブスクごとに1回だけ手動で行う**。以後は自動。

**1サブスクあたりの設定手順（約40秒）**

| ステップ | 操作 | 所要時間 |
|---|---|---|
| 1 | SubBuddy のサブスク詳細画面で QR コードを表示する | 10秒 |
| 2 | iPhone のカメラで QR コードを読み取る | 5秒 |
| 3 | ショートカットアプリで対象アプリ（例：Netflix）を選ぶ | 15秒 |
| 4 | 「実行前に尋ねる」をオフにする | 5秒 |
| 5 | 「完了」を押す | 5秒 |

**登録件数ごとの合計時間**

| 登録件数 | 合計時間 |
|---|---|
| 3件 | 約2分 |
| 5件 | 約3分半 |
| 10件 | 約7分 |
| 20件 | 約13分 |

**補足**

- 全サブスクに設定する必要はない。**能動利用（前面・背景）のサブスクだけ**が対象
- 受動利用（Dropbox等）・権利保有（Amazon Prime等）は設定不要（P1 を適用しないため）
- 設定しなくても P2〜P7 のレコメンドは出る。Shortcuts は P1 の精度を上げるためのオプション

### 5.2 Shortcuts の送信方式

**構成**

```
iPhone（ショートカットのオートメーション）
  → 対象アプリを開いたとき自動実行
    → インターネット経由で HTTP POST
      → SubBuddy の API サーバ
        → DB に保存
```

- iPhone と Mac が同じ Wi-Fi にいる必要はない。API サーバをインターネット上に立てれば、外出先（4G/5G）からでも送信できる
- iOS アプリの開発は不要。iPhone 標準のショートカットアプリと API サーバだけで完結する

**送信するデータ（パラメータ）**

| パラメータ | 型 | 値の例 | 説明 |
|---|---|---|---|
| `subscriptionId` | string | `sub_abc123` | SubBuddy がサブスク登録時に発行する ID |
| `date` | string | `2026-06-08` | ショートカット実行時の日付（YYYY-MM-DD） |
| `used` | boolean | `true` | 固定値。起動した＝使った事実 |
| `usageBucket` | string | `1m_plus` | 固定値。起動の事実のみで時間は不明なため最小バケット |
| `source` | string | `ios_shortcut` | データの取得元を示す識別子 |

**リクエスト例**

```json
POST /api/usage/daily

{
  "items": [
    {
      "subscriptionId": "sub_abc123",
      "date": "2026-06-08",
      "used": true,
      "usageBucket": "1m_plus",
      "source": "ios_shortcut"
    }
  ]
}
```

- この形式は既存の `usageDailyBatchSchema`（`apps/web/src/schemas/usage.ts`）にそのまま適合する
- `source` に `ios_shortcut` を追加するだけで、DeviceActivity 経由のデータと区別できる

**ID とサブスクリプション名の紐付け**

- ショートカットから送られるのは `subscriptionId`（例：`sub_abc123`）だけ
- SubBuddy は DB の `subscriptions` テーブルで `sub_abc123 = Netflix` と照合する
- 紐付けのための新しいテーブルは不要。既存の `subscriptions.id` をそのまま使う
- サブスクリプション名はユーザーが登録時に入力する。ID は SubBuddy が自動発行する

**QR コードによる設定の簡略化**

- SubBuddy のサブスク詳細画面（Mac のブラウザ）に QR コードを表示する
- QR コードには API の URL と subscriptionId が含まれる
- ユーザーが iPhone のカメラで読み取ると、ショートカットの設定画面に URL と ID が入った状態で開く
- iOS アプリは不要。QR コードは Mac のブラウザに表示し、iPhone の標準カメラで読み取る

## 6. 具体例

**Netflix（能動・前面＋別端末）**

```
該当パターン：
  P1：過去30日間の前面利用が0分です → 解約検討
      ※ただしTV/PCでの利用は検出できないため、その旨を明記
  P2：動画配信カテゴリに Disney+（¥990/月）もあります → 重複
  P3：広告つきスタンダード（¥790/月）に変更できます → ダウングレード検討

表示例：
  推奨：解約検討
  根拠：
    ・iPhoneでの利用が30日間ありません（TV/PCでの利用は計測外です）
    ・同カテゴリに Disney+ があります
    ・広告つきプラン（¥790/月）に変更すると年間¥9,600の節約
  解約した場合の節約額：年間¥19,080
```

**Dropbox（受動利用）**

```
該当パターン：
  P1：適用不可（受動利用のため「使っていない」判定はしない）
  P4：Google Drive（¥250/月・100GB）があります → 乗り換え検討

表示例：
  推奨：乗り換え検討
  根拠：
    ・同カテゴリに Google One（¥250/月）があります
  乗り換えた場合の節約額：年間¥12,600
  注意：ファイルの移行が必要です
```

**Amazon Prime（権利保有）**

```
該当パターン：
  P1：適用不可（権利保有のため「使っていない」判定はしない）
  P2〜P4：代替が少なく該当なし
  P6：月額¥600で12ヶ月以上継続 → 月額¥2,000未満のため該当なし

表示例：
  推奨：継続
  根拠：
    ・解約検討に該当する条件がありません
```

**AIツール（能動・前面＋別端末）**

```
該当パターン：
  P1：過去30日間のiPhoneでの前面利用が0分です → 解約検討
      ※ただしPC/Webでの利用は検出できないため、その旨を明記
  P2：AIツールカテゴリに ChatGPT Plus と Claude Pro があります → 重複
  P6：月額¥2,960で18ヶ月継続 → 高額長期

表示例：
  推奨：見直し検討
  根拠：
    ・iPhoneでの利用が30日間ありません（PC/Webでの利用は計測外です）
    ・同カテゴリに3件の契約があります
    ・18ヶ月継続中（累計¥53,280）
```

## 7. サービスカタログの `usage_type` フィールド

サービスカタログの各サービスに `usage_type` を持たせることで、パターン P1 の適用可否を自動判定する。

```json
{
  "id": "netflix",
  "display_name": "Netflix",
  "category": "video_streaming",
  "usage_type": "active_foreground",
  ...
}
```

| `usage_type` の値 | 意味 | P1 の適用 |
|---|---|---|
| `active_foreground` | 能動利用（前面） | する |
| `active_background` | 能動利用（背景） | 補助的に |
| `active_other_device` | 能動利用（別端末） | しない |
| `passive` | 受動利用 | しない |
| `entitlement` | 権利保有 | しない |
| `capacity` | 容量ベース（iCloud+専用） | しない（容量判定） |

カタログにないサービス（`matched_service_id = null`）は、ユーザーが登録時にカテゴリと `usage_type` を選ぶ。

## 8. ユーザーストーリー

| # | ストーリー |
|---|---|
| US-1 | ユーザーとして、登録したサブスクに対して SubBuddy が「継続／様子見／解約検討」のレコメンドを**具体的な根拠付きで**出してくれるので、どのサブスクを見直すべきか判断できる |
| US-2 | ユーザーとして、レコメンドに「30日間利用なし」「同カテゴリに重複あり」「安いプランあり」等の具体的な理由が表示されるので、なぜその判定になったか理解できる |
| US-3 | ユーザーとして、解約・ダウングレード・乗り換えした場合の年間節約額が表示されるので、行動する動機が生まれる |
| US-4 | ユーザーとして、サブスク登録時にカタログから選択できるので、入力が楽で代替サービスとの比較や利用の性質の判定も自動で受けられる |
| US-5 | ユーザーとして、受動利用のサブスク（クラウドストレージ等）に対して「使っていないから解約」という誤ったレコメンドが出ないので、信頼できる |
| US-6 | ユーザーとして、iPhone 前面で使うアプリは利用時間が自動計測されるので、利用実態に基づいた正確なレコメンドを受けられる |

## 9. 受け入れ条件

| # | 条件 | 検証方法 |
|---|---|---|
| AC-1 | 全登録サブスクに対してレコメンド（P1〜P7 のいずれか）が表示される | 合成データで各 `usage_type` のサブスクを登録し、全件にレコメンドが出ることを確認 |
| AC-2 | レコメンドに該当パターンの具体的な根拠が表示される | 画面上にパターン名と数値が1つ以上表示されることを確認 |
| AC-3 | 解約検討・ダウングレード検討のサブスクに年間節約額が表示される | 該当判定のサブスクで節約額が表示されることを確認 |
| AC-4 | サービスカタログからの選択式登録が動作する（あいまい検索付き） | 「ネトフリ」入力で「Netflix」が候補に出ることを確認 |
| AC-5 | カタログから選択すると `usage_type` が自動で設定される | Netflix を選択して `active_foreground` が設定されることを確認 |
| AC-6 | `usage_type` が `passive` または `entitlement` のサブスクに P1（使っていない）が適用されない | Dropbox（passive）を登録し、利用データなしでも P1 が出ないことを確認 |
| AC-7 | `usage_type` が `active_foreground` のサブスクに DeviceActivity の利用時間データが反映される | iOS 利用量データを受信し、P1 の判定に使われることを確認 |
| AC-8 | Shortcuts から POST された起動記録が DB に保存される | curl 等で `/api/usage/daily` に `source: "ios_shortcut"` を送り、DB に保存されることを確認 |
| AC-9 | サービス知識ベースの `verified_at` が6ヶ月超過で信頼度が低下する | `verified_at` を6ヶ月前に設定し、P3/P4 の判定への影響を確認 |
| AC-10 | 知識ベースの料金比較で無料プランが比較対象から除外される | 無料プランのみの代替があるサービスで P3/P4 が出ないことを確認 |
| AC-11 | 既存の E2E テスト（Playwright）が壊れない | `npm run test:e2e` が全件パスする |

## 10. 制約事項

| # | 制約 | 理由 |
|---|---|---|
| C-1 | レコメンドの判定は**定量データとパターンマッチ**で行う。LLM による判定は使わない | 再現性・説明可能性の確保 |
| C-2 | サービス知識ベースの料金データに LLM の出力を**スコアに直接反映しない**。LLM は候補提案・開発者レビュー支援のみ | ハルシネーションリスクの回避 |
| C-3 | 料金比較では**無料プランを比較対象にしない**。有料プラン同士のみで比較する | 無料プランは機能制限があり有料の代替にならない |
| C-4 | 受動利用・権利保有のサブスクに P1（使っていない）を適用しない | 受動利用は稼働している限り「利用中」。誤った解約レコメンドを防ぐ |
| C-5 | ユーザーへの継続的な手入力を求めない。初回登録時の1問と Shortcuts の1回設定のみ | product-requirements 方針 |
| C-6 | PII・機微データ方針を遵守する。Shortcuts が送るデータは「サブスクID＋日付＋起動した事実」のみ | CLAUDE.md PII 方針 |
| C-7 | 既存の `computeRecommendation`（純関数）・`config/scoring.ts`（しきい値外出し）のアーキテクチャ方針を維持する | 既存設計との整合 |
| C-8 | サービスカタログは MVP で主要50件から開始する。全サービス網羅は求めない | 個人開発のリソース制約 |
| C-9 | Shortcuts 自動記録に iOS アプリの開発は不要。API サーバへの HTTP POST で完結する | 調査で確認済み |
| C-10 | iPhone 前面利用アプリの利用量取得には iOS アプリ（DeviceActivity Monitor 拡張）が必要 | Apple の API 仕様 |

## 11. スコープの線引き

### 本ステアリングに含まれるもの

| 項目 |
|---|
| パターン判定ロジックの実装（P1〜P7） |
| `usage_type` によるパターン適用可否の制御 |
| サービス知識ベース（主要50件のカタログ＋あいまい検索＋陳腐化管理） |
| サービスカタログ選択式の登録 UI |
| Shortcuts 起動記録の受信（既存 API の `source` 拡張） |
| QR コード表示（Shortcuts 設定用） |
| レコメンド表示 UI（パターン名・根拠・節約額の表示） |
| product-requirements の該当箇所の改訂 |

### 本ステアリングに含まれないもの（別ステアリングまたはポストMVP）

| 項目 | 理由 |
|---|---|
| iOS DeviceActivity の実装（P1 の利用量取得） | iOS Spike（`.steering/20260606-ios-screen-time-spike/`）で別途扱う。本ステアリングでは P1 のロジックを定義するが、DeviceActivity の実装は含まない |
| API サーバのクラウド化 | ポストMVP のインフラ判断 |
| Gemini による料金自動チェック（月次バッチ） | 運用ツール。機能実装後に別途構築 |
| LLM による自然文レコメンド生成 | product-requirements フェーズ2 |

## 12. product-requirements 改訂の必要箇所

本ステアリングの実装にあたり、`docs/product-requirements.md` の以下の箇所を改訂する必要がある。

| 箇所 | 現行 | 改訂内容 |

product-requirements 改訂は本ステアリングの実装と同時に行う。
