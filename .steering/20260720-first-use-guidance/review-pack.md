## 決定内容

- iPhoneの既存初回導線は残し、途中から再開できるようにする。
- iPhoneホームにはチェックリストを追加せず、既存の「次に確認すること」で未完了の1項目だけを示す。
- Webは契約0件なら短い初回案内、既存契約ありなら全体チェックリストを示す。
- 棚卸し、支出、見直し、任意の利用状況設定を共通の4段階とする。
- Screen Timeは後から設定でき、「今回は使わない」でも案内を完了できる。
- 進捗は実操作から自動判定し、アカウント単位でiPhoneとWebに共有する。
- 各画面の説明は短い画面内案内とし、操作を遮るツアーを使わない。
- Webの「レコメンド」を利用者向け表示では「見直し」へ統一する。

## 前提・未決事項

### 要ユーザー判断（承認前に解消）

- なし。iPhone初回導線の維持、クライアント別表示量、4段階、自動進捗、任意計測、初回説明、画面名、最小限の進捗保存、中断再開は事前のgrillingで確認済み。

### 設計上の前提（崩れると設計が変わるもの）

- iPhoneが主製品で、Webは任意の補助クライアントである。
- iPhoneの初回説明は、データの扱いを理解してからサインインするために必要である。
- 案内は通常機能を禁止する関門ではなく、いつでも中断・再開・再表示できる補助である。
- Screen Timeは任意であり、計測なしでも料金・更新日等による見直しを利用できる。
- 進捗はアカウント単位の少数項目に限定し、外部行動分析SDKを使わない。
- local modeとcloud modeは同一モデル・APIを使う。

## 影響範囲

- `docs/`への影響 / 更新案:
  - `product-requirements.md`: 初回利用後に迷わないユーザーストーリー、4段階、任意計測、成功条件、最小進捗保存を追加する。
  - `functional-design.md`: 案内機能、`user_guidance_progress`、GET/PATCH API、iPhone/Webの表示差、初回説明、画面名を追加する。
  - `glossary.md`: 「はじめ方」「案内進捗」「この画面について」を追加する。
  - `architecture.md`、`repository-structure.md`、`development-guidelines.md`: 技術選定・配置原則・開発規約を変えないため更新不要。
- 既存コード・機能への影響: iPhoneの初回導線・ホーム・設定・主要画面、Webのダッシュボード・サイドバー・主要画面、API、Prismaへ影響する。契約・支出・見直しの計算ロジックは変更しない。
- マイグレーション・後方互換: 追加テーブルのみ。既存進捗行は不要で、既存契約の有無から安全な初期表示を求める。既存URL`/recommendations`とAPI名は維持する。
- 失敗時の影響: 進捗APIが失敗しても通常機能を止めず、iPhoneは端末内状態、Webは契約データから安全な案内を表示する。

## セルフレビュー結果

| 観点 | 指摘 | 対応 |
|---|---|---|
| PM / UX | iPhoneへ4項目を常設すると、支出や見直しより案内が目立つ | 既存の「次に確認すること」を再利用し、1項目だけ表示する。全体は設定へ分離する |
| PM / UX | WebでiPhoneと同じ説明を繰り返すと、既存利用者を足止めする | 契約0件と既存契約ありで表示を分ける |
| 情報設計 | 「レコメンド」と「見直し」が混在すると、同じ機能だと分かりにくい | 利用者向け表示を「見直し」へ統一し、内部URL・型名だけ維持する |
| iOS | サーバー取得を待たないと初回画面を決められず、通信断で利用不能になる | 最終進捗と未送信固定イベントを端末内に保持し、案内失敗で通常機能を止めない |
| Web / API | 単なるチェック更新APIでは、画面を見ていないのに完了できる | 固定イベントを実際の画面表示・操作からだけ送信し、契約存在はDBから導出する |
| データ整合性 | iPhoneとWebの同時更新で完了状態が戻る可能性がある | 日時イベントを単調・冪等にし、最初の日時を保持する。利用状況だけ明示的な状態遷移にする |
| セキュリティ / プライバシー | 行動分析の名目で契約内容や詳細な閲覧履歴が増える恐れがある | 固定された少数の完了状態だけを保存し、外部SDKと契約内容の複製を禁止する |
| QA | 正常系だけでは、0件、既存利用者、中断、オフライン、再送、別アカウントで破綻する | 各状態とサインアウト後消去、越境拒否を自動・画面テストへ含める |
| アクセシビリティ | 小さい説明文や折りたたみだけでは支援技術利用者が案内へ到達できない | Dynamic Type、VoiceOver、44ポイント操作領域、Webキーボード・200%拡大を受け入れ条件にする |

### 第二意見（フル区分は必須）

- 手段: `grilling` Skillによる利用者視点の設計レビューと、PM・iOS・Web・セキュリティ/プライバシー・QAの別観点によるアドバーサリアル・セルフレビュー。
- 要点:
  - 既存iPhone初回導線を置き換えると、データの扱いを理解してから使う信頼体験を失う。
  - 大きな共通チェックリストはiPhoneの画面を圧迫するため、クライアントごとに表示量を変える必要がある。
  - Screen Timeを初回必須にすると、支出・更新日だけで得られる価値を体験する前に離脱しやすい。
  - Webに既存データがある利用者へ同じ導線を強制すると、iPhoneで完了した作業を繰り返させる。
  - 画面名の不一致と、途中再開できない状態は、チュートリアルを追加しても迷いを再発させる。
- 反映した内容:
  - iPhone初回導線を維持し、計測説明を最初の見直し後の任意手順へ置いた。
  - iPhoneは既存カードの1項目、Webは全体チェックリストに分けた。
  - 案内進捗をアカウント単位で共有し、固定イベントの冪等な再送を設計した。
  - 主要画面名を統一し、画面ごとの短い説明と再表示入口を追加した。
  - 外部SDKを使わず、契約内容を含まない最小進捗だけを保存する方針にした。
