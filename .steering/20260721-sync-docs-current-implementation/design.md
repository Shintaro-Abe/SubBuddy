# 設計 — 現行実装と文書の同期

## 実装アプローチ

### 1. 根拠の優先順位

文書ごとの主張を、次の順で照合する。

1. 実装コード、Prismaスキーマ、migration、iOSの端末内保存構造
2. `.audit/test-status.md`の実施済み試験
3. 後続の承認済みステアリング
4. `docs/`の現行仕様
5. 日付付きステアリング、Obsidianメモ、手順書の過去記述

実装済み、検証済み、配布可能を別状態として扱う。コードがあっても監査証跡がなければ「実装済み・実機未確認」と書く。

### 2. 文書種別ごとの更新方法

| 文書種別 | 更新方法 |
|---|---|
| `docs/` | 現行仕様を本文へ反映し、最終更新日と実装状態表を更新する |
| `manuals/` | 現行UI・操作へ修正し、Markdown変更時は対応HTMLを再生成する |
| `obsidian/` | 元の調査記録を保持し、冒頭の現行性注記と索引を更新する |
| `.steering/` | `.steering/README.md`で優先関係を更新し、実績型文書だけを監査証跡に合わせる。旧要求・設計は保持する |

### 3. 主な照合軸

- iPhone: 3タブ、設定、初回導線、次の1操作、画面説明、Appleボタン寸法、動的操作色。
- Web: 使い方、設定、ログアウト、案内進捗共有。
- Screen Time: 1契約1アプリ、自動開始、変更・解除、認可取消、自動同期3契機、失敗保持、手動再送、通常同期では見直しを再計算しない境界。
- クラウド: Apple認証、セッション、CSRF、テナント境界、案内進捗、利用量削除API。
- 検証状態: Mac・実機で完了した範囲と、7日連続、日付境界、最新変更後のiOS回帰、Archive/codesign等の未確認範囲。

### 4. 検査方法

- `rg`で現行・旧表現、UI文言、API名、状態表現を照合する。
- Markdown相対リンクを機械検査する。
- コードフェンスの偶数性と、変更したHTMLの単一ファイル性・コピー対象数を検査する。
- メール、電話番号、秘密鍵、APIキー形式を合成例の許可範囲を除いて検査する。
- `git diff --check`で差分形式を検査する。

## 変更対象

| 対象 | 予定する変更 | 対応AC |
|---|---|---|
| `docs/*.md` | 現行仕様・状態・構造の同期 | AC-1, AC-7, AC-8 |
| `manuals/README.md`と関連手順 | 索引、現行UI、未確認範囲の同期 | AC-2, AC-3, AC-7, AC-8 |
| `obsidian/*.md` | 現行性注記と索引の同期 | AC-4, AC-7, AC-8 |
| `.steering/README.md`と関連実績文書 | 正本関係、後続上書き、実績状態の同期 | AC-5, AC-6, AC-7, AC-8 |
| `.audit/test-status.md` | 今回の文書監査結果を記録 | AC-7, AC-8 |

具体的な変更ファイルは照合結果に応じて限定する。内容が現行実装と一致する文書は更新しない。

## 影響範囲

- 基本設計文書を更新するため区分はフル。
- 実行コード、API契約、DB、認証、画面動作への影響はない。
- 手順HTMLは対応するMarkdown正本からのみ再生成する。

## 設計上の前提

- 過去文書の不一致は誤りとは限らず、後続変更による履歴である。
- 将来要求と現行実装を混同せず、状態表で分離する。
- 外部サービスの最新仕様調査ではなく、リポジトリ内の現行実装との整合を対象とする。
