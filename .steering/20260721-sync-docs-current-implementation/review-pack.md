## 前提・未決事項

### 要ユーザー判断

- なし。AGENTS.mdの文書管理原則に従い、過去ステアリングの承認内容は履歴として保持する。

### 設計上の前提

- 「現在の実装」は`main`のコード・スキーマを指す。
- 試験状態は`.audit/test-status.md`を超えて更新しない。
- 外部仕様の最新性調査は今回の対象外で、既存手順と実装の対応だけを扱う。

## 影響範囲

- `docs/`: 全6正本文書を照合し、必要なファイルだけ更新。
- `manuals/`: 索引と現行実装に関連する手順を照合。変更したMDに対応するHTMLを再生成。
- `obsidian/`: 全メモの現行性注記を照合。
- `.steering/`: 索引、直近の実績型文書、後続上書き関係を更新。旧要求・設計は保持。
- コード、API、DB、認証、WBS: 変更しない。

## アドバーサリアル・セルフレビュー

| 観点 | 指摘 | 対応 |
|---|---|---|
| PM | 将来要求を実装済みと誤記すると、リリース判断を誤る | 状態表で実装・検証・配布を分離する |
| アーキテクト | 文書を一括置換すると、ADRや過去判断の因果関係を失う | 恒久文書だけを現行化し、履歴文書は注記と索引で上書きを示す |
| QA | コードだけを見て未実施試験まで合格にする恐れがある | 監査台帳に証跡がある範囲だけ状態更新する |
| セキュリティ | 手順更新時に実在の接続情報や識別子を混入しやすい | 合成例だけを使い、機微情報形式を検査する |
| UX | 手順のUI文言が古いと初学者が操作できない | 現行SwiftUI/Next.jsの表示文言と直接照合する |
| 保守性 | 全文書を変更すると差分が大きく、次回も同期不能になる | 一致するファイルは触らず、索引と現行状態表を中心に更新する |

### 第二意見（grillingによる反証）

- 「すべての過去文書を現行文へ書き換える」案は、承認時点の判断を消すため不採用。
- 「最新コミットの説明だけを追記する」案は、API・データ・検証状態の横断齟齬を残すため不十分。
- 「監査台帳を見ず実装有無だけで完了判定する」案は、実機未確認を隠すため不採用。
- 採用案は、正本文書を現行化し、履歴文書は優先関係と実績だけを更新する方式。
