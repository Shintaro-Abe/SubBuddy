# 設計 — 現行実装と利用者向け文書の同期

## 実装アプローチ

### 根拠の優先順位

文書の主張を次の順で照合する。

1. `main`の実装コード、Prismaスキーマ、migration、iOSの端末内状態
2. `.audit/test-status.md`の実施済み試験
3. 後続の承認済みステアリング
4. `docs/`の現行仕様
5. `manuals/`と`obsidian/`の日付付き記述

コードが存在しても監査証跡がなければ「実装済み・対象環境では未確認」と記す。将来要求だけの場合は「未実装」と記す。

### 文書種別ごとの更新方法

| 文書種別 | 更新方法 | 対応AC |
|---|---|---|
| `docs/` | 現行仕様と実装状態を同期し、将来要求を分離する | AC-1〜AC-4, AC-8, AC-9 |
| `manuals/` | 現行UI・操作・判定基準へ修正し、変更したMDからHTMLを再生成する | AC-5, AC-6, AC-8, AC-9 |
| `obsidian/` | 元の調査記録を保持し、冒頭の現行性注記と索引を更新する | AC-7〜AC-9 |
| `.audit/test-status.md` | 文書監査の開始・結果・未確認範囲を記録する | AC-8, AC-9 |

### 重点照合項目

- Web: 320px以上のレスポンシブUI、下部ナビ、「その他」、1列表示、PC表示維持、設定、ログアウト。
- iPhone: 初回案内、3タブ、画面説明、動的操作色、Appleボタン、契約・支出・見直し・設定。
- 認証: Appleサインイン、短期・更新トークン、Cookie、Keychain、端末・セッション失効。
- Screen Time: 1契約1アプリ、自動開始、変更・解除、自動同期3契機、失敗保持、通常同期では見直しを自動再計算しない境界。
- 見直し: 利用日・利用分・単価の保存修正済み範囲と、構造化された代替候補・節約額・確認優先度移行の未完了範囲。
- 通知: 無効化された端末内設定UIと、未接続の許可要求・予約・配信。初回版5種類は要求であり実装済みではない。

### 検査方法

- `rg`とファイル一覧で、現行・旧表現、UI文言、API名、状態表現を照合する。
- 対象Markdownの相対リンクとコードフェンスを機械検査する。
- 変更した手順HTMLについて、Markdown正本との見出し・本文・コピー対象数と、外部アセット非依存を検査する。
- メールアドレス、電話番号、秘密鍵、トークン等の形式を検査し、合成例以外を残さない。
- `git diff --check`で差分形式を検査する。

## データ構造の変更

なし。文書と監査台帳だけを変更し、スキーマ、型、migration、APIインターフェースは変更しない。

## 影響範囲

- `docs/`: 全正本文書とADRを照合し、必要なファイルだけ更新する。
- `manuals/`: READMEと全手順を照合し、変更したMDに対応するHTMLだけ再生成する。
- `obsidian/`: READMEと全メモの現行性注記を照合する。
- `.audit/test-status.md`: 今回の独立した文書監査を実行前後に記録する。
- コード、DB、API、WBS、過去ステアリング: 変更しない。

## 設計上の前提

- 2026-07-21の文書同期後の実装差分は、2026-07-21のiPhone向けWeb UIコミットである。
- 同コミットで更新済みの文書も再照合するが、一致する場合は変更しない。
- 一時的に回復したAppleサインイン障害は原因未確定でコード変更もないため、恒久仕様や修正済み障害として記録しない。
- 外部仕様の最新性ではなく、リポジトリ内の実装と文書の整合を対象とする。
