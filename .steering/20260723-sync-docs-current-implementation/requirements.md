# 要求内容 — 現行実装と利用者向け文書の同期

## 背景・目的

2026-07-21の横断同期後、iPhoneのブラウザで利用するWeb UIが実装され、主要な正本文書と実機確認手順も更新された。現在の`docs/`、`manuals/`、`obsidian/`を改めてコード、DBスキーマ、テスト監査台帳と照合し、実装済み・一部実装・未実装・未確認の区別を最新状態へそろえる。

特に、見直しの構造化出力と通知は将来要求と一部UIが存在する一方、現時点では完成していない。文書だけで完成扱いにせず、実装済みの範囲を正確に記録する。

## 変更する内容

### やること

- `docs/`の要求、機能設計、技術構成、構造、開発規約、用語を現行実装と照合する。
- `manuals/`の索引と手順を現行画面・操作・検証状態へ合わせる。
- Markdown正本を変更した手順は、対応する単一ファイルHTMLも同時に再生成する。
- `obsidian/`の履歴を保ちつつ、現行性注記と索引を更新する。
- iPhone向けWeb UI、初回案内、認証、Screen Time自動同期、見直し、通知の実装境界を重点的に確認する。
- 相対リンク、コードフェンス、生成HTML、廃止済み表現、機微情報形式を検査する。

### やらないこと

- アプリ、API、DBスキーマ、認証、通知、見直し計算を変更すること。
- `wbs/`や過去の`.steering/`を現行仕様へ書き換えること。
- 外部サービスの最新仕様、料金、法令を調査すること。
- 監査証跡のない試験を合格または配布可能と記載すること。
- 実在の契約、利用量、アカウント、メールアドレス、識別子、資格情報を参照・記録すること。

## ユーザーストーリー

- 開発者として、正本文書を読めば現在使える機能と残作業を区別したい。なぜなら、将来要求を実装済みと誤認せず次の作業を判断したいから。
- 利用者・確認担当者として、手順書どおりに現行画面を操作したい。なぜなら、古い画面名や存在しない操作で迷いたくないから。
- 保守担当者として、日付付き技術メモの適用範囲を判断したい。なぜなら、過去の調査記録を現在の仕様と混同したくないから。

- [x] AC-4: 通知は設定UIだけが無効状態で存在し、許可要求・予約・配信は未接続である現状と、将来の5種類の要求を区別する。

## 制約事項

- 現行仕様の正本は`docs/`、実装の根拠はコードとDBスキーマ、試験結果の正本は`.audit/test-status.md`とする。
- 実装済み、対象環境で検証済み、配布可能を別の状態として扱う。
- 内容が現行実装と一致する文書は、更新日だけを変える目的では編集しない。
- 実在PII、実データ、資格情報を読まず、合成例だけを扱う。
