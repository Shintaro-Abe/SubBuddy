## 前提・未決事項

### 要ユーザー判断

- なし。対象は指定された3ディレクトリに限定し、コードやWBSを変更しない。

### 設計上の前提

- 「現在の実装」は`main`のコードとDBスキーマを指す。
- 試験状態は`.audit/test-status.md`を超えて更新しない。
- 外部サービスの最新仕様調査は対象外とする。
- 回復済みで原因未確定のAppleサインイン事象は、恒久仕様や修正済み障害として扱わない。

## セルフレビュー結果

| 観点 | 指摘 | 対応 |
|---|---|---|
| PM | 将来の通知5種類や見直し完成形を実装済みと誤記すると優先順位を誤る | 状態を実装済み・一部実装・未実装に分ける |
| アーキテクト | 過去メモを全文書き換えると判断経緯を失う | Obsidian本文は保持し、現行性注記だけを更新する |
| QA | コードの存在だけで実機確認済みと書く恐れがある | 監査台帳に証跡がある範囲だけ検証済みにする |
| セキュリティ | 手順書から実在の接続情報が混入しうる | 合成例だけを使い、機微情報形式を検査する |
| UX | 現行UIと異なる画面名・ナビゲーションは手順を止める | SwiftUIとNext.jsの表示文言を直接照合する |
| 保守性 | 全ファイルの更新日だけを変えると有効な差分が埋もれる | 内容が一致するファイルは編集しない |

### 第二意見（grillingによる反証）

- 問い「前回同期後の全履歴を再記述すべきか」: 不採用。差分はiPhone向けWeb UIに限定され、全履歴の再記述は過剰である。
- 問い「通知設定のトグルがあるため通知実装済みと書けるか」: 不採用。現行コードでは画面全体が無効で、許可要求・予約・配信が未接続である。
- 問い「見直し画面に注意点が出るため見直し全出力は完成か」: 不採用。スナップショットには代替候補等の専用保存項目がなく、完成条件を満たさない。
- 問い「Appleサインインの一時障害を修正済みと記録すべきか」: 不採用。回復は確認できても原因と修正は確認できていない。
- 反映結果: 状態境界を重点照合項目と受け入れ条件に追加し、差分限定方式を採用した。
