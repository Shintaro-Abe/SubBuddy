# 要求内容 — プロダクト基盤の補強（棚卸し・カタログ運用・信頼・事業方針）

> ステアリング：`20260704-product-foundation`
> 作成日：2026-07-04
> 位置づけ：「役立つ・長く使われる・選ばれる」の分析で見つかった、機能以外のレイヤー（獲得の摩擦・運用・信頼・事業）の欠落6点を、機能／運用ツール／文書に仕分けて落とす親ステアリング。

---

## 1. 背景（欠落6点）

1. 棚卸し（発見）の摩擦：PRD が最重要体験と定める初回棚卸しに、支援機能が手動登録＋あいまい検索しかない。銀行連携（恒久除外）の穴を埋める中間案が未計画。
2. カタログ運用：判定（安いプラン・安い競合・解約導線・利用の性質）はカタログ依存だが、主要50件開始・料金月次チェックは設計のみで未構築。
3. 収益モデルと存続性：docs に収益の記述がない。金融データを預かるアプリは「何で維持されるか」が信頼に直結する。
4. プラットフォーム境界：判定エンジンは Web 単独で成立する構造なのに、iPhone 専用という境界が明文化されていない。
5. 判定への納得フィードバック：判定に「合っている／違う」を返す装置がなく、精度改善のループが閉じない。
6. 信頼の証明：プライバシー設計は強いが、ユーザーに見える証明装置（透明性画面・エクスポート・ロジック公開）が最低限しかない。

## 3. 実装上の設計方針

- N-17 と N-18 は同じ保存先に落とす：どちらも「判定に対するユーザーの応答」なので、判定スナップショットに紐づくフィードバック1テーブルで受け、判定→応答→行動を1クエリで追えるようにする。理由選択肢は合議で特定済みの誤判定源（家族利用・別端末・保険価値）をそのまま使う。
- N-20 は「共有ドキュメントに内部項番を書かない」ルールに従い、「使っていない」「重複で割高」「安いプランがある」等の内容名で書く。煽らない原則のもとでロジックに隠す理由がなく、公開自体が差別化になる。
- v1 の計測3点（棚卸し所要時間・カタログ外率・判定→行動転換）が本書の優先順位を更新する：カタログ外率が低ければ N-14 で足り、高ければ棚卸し支援（N-11〜N-13）を繰り上げる。

## 4. やらないこと

- 銀行・カード API 連携、スクレイピング、自動ログイン（恒久除外）。
- 家族共有の統合管理（恒久除外）。N-18 の「家族が使う」は判定補正のフラグに留める。
- 収益モデルの具体設計（価格・課金形態）。小規模検証の結果を見て別ステアリングで決める。
