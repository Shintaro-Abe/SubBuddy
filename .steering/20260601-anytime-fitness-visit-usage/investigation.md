## 1. 背景・要求

- ユーザーはエニタイムフィットネス（ジム会員サブスク）も SubBuddy の**利用量評価の対象**にしたい。
- ユーザーの明言：**「専用アプリに来館日が記録されている。これが一番正しい情報」**。
- よって理想は「来館日（来館日数）」を利用量シグナルとし、`月額 ÷ 来館日数 = 1回あたりのジム単価`（`cost_per_visit`）で割高判定・レコメンドに乗せること。

### 設計上の論点
現行の利用量モデルは **iPhone Screen Time（DeviceActivity）= アプリ滞在時間**前提。
ジムは「アプリ滞在 ≠ 来館」のため Screen Time では測れない。これは **iCloud+ と同型の問題**
（iCloud+ は時間で測れず「容量」軸 → ジムは時間で測れず「来館日数」軸）。

---

## 2. 調査で判明した事実（出典付き）

1. **来館はすべて記録される（来館履歴は正本）**
   - 入館には必ず**セキュリティキー（キーフォブ／アクセスパス）のスキャン**が必要で、会員に紐づく。
   - アプリの「ご利用履歴」で**過去の入館記録をすべて**（当月／前月／全期間・月別）確認できる。
   - → 入館手段に関わらず全来館がサーバ側に記録される **漏れのない正規ログ**。ユーザーの認識は正しい。
2. **公式 API / エクスポートは存在しない**
   - 来館履歴を取り出す公式 API・エクスポート機能は確認できず。データ不整合時は「会員連携のやり直し（relink）」で同期、という記述のみ。
3. **Apple Health 連携は“来館”を持たない**
   - AF アプリは Apple ヘルスケアと連携するが、それは**活動量・睡眠・心拍の共有**や**ワークアウト記録**（屋外/自宅含む）であり、
     **ジムへの来館（入館）イベントを HealthKit に書き出すものではない**。来館の正本源にはならない。
4. **第三者集約**（FitnessSyncer 等）はあるが、エニタイム来館履歴の公式連携は確認できず。

## 3. 取得手段のクリティカル評価

| 取得手段 | 正確性 | 制約整合（TC-2: ログイン/スクレイピング禁止, PW非保存） | 使用感（手間） | 実現性 | 判定 |
|---|---|---|---|---|---|
| 手動で来館日入力 | ◎ 正本転記 | ◎ | **× 日々の手間（ユーザー却下）** | ◎ | 不採用 |
| ご利用履歴スクショの OCR/AI | ◎ | ◎ | **× 撮影・取込の手間（ユーザー却下）** | △(フェーズ2) | 不採用 |
| 位置情報ベース来館自動検出（ジオフェンス/CLVisit） | △ 近似（場所滞在＝来館とみなす） | ◎ | ◎ 初回設定のみ・日々ゼロ操作 | △ 要 iOS Spike | **有力候補（未確定）** |
| HealthKit ワークアウトを来館とみなす | △ 自宅/屋外運動混在・代理指標 | ○ | △ ワークアウト記録の手間が残る | ○ | 補助のみ |
| 公式 API / エクスポート | ◎ | ― | ◎ | **× 存在しない** | 不可 |
| 自動ログイン / スクレイピング | ◎ | **× 恒久禁止（TC-2／要求7）** | ◎ | × | **永久対象外** |
| iPhone Screen Time | × アプリ滞在≠来館 | ○ | ◎ | ― | ジムに不適 |

## 4. 重要な結論（トレードオフ）

> **「完全自動」かつ「アプリのご利用履歴と完全一致（正本）」を両立する手段は、本プロジェクトの制約内には存在しない。**

- 正本（来館履歴）への自動アクセスは API 不在のためスクレイピング/自動ログインが必要 → **恒久禁止**に抵触。
- HealthKit には来館イベントが無い。
- したがって選択は実質 **「正本だが手動」 vs 「自動だが近似」** の二択。

### ユーザーの意思決定（これまで）
- 「手動入力」「スクショOCR」＝**不採用**（利用者の手を煩わせ、アプリの使用感を損なうため）。
- → 残るのは **「自動だが近似」= 位置情報ベースの来館自動検出**のみ。ただし本採用は**未確定（保留）**。

## 5. 有力候補の詳細：位置情報ベース来館自動検出（未確定）

既存の「iPhone＝センサー」モデルの拡張として位置づく。

| | アプリ系サブスク | エニタイム（ジム） |
|---|---|---|
| センサー | Screen Time（DeviceActivity） | 位置情報（ジオフェンス / CLVisit） |
| 検知 | アプリ滞在 | ジム滞在＝来館 |
| Mac へ送る値 | 集計値（利用日・バケット） | 来館日（集計値）のみ。詳細位置は送らない |

- ジムにジオフェンスを張り、**一定時間滞在（例: 20分超）で来館と自動判定**。バックグラウンドで完全自動・AFアカウント不要。
- セットアップは**一度きり**（地図ピン留め or「今ジムにいる」で現在地登録）。日々の操作ゼロ＝使用感を損なわない。

### 留意点（クリティカル）
- **正本でなく近似**：検知は「電話がジム位置にあった」であり入館ゲート記録そのものではない。自宅/職場が至近だと誤検知余地 → 滞在時間しきい値で抑制。
- **「常に許可」の位置権限**が必要（機微）。送信は日付集計のみでプライバシー訴求と整合。
- **他店舗利用**はホーム店ジオフェンス外だと取りこぼし（複数店ピン留めで緩和）。
- **要 iOS Spike**：ジオフェンス発火・バックグラウンド起動・滞在判定・バッテリーを DeviceActivity Spike と同枠で検証。
