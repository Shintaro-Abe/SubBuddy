# 要求内容 — iPhoneで快適に使えるWeb UI

## 背景・目的

現行Webは幅236pxの左サイドバー、左右56pxの本文余白、2列カードを画面幅に関係なく使うため、iPhoneでは本文幅が不足してレイアウトが崩れる。端末モデルやブラウザごとの専用画面を増やさず、iPhoneのSafari・Chromeから主要機能を迷わず利用できるレスポンシブUIにする。

## やること

- 画面幅に応じてPC向けWebとiPhone向けWebのナビゲーション・配置を切り替える。
- iPhone向けWebの下部ナビゲーションを「ホーム・契約・見直し」の3項目にする。
- 「支出の内訳・更新間近・使い方・設定」を右上の「その他」メニューと各画面内の直接導線から開けるようにする。
- iPhone向けホームでは、今月の支出、未完了の次の操作、更新・見直し、詳しい概況の順で表示する。
- 契約追加は契約画面右上の「＋追加」と、0件時の主ボタンから開始できるようにする。
- 支出の月次推移を、iPhoneでは月・比較バー・金額の縦リストにする。
- 契約・見直し一覧、詳細、フォーム、案内、ログイン、ダイアログを狭い画面向けに再配置する。
- 320〜430px、iPhone横向き、文字拡大、VoiceOver、ソフトウェアキーボード、Safariの安全領域を確認する。
- PC版の静謐なエディトリアル表現と左サイドバーを維持する。

## やらないこと

- Safari、Chrome、iPhoneモデルごとに別HTML・別画面を作ること。
- User-Agentや端末名でレイアウトを分岐すること。
- iPhoneアプリのSwiftUI画面を見た目まで完全に複製すること。
- API、DB、認証方式、集計、見直し判定を変更すること。
- iPad専用UI、Android専用UI、PWA化、通知、オフライン編集を追加すること。

## ユーザーストーリー

- iPhoneのブラウザ利用者として、横スクロールせずSubBuddyの数字と文章を読みたい。
- iPhoneアプリ利用者として、Webでも「ホーム・契約・見直し」の位置関係を覚え直さずに使いたい。
- 契約を登録する利用者として、キーボード表示中も入力・保存・戻る操作を安全に行いたい。
- 見直しを行う利用者として、長いサービス名や理由が押しつぶされず、詳細へ迷わず進みたい。
- PC利用者として、現在の広い画面向けレイアウトを引き続き使いたい。

## 受け入れ条件

- [x] AC-1: 320px、375px、390px、430pxの全対象画面でページ全体の横スクロールが発生しない。
- [x] AC-2: 1024px以上では現行の左サイドバー、色、書体、主要な情報階層を維持する。
- [ ] AC-3: 1023px以下の閲覧画面では「ホーム・契約・見直し」の下部ナビを表示し、現在地を識別でき、Safariのホームバーと重ならない。
- [x] AC-4: ログイン、契約追加・編集、削除確認では下部ナビを表示せず、戻る・取消・保存を44px以上の操作領域で利用できる。
- [ ] AC-5: 全閲覧画面の右上に「その他」があり、支出の内訳、更新間近、使い方、設定へキーボードとVoiceOverで移動できる。
- [x] AC-6: iPhone向けホームは、今月の支出、未完了の次の操作、更新・見直し、詳しい概況の順で読むことができる。
- [x] AC-7: 契約画面は右上の「＋追加」と0件時の主ボタンを持ち、下部ナビや常時浮動ボタンへ追加操作を混在させない。
- [x] AC-8: 支出の月次推移はiPhoneで6か月分の月・比較バー・金額を縦に表示し、今月を識別できる。PCは棒グラフを維持する。
- [x] AC-9: 契約・見直し一覧は1列カードとなり、名称は最大2行、理由は最大2行、金額・更新日は省略せず表示し、カード全体から詳細へ進める。
- [x] AC-10: 契約詳細は概要、契約情報、見直し、利用状況を1列で表示し、編集を上部、削除を末尾の「その他の操作」へ配置する。詳細の名称・理由・メモは全文を読める。
- [x] AC-11: 契約フォームは1列、入力文字16px以上、操作領域44px以上、長い選択肢は折り返し、保存は末尾の全幅ボタンとし、項目別エラーを直下へ表示する。
- [x] AC-12: 画面説明は初回だけ見出し直下へ表示し、閉じた後は「この画面について」だけを表示する。説明と再表示ボタンは同時に表示せず、入力とスクロール位置を失わない。
- [ ] AC-13: ログインと全ダイアログは320pxで収まり、閉じる・取消・主操作がキーボード、VoiceOver、文字200%相当で利用できる。
- [x] AC-14: 長い英数字名、大きな金額、長い理由・メモ、合成200契約でも本文を極端に小さくせず、主要操作と情報を利用できる。
- [ ] AC-15: iPhoneのSafari・Chromeで縦向き・横向き、ソフトウェアキーボード、下部ナビ、安全領域を実機確認する。
- [x] AC-16: API、DBスキーマ、認証・CSRF、集計、見直し判定、既存URLを変更せず、Webの既存自動テストとPC回帰が合格する。

## 制約事項

- 合成契約、合成金額、合成見直しだけをテストに使い、実在PII・資格情報を扱わない。
- ブレークポイントは画面幅と内容に基づき、端末・ブラウザ名に依存させない。
- Web版をブランドの正本とするADR 0013を維持し、操作構造だけをiPhoneアプリと揃える。
- `prefers-reduced-motion`、安全領域、フォーカス可視化、意味のあるDOM順序を維持する。
