# 要求内容 - リリースロードマップ再基準化

## 背景・目的

現行 `wbs/wbs.yml` は、実装履歴、開発ツール、TestFlight、発表準備、配信後バックログが同じ列で管理されている。TestFlight 配信を「リリース」と呼ぶ一方、一般公開までの到達条件がなく、実装完了と実環境での検証完了も区別できない。

このため、SubBuddy のリリースを段階別に定義し直し、考慮漏れをリリースゲート（＝次段階へ進んでよいかの判定条件）と実行可能なタスクへ落とす。

## 変更する範囲

- やること:
  - リリースを「社内実機確認」「TestFlight 小規模検証」「一般公開」の3段階に分ける。
  - 各段階の開始条件、完了条件、中止条件、証跡、責任者を定義する。
  - 既存WBSを「プロダクトリリース」「配信後候補」「開発運用」に分離する。
  - 機能、UI/UX、ブランド・アイコン、品質、セキュリティ・プライバシー、法務・ストア、運用、サポート、計測の漏れを追加する。
  - 先行条件、クリティカルパス、外部依存、見積もり方法を修正する。
  - `docs/` の TestFlight と一般公開の要件差を明文化する。
- やらないこと:
  - この作業内で iOS / Web の機能を実装すること。
  - App Store 一般公開日を、TestFlight の実測前に確約すること。
  - 実在ユーザーのデータ、資格情報、本番DBを参照すること。
  - 配信後候補を根拠なく v1 必須へ格上げすること。

## ユーザーストーリー

- 開発者として、次に着手すべき作業とその先行条件を一意に判断したい。手戻りと見せかけの進捗を減らすため。
- リリース責任者として、配信可否を証跡に基づいて判断したい。重大な漏れを勢いで通さないため。
- テスターとして、問題報告、データ削除、問い合わせの方法が分かる状態で参加したい。
- 利用者として、機微な支出・利用情報の保存範囲と削除方法を理解し、安全に利用を止められるようにしたい。

## 制約事項

- TestFlight 配信は一般公開ではなく、仮説検証を目的とする限定リリースとして扱う。
- Screen Time計測、Web同期、レコメンドは中心価値を支える補助手段とし、それ自体をTestFlightの成功目的にしない。
- 一般公開の必須範囲は、TestFlight の結果と重大不具合の収束後に確定する。
- iPhone からクラウドへ送る利用情報は集計値に限定する。
- しきい値、保持期間、アラート条件、成功・中止基準は設定または運用文書へ外出しする。
- 合成データのみをテスト、seed、スクリーンショット、運用訓練に使う。
- App Store用スクリーンショットにも実在の契約、金額、メールアドレス、端末識別情報を表示しない。
- iPhoneを主製品とする。TestFlight版から主要操作をiPhone単体で完結させ、Mac/Webを利用条件にしない。
- Webも正式クライアントとし、iPhoneと同じクラウドデータを利用できるようにする。ただしTestFlight期間中の初回アカウント有効化は招待済みiPhoneアプリからだけ許可する。
- ローカルDBに引き継ぐ実利用データはないため、`user_local`からクラウドへの移行機能は作らない。開発用合成データも移行しない。
- 課金方式・価格・無料有料境界は今回の認証設計から外し、TestFlightの価値検証後に別ステアリングで扱う。
- 一般公開移行基準の数値はSubBuddy内部の運用情報とし、Appleへ成功基準として提出しない。Appleには実際に収集する操作イベントと任意アンケート情報の種類・目的を申告する。
