# 既存機能の維持監査

## 結論

UIの再編は、既存機能の削除ではなく、利用目的に合わせた配置変更として行う。PRDと機能設計でMVPに定義され、実装済みの機能は、明示的な再承認なしに削除・TestFlight後へ降格しない。

前のUI案では、レコメンドを「見直し」へ統合した説明が不足し、支出内訳・月次推移・単価・iCloud+容量管理が画面仕様から落ちていた。以下のとおり復元する。

## クリティカル監査結果

### 外部TestFlightを止める問題

| 重要度 | 問題 | 根拠 | 必須対応 |
|---|---|---|---|
| 致命的 | iOSへ移植する主要APIにクラウド認証・認可がない | subscriptions / recommendations / spending / renewals / summary / recompute が `getCurrentUserId()` の `user_local` 固定。利用量同期だけがdevice token認証 | Apple本人確認または用途別トークンから `userId` を解決する共通認証境界を実装し、全クエリをテナント限定する |
| 致命的 | 「画面をiOSへ移す」だけでは主要機能を呼べない | iOS用API clientは認証・デバイス登録・利用量同期のみ。契約CRUD、支出、レコメンド、更新一覧の安全なiOS clientがない | iOS用の認証済みAPI client、エラー型、再認証、401復帰、契約・集計・見直しAPIを実装する |
| 高 | レコメンドの再計算契機が移植表にない | 現Webは手動 `RecomputeButton`、容量更新時は明示POST。契約登録・編集・利用量同期後の一貫した再計算仕様がない | 登録・編集・容量更新・同期後の再計算契機と、失敗時の古い結果表示を定義する |
| 高 | 契約フィールドの移植が画面名だけで、項目単位に保証されていない | status / importance / notes / signupChannel / usageType / initialValueAnswer / cancellationUrl / capacity項目が存在 | 下記フィールド対応表をiOSフォーム・詳細・API契約・テストへ落とす |
| 高 | レコメンドの出力項目が「根拠」だけに要約され、詳細フィールドの保証が弱い | observationDays / daysUntilReady / usageDays30d / usageMinutes30d / daysSinceLastUse / confidence / generatedAt 等が存在 | 下記出力対応表どおり、表示・非表示・内部利用を明示する |
| 高 | 利用実績と単価がスナップショット保存時に失われる | `appendRecommendationSnapshot()` が `usageDays30d: 0`、`usageMinutes30d: 0`、`costPerUsageDay: null` を固定保存 | 集計結果を正しく永続化し、既存データの扱いと回帰テストを決めてからiOSへ移植する |
| 高 | 安いプラン・代替サービス・節約額が構造化データとして永続化されない | 計算結果は `RecommendationResult` にあるが `RecommendationSnapshot` に専用項目がなく、画面は解約時の年額だけを表示 | API契約と保存方式を決め、下位プラン・代替案・各節約額を取り違えず表示する |
| 高 | caveat（注意書き）が利用者へ十分表示されない | JSON解析はするが、詳細画面はチップのtitleへevidenceを入れるだけでcaveat本文を表示しない | 注意書きを見直し詳細の本文へ表示し、VoiceOverでも読めるようにする |

### 移植漏れではないが分離が必要なもの

| 項目 | 判定 |
|---|---|
| 請求履歴 `BillingEvent` | DBモデルと合成seedはあるが、専用API・利用者UIはない。完成済み機能の移植対象ではなく、未完成機能として別管理する |
| 契約経路 `signupChannel` | DB・Zodにはあるが現Webフォームから入力できない。完成済みUI機能として数えず、必要性を再確認する |
| 通貨 `currency` | API・DBは3文字コードを持つが現Webフォームは円固定。初期リリースをJPY限定にするなら明記する |
| 判断記録 | 既存MVPにはない追加候補。情報提供の中心価値と分け、TestFlight必須にしない |
| カテゴリ別棚卸しウィザード | 既存MVPにはない追加候補。登録UI改善として個別に評価する |

## 認証機能の移行境界

| 現在 | 配布版の移行先 | 判定 |
|---|---|---|
| `getCurrentUserId()`が常に`user_local`を返す | Apple認証済み主体から内部`users.id`を解決 | 固定ユーザー経路は開発環境だけに隔離 |
| iOSのApple identity tokenを認証・端末登録に使用 | 初回検証後、短期アクセストークンと更新トークンを使用 | identity tokenの通常API使い回しは禁止 |
| 利用量同期だけdevice token認証 | 主計測iPhone1台へ端末同期トークンを発行 | 既存同期方式を所有ユーザー・端末失効へ接続 |
| Web主要画面・APIにクラウドセッションがない | `HttpOnly` CookieのWebセッション | Appleサインイン済みユーザーだけ利用可能 |
| TestFlight招待とWeb利用許可が未接続 | iPhoneで初回有効化した参加者だけWebを許可 | 招待外のクラウド登録を拒否 |
| アカウント削除はApple token再提示でDB cascade | 再認証、全関連データ・セッション削除、Apple認可取消 | 削除専用コード経路も追加 |

ローカルDBに引き継ぐ実利用データはないため、`user_local`からクラウドユーザーへの移行機能は作らない。これは既存機能の削除ではなく、開発用合成データを配布環境へ持ち込まないための環境分離である。

## 既存実装の修復ゲート

次を満たすまで「既存レコメンド機能のiOS移植完了」としない。

- 合成利用記録から算出した利用日数・概算時間が、再計算後のAPI応答と保存スナップショットで一致する。
- 利用日数が0なら単価を作らず、1日以上なら料金と対象期間から一貫して算出する。
- 解約した場合の年額、下位プランとの差額、代替サービスとの差額を別項目として扱う。
- 安い候補の名称・金額・確認日・情報源を表示できる。
- evidence（根拠）とcaveat（注意点）を別々に表示できる。
- 観測中、容量未確認、情報鮮度切れ、該当なしの各状態をAPIとUIで再現する。

### 対応表の読み方

- **維持**: 新版でも同じ利用価値と情報を提供する。
- **統合**: 独立画面をやめるが、機能は別画面内に残す。
- **分割**: 情報量が多いため、利用目的ごとの画面へ分ける。
- **内部化**: API URLや内部IDなど、処理には必要だが利用者が操作する必要のない項目を隠す。
- **廃止**: 機能そのものを削除する。現時点の対応表に、承認済みの廃止機能はない。

## レコメンドを「見直し」に配置する意味

「見直し」はレコメンド機能の表示名であり、機能を単純化して1件だけ出す意味ではない。

### 見直し一覧

- 更新が近い
- しばらく使っていない可能性
- 似た契約が重複している
- より安いプランがある
- より安い代替サービスがある
- 高額な契約が長期間続いている
- 現時点では急いで確認する材料が少ない
- 観測中・情報不足

内部のパターン番号や英語の判定コードは表示しない。ユーザーには内容を表す日本語で示す。

### 見直し詳細

- 現在の確認優先度
- 判定理由と当てはまった根拠
- 対象期間、最終同期、データの鮮度
- 1利用日あたり単価
- 年間支出と年間節約額の目安
- 安いプラン・代替サービス
- 誤判定の可能性や不足情報
- 公式の解約・プラン変更ページ

自動解約は行わず、判断記録もTestFlight必須にはしない。画面の主目的は判断材料の提供とする。

## 追加機能との区別

以下は既存MVP機能ではなく、今回のUI検討で候補に出た追加機能である。

- 判断記録
- 判断の再通知
- カテゴリ別の棚卸しウィザード
- iPhone単体での全機能提供

これらは既存機能の維持とは分けて扱う。特に判断記録は情報提供という中心価値を曖昧にする可能性があるため、TestFlight必須から外す。
