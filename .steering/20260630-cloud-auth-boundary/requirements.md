# 要求内容 — クラウド認証境界設計

## 背景・目的

SubBuddy は現在、ローカルで動く MVP 構成である。次の配布目標は、TestFlight で 20〜50 人程度に配布する小規模検証版であり、Web・API・DB をフルクラウド化する方針を採用した。

この変更では、いきなり PaaS へデプロイしたり iPhone アプリを接続したりする前に、`local mode` / `cloud-testflight mode` / `production mode` の実行モードと、認証・テナント分離の境界を設計する。クラウドに支出・利用状況を保存するため、認証済みユーザーの `user_id` 解決、デバイス同期トークン、Route Handler 以降の共通処理、テナント越え防止のテスト方針を先に固める。

関連する決定:

- `docs/adr/0001-cloud-testflight-is-full-cloud.md`
- `docs/adr/0002-keep-local-as-local-mode.md`
- `docs/glossary.md` の `小規模検証版` / `local mode` / `cloud-testflight mode` / `production mode`

## 変更・追加する機能の説明

### やること

- 実行モード `local` / `cloud-testflight` / `production` の差分と共通部分を定義する。
- Route Handler 以降が同じ形で扱える内部認証モデルを設計する。
- local auth / Apple auth / device token auth の責務境界を設計する。
- 既存 `USAGE_SYNC_TOKEN` をローカル簡易認証として位置づけ直す。
- クラウド版の iPhone 同期は、ユーザーID直指定ではなくデバイス同期トークンで認証する方針を定義する。
- 多ユーザー化に向けて API / repository が必ず認証済み `user_id` で絞り込む方針を定義する。
- テナント越え防止のテスト方針を定義する。
- 必要な永続ドキュメント更新案を整理する。

### やらないこと

- PaaS への実デプロイはしない。
- Apple サインインの実装はしない。
- DeviceActivity / iPhone アプリの本実装はしない。
- Prisma schema のマイグレーションはこの作業では行わない。
- 実ユーザー・実支出・実利用ログなどの PII は扱わない。
- 外部サービスの ID / パスワード保存、自動ログイン、スクレイピングは行わない。

## ユーザーストーリー

- 開発者として、ローカル版とクラウド版を同一コードベースで扱いたい。なぜなら、本番との差分を小さくしつつ開発速度を保ちたいから。
- 小規模検証版のテスターとして、Apple サインインでログインし、Mac ローカルサーバーを用意せずに SubBuddy を使いたい。なぜなら、サブスク登録と iPhone 集計値送信を最小の設定で試したいから。
- 開発者として、iPhone からの同期 API がユーザーIDを直接信じない設計にしたい。なぜなら、テナント越え事故を防ぎたいから。
- 開発者として、Route Handler 以降の処理を認証方式に依存させたくない。なぜなら、local auth / Apple auth / device token auth を差し替えてもドメインロジックを共通化したいから。

## 受け入れ条件

- [ ] AC-1: `local` / `cloud-testflight` / `production` の共通部分と差分が設計に明記されている。
- [ ] AC-2: Route Handler 以降で使う内部認証モデルが定義されている。
- [ ] AC-3: local auth / Apple auth / device token auth の責務と入力・出力が定義されている。
- [ ] AC-4: 既存 `USAGE_SYNC_TOKEN` の扱いが、ローカルMVP用とクラウド版用に分けて整理されている。
- [ ] AC-5: iPhone 同期 API がクライアント指定の `user_id` を信じず、同期トークンからユーザーを解決する方針になっている。
- [ ] AC-6: API / repository が認証済み `user_id` でテナント分離する方針とテスト観点が定義されている。
- [ ] AC-7: 小規模検証版でクラウドに保存するデータと保存しないデータの境界が明記されている。
- [ ] AC-8: Apple サインイン、デバイス登録、デバイス同期トークンの関係がフローとして説明されている。
- [ ] AC-9: 後続の実装順が、Web/API/DB/認証を先に固めてから iPhone クラウド送信へ進む順番になっている。
- [ ] AC-10: 必要な永続ドキュメント更新案と、今回変更しない範囲が明記されている。

## 制約事項

- 実 PII・実支出・実利用ログ・実メールアドレス・資格情報を開発成果物に含めない。
- iPhone から受け取るのは詳細 Screen Time ログではなく集計値のみとする。
- クラウド版でも自動解約、外部 ID / PW 保存、自動ログイン、スクレイピングは行わない。
- クラウド版は正式認証、ユーザー別テナント分離、TLS、削除導線、プライバシーポリシーを前提にする。
- `local mode` は残すが、クラウド版と別プロダクトにしない。
- スコアリング・レコメンドのドメインロジックは認証方式に依存させない。
