# 要求内容 — Screen Time計測ライフサイクルの自動化

## 背景・目的

現行iOS画面では、Screen Timeの許可と契約への計測対象アプリの対応付けに加えて、利用者が計測の開始・停止を操作する。これは内部の監視状態を利用者に管理させる設計であり、「対応付け済みだが停止中」という意味の曖昧な状態や、開始忘れによる利用量欠損を生む。

計測状態をScreen Time認可、契約の存在、計測対象アプリとの対応付けから自動的に決定する。計測対象を変更または解除した場合は、別アプリの利用量が同じ契約の実績として混ざらないよう、確認後に旧対象由来の利用量と見直し結果を削除する。

## 変更・追加する機能の説明

### やること

- Screen Time認可済みの端末で、契約に計測対象アプリを1つ対応付けた時点から自動的に計測を開始する。
- 認可前に対応付けがある場合は、認可完了後の状態照合で自動的に計測を開始する。
- 利用者向けの「計測を始める」「計測を停止」操作と、独立した手動開始・停止状態を廃止する。
- 計測対象アプリを変更する前に、対象契約の過去のScreen Time利用量と、それに基づく見直し結果が削除されることを確認表示する。
- 変更確定後、旧対象の監視停止、端末内の対応表・未送信集計・集計値、クラウドの同期済み日別集計・見直し結果の削除を行い、新対象の監視を自動開始する。
- 対応付けを解除する前にも同じ削除範囲を確認表示し、確定後は監視停止と対象契約の計測データ削除を行う。料金、更新日、メモ等の契約情報は残す。
- 契約削除時は従来どおり、確認後に契約情報、監視、対応表、端末内・クラウドの利用量、見直し結果を削除する。
- アプリ起動時、フォアグラウンド復帰時、認可変更時、契約一覧更新時に状態を照合し、条件を満たす対応付けだけを監視状態へ収束させる。
- 対象変更・解除の途中で通信や削除に失敗した場合、旧データと新データを混在させず、安全に再試行できる状態を保持する。

### やらないこと

- Screen Time認可をSubBuddyの利用条件にはしない。拒否・制限時も料金、更新日、重複等による見直しを継続する。
- Screen Time認可の取消だけでは、過去の利用量や対応付けを自動削除しない。再認可後に既存の対応付けから計測を再開する。
- 旧対象と新対象の利用量を世代別に保持・合算しない。初回版では対象変更時に旧計測データを削除する。
- 詳細なScreen Timeログ、アプリ一覧、FamilyActivityTokenの中身を端末外へ送らない。
- 複数アプリを1契約へ対応付けない。

## ユーザーストーリー

- 契約を管理する人として、Screen Timeを許可して計測対象アプリを選んだら、その場で計測を始めてほしい。内部の開始操作を覚えたくないため。
- 計測対象を選び直す人として、旧アプリの利用量が新アプリの利用量へ混ざらないようにしたい。誤った見直し結果を避けるため。
- 計測をやめる人として、対応付けを解除する1回の操作で、監視とその契約の計測データを片付けたい。停止状態と対応付け状態を別々に管理したくないため。
- Screen Timeを許可しない人として、計測なしでも契約管理と見直しを続けたい。機微な権限を強制されたくないため。

## 受け入れ条件

- [ ] AC-1: Screen Time認可済みかつ有効な契約に計測対象アプリが1つ対応付いている場合、追加の開始操作なしで契約単位の監視が開始される。
- [ ] AC-2: 認可前から有効な対応付けがある場合、認可完了後の状態照合で自動的に監視が開始される。
- [ ] AC-3: 利用者向け画面に手動の計測開始・停止操作がなく、計測状態は認可・契約・対応付けから導出される。
- [ ] AC-4: 計測対象変更の確定前に、削除対象、契約情報は残ること、他契約へ影響しないこと、元に戻せないことが表示される。
- [ ] AC-5: 計測対象変更の確定後、旧対象の監視、端末内対応表・未送信集計・集計値、同期済み日別集計、旧利用量に基づく見直し結果が対象契約に限定して削除され、その後に新対象の監視が自動開始される。
- [ ] AC-6: 対応付け解除の確定前に同じ削除範囲が表示され、確定後は対象契約の監視と計測データだけが削除され、料金・更新日・メモ等の契約情報は残る。
- [ ] AC-7: 契約削除時は事前の対応付け解除を要求せず、契約とそれに属する計測設定・利用量・見直し結果が削除される。
- [ ] AC-8: Screen Time認可が拒否・制限・取消状態でも契約管理と非計測の見直しを利用でき、認可取消だけでは過去の利用量と対応付けが削除されない。
- [ ] AC-9: 対象変更・解除処理は再実行しても安全で、通信失敗・アプリ終了・再起動を挟んでも旧対象と新対象の利用量が混在せず、完了または再試行可能な状態へ収束する。
- [ ] AC-10: クラウドの計測データ削除は認証済み利用者が所有する契約に限定され、別利用者・別契約のデータを削除できない。
- [ ] AC-11: 変更・解除・契約削除後は対象契約の見直しを利用量なしで再計算し、削除済み利用量を根拠として表示しない。
- [ ] AC-12: 自動テストと実機確認は合成データだけを使い、自動開始、変更、解除、契約削除、認可取消、通信失敗、再起動、別契約保持を確認する。

## 制約事項

- 実在の契約、利用量、アプリ選択、認証情報を開発・テスト・証跡に使わない。
- iPhoneから送るのは日別の集計値だけとし、アプリ選択トークンはApp Group内に限定する。
- 計測対象変更・解除では対象契約の計測データを削除するが、他契約と契約情報は保持する。
- 手動開始・停止フラグを新設しない。計測可否は認可、契約、対応付けから導出する。
- 計測しきい値は既存の`AppConstants.thresholdMinutes`（15分/30分/60分/120分）へ集約し、本変更では利用者設定や新しいしきい値を追加しない。保持規則も変更しない。
- `local mode` と `cloud-testflight mode` でAPI契約とドメイン上の削除意味を共通にする。
