# 要求定義（Requirements） — WBS 同期の仕組み

> プロジェクト名 / アプリ名：**SubBuddy**
> ドキュメント種別：作業単位ドキュメント（`.steering/20260602-wbs-sync/`）
> 作成日：2026-06-02
> 関連：`docs/development-guidelines.md`（開発プロセス・Git 規約）、`docs/repository-structure.md`（配置）、`docs/glossary.md`（用語）、`CLAUDE.md`（PII 方針・段階承認）

---

## 1. 本作業のスコープ

SubBuddy の **開発進捗を WBS（Work Breakdown Structure）で管理する仕組み**を導入する。
WBS を Google スプレッドシートで可視化しつつ、**進捗の正本（Source of Truth）はリポジトリ側の機械可読な WBS spec に置き**、必要なタイミングで Claude Code が手動トリガで Sheets へ同期する。

これは **SubBuddy プロダクト本体の機能ではなく、開発プロセスを支える運用基盤（メタ機能）**である。プロダクトのドメインロジック（スコアリング・取り込み等）には影響しない。

---

## 2. 背景・目的

- 開発進捗を体系的に管理し、抜け漏れ・遅延を早期に把握したい。
- WBS の閲覧・共有・ガント的な俯瞰は **Google スプレッドシートの強み**を活かしたい。
- 一方、進捗の更新は **AI（Claude Code）が実装の一次情報（コミット・PR・タスク完了）から起こせる**ようにし、手入力依存と「WBS と実態の乖離」を減らしたい。
- 上記は **AI-DLC（AI-Driven Development Life Cycle）** の思想——計画/進捗をコード近接の機械可読アーティファクトに置き、AI が短サイクル（Bolt）で更新し人間が検証する——と整合する。

---

## 3. 確定した採用方針（本作業の前提）

検討（メリット/デメリット整理・最新調査）を経て、以下を確定方針とする。

| 論点 | 決定 | 理由 |
|---|---|---|
| 同期方向 | **B 寄り：リポジトリ正本 → Sheets 出力**（ビュー） | AI が進捗を自動更新でき、AI-DLC と整合。競合が起きにくい |
| 正本の所在 | **リポジトリ内の WBS spec（Markdown）** | Git で履歴・差分・レビューが効く。AI が編集可能 |
| Sheets の位置づけ | **人間向けダッシュボード / レポートビュー**（生成物） | ガント・共有・フィルタ・スマホ閲覧を活用 |
| 同期の実行 | **自動トリガ（イベント起動）＋ 実行前の必須確認（ask）** で spec → Sheets を upsert | 起動は自動化しつつ、書き込み前に必ず人間が承認。AI-DLC の「AI 実行＋人間検証」と一致 |
| Sheets 連携手段 | **Google 公式 Workspace CLI（MCP サーバ内蔵）を第一候補** | 公式・将来性。動かない場合のフォールバックは design で扱う |
| WBS 列 | **標準 WBS 一式**（§4） | 一般的な進捗管理に必要十分 |

> 「Sheets を主に人間が手編集する」運用は本方針では採らない（Sheets はビューのため、手編集は次回同期で上書きされうる）。手編集主体に変えたくなった場合は別作業として再設計する。

## 4. 用語（本スコープ）

| 用語 | 定義 |
|---|---|
| WBS spec | 進捗の**正本**。リポジトリ内の機械可読な Markdown（配置は design で確定。候補：`docs/wbs.md`）。 |
| WBS ID | 階層番号（`1` / `1.1` / `1.1.1`）。spec と Sheets 行を対応づける**結合キー**。安定・不変。 |
| Sheets ビュー | WBS spec から生成される Google スプレッドシート。閲覧・共有用。 |
| 同期（sync） | WBS spec を読み、WBS ID をキーに Sheets の行を **upsert**（追加・更新）する処理。**自動トリガ（イベント起動）で開始し、Sheets へ書き込む前に必ずユーザーへ確認（ask）**を行う。 |
| 同期トリガ | 同期を自動起動させるイベント。**採用：Bolt / `.steering/*/tasklist.md` 完了時**。検知方式の具体は design で確定。 |
| 確認ゲート（ask） | 書き込み実行の直前に、差分を提示して**必ず**ユーザー承認を求める必須ステップ。承認なしに書き込まない。 |
| Bolt | AI-DLC の短い作業サイクル（数時間〜数日）。完了時が同期の主要タイミング。 |

| 分類（フェーズ） | 設計 / 実装 / テスト | グルーピング |
| タスク名 | 「WBS 同期スクリプト実装」 | — |

| 開始実績 / 終了実績 | — | 実績 |

| 先行タスク | `1.1` | 依存 |
| 成果物 / 備考 | PR リンク等 | — |

## 5. ユーザーストーリー

- **US-1**：開発者として、WBS の全体像と進捗を **Google スプレッドシートで俯瞰**したい。共有・閲覧が容易だから。
- **US-2**：開発者として、進捗の正本を **リポジトリ内に Git 管理**で持ちたい。変更履歴とレビューを残したいから。
- **US-3**：開発者として、Bolt 完了などの節目で **同期が自動的に起動**し、内容を確認・承認するだけで **spec の最新が Sheets に反映**されてほしい。同期を頼み忘れたり手作業でコピーしたりを避けたいから。（→ FR-4 / FR-8）
- **US-4**：開発者として、進捗率・実績を **コミット/PR/タスク完了から AI が提案**し、自分が承認して確定したい。手入力の手間と乖離を減らしたいから。
- **US-5**：開発者として、WBS と `.steering/*/tasklist.md` の今回タスクが **WBS ID で対応づく**ようにしたい。日々の作業と全体計画をつなげたいから。
- **US-6**：開発者として、同期の前に **追加/更新される行の差分を確認**してから書き込みたい。意図しない上書きを防ぎたいから。（→ FR-8 / AC-4）
- **US-7**：開発者として、WBS spec・Sheets に **PII・秘密情報が混入していない**状態を保ちたい。Sheets を安心して共有・公開したいから。（→ NFR-1 / NFR-2 / AC-5 / AC-6）
- **US-8**：進捗を報告する立場として、Sheets 上で **フェーズ別・全体の進捗率（ロールアップ）** を俯瞰したい。状況報告を素早く行いたいから。
- **US-9**：開発者として、スコープが変わったら **spec に WBS 行を追加・修正**するだけで、その変更が次回同期で Sheets に反映されてほしい。計画変更に追従したいから。（→ FR-3）
- **US-10**：開発者として、中止・統合などで **spec から消えたタスクが Sheets で勝手に消えず、アーカイブ/明示の扱い**になってほしい。経緯を失いたくないから。（→ FR-3 で削除時の扱いを定義）
- **US-11**：リポジトリにアクセスしないステークホルダーとして、**共有された Sheets を閲覧するだけ**で最新の進捗を把握したい。開発環境に触れずに状況を知りたいから。
- **US-12**：開発者として、各 WBS 項目が **どのコミット/PR/成果物に対応するか**を辿れるようにしたい（成果物列）。計画と実装の追跡性を保ちたいから。（→ §4 成果物列 / FR-7）

> ストーリーの主要ペルソナは **開発者（編集・同期する人）** と **閲覧ステークホルダー（Sheets を見るだけの人）**。MVP は単一ユーザーだが、Sheets ビューは共有閲覧を想定する。

## 6. 機能要件（FR）

- **FR-1（正本）**：WBS spec をリポジトリ内に機械可読な Markdown で持つ。§4 の標準 WBS 一式の列を表現できること。
- **FR-2（同期方向）**：同期は **spec → Sheets の片方向**を基本とする。Sheets 側の手編集は正本としない。
- **FR-3（upsert）**：同期は WBS ID をキーに Sheets 行を **追加・更新**する。spec から消えた行の扱い（削除/アーカイブ表示）は design で定義。
- **FR-4（自動トリガ＋必須確認）**：同期は**自動トリガ（イベント起動）で開始**する。ただし Sheets へ書き込む前に **必ずユーザー確認（ask）を挟み、承認された場合のみ実行**する。承認なしの書き込み（完全無人同期）は行わない。明示指示（例：「WBS を同期して」）による手動起動も併用可能とする。
  - **採用する自動トリガ：Bolt / `.steering/*/tasklist.md` の完了時**（タスク群が完了に達した節目）。検知方式（Claude Code フック等の具体）は design で確定。コミット/PR マージ・セッション終了などの他イベントは将来拡張とする。
  - 確認ゲートはユーザーが応答できる**対話コンテキスト内**で発火する（応答不能な完全 headless 実行は対象外。→ §9）。
- **FR-5（進捗の起こし）**：進捗率・実績・ステータスは、コミット/PR/`.steering` タスク完了などの一次情報から **AI が提案 → 人間が承認**して spec に反映する。
- **FR-6（連携手段）**：Sheets への書き込みは公式 Workspace CLI（MCP）経由を第一候補とする。導入・認証手順を整備する（design / tasklist）。
- **FR-7（対応づけ）**：WBS ID により WBS spec と `.steering/*/tasklist.md` の今回タスクを相互参照できる。
- **FR-8（差分の可視化）**：同期前に「追加/更新される行」の差分を提示し、人間が確認できる（破壊的更新の抑止）。

## 7. 非機能要件（NFR）

- **NFR-1（セキュリティ／PII）**：WBS spec・Sheets に **エンドユーザーの PII・機微データ（実サブスク契約、請求履歴、メール等）を一切含めない**。扱うのは開発タスクのメタ情報（タスク名・工数・進捗）のみ（`CLAUDE.md` PII 方針）。
- **NFR-2（秘密情報）**：Google 認証情報（OAuth トークン / サービスアカウント鍵）は **コミットしない**。環境変数・`.gitignore` で除外し、`.env.example` はダミーのみ（`development-guidelines.md` §2 / §6.3）。
- **NFR-3（冪等性）**：同じ spec で複数回同期しても Sheets の結果が一意（重複行を作らない）。
- **NFR-4（陳腐化耐性）**：公式 Workspace CLI は Preview 段階のため、**バージョンを固定し追従**する。CLI 仕様への依存箇所を局所化する。
- **NFR-5（最小構成）**：MVP 思想に従い、過剰実装を避ける。まず spec→Sheets を薄く動かし、削除検知・双方向等は必要時に拡張（`repository-structure.md` の「薄く始める」）。
- **NFR-6（プロセス整合）**：本作業は `.steering` の requirements → design → tasklist を段階承認で進める（`CLAUDE.md`）。

## 8. 受け入れ条件（AC）

- **AC-1**：リポジトリに WBS spec が存在し、標準 WBS 一式の列を持つサンプル WBS を表現できる。
- **AC-2**：同期トリガの発火（自動）または明示指示（手動）により、確認・承認を経て WBS spec の内容が Google スプレッドシートに反映される（行が WBS ID で一致）。
- **AC-2b**：同期トリガが発火しても、**ユーザーが承認するまで Sheets への書き込みは行われない**。ユーザーが拒否/中断した場合は書き込みを行わない。
- **AC-3**：同一 spec で再同期しても Sheets に重複行が生じない（冪等）。
- **AC-4**：同期前に追加/更新行の差分が提示され、人間が承認してから書き込まれる。
- **AC-5**：spec・Sheets・コミット履歴のいずれにも PII・秘密情報が含まれない（シークレットスキャン通過）。
- **AC-6**：Google 認証情報がリポジトリに含まれていない（`.gitignore` で除外、`.env.example` はダミー）。
- **AC-7**：WBS spec の更新がコミットされ、Git 差分で進捗の変遷を追える。

---

## 9. 制約事項

- 公式 Workspace CLI は **Preview**。破壊的変更の可能性があり、バージョン固定と動作確認を要する。
- Sheets API には**レート制限**があり、大規模一括更新はバッチ化が必要（design で考慮）。
- 本スコープの自動化は **「自動起動＋必須の確認ゲート」** まで。**確認なしの完全無人同期は対象外**。
- 確認ゲートはユーザー応答を要するため、**応答不能な完全 headless 実行（CI/cron 等からの無人実行）は対象外**（将来拡張）。双方向同期も対象外。
- Sheets 標準では複雑な依存関係・ガント表現に限界があり、必要時はアドオン等を別途検討。

## 11. スコープ外（Non-Goals）

- Sheets を正本とする手編集主体の運用（A/C 案）。
- 双方向同期、**確認ゲートなしの完全無人同期**、CI/CD・cron からの応答不能な headless 自動実行。
- GitHub Issues / Projects との連携。
- ガントチャート専用ツールや PM SaaS の導入。
- プロダクト本体（SubBuddy アプリ）の機能変更。
