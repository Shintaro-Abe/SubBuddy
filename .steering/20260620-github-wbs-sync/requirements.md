# 要求内容：WBS を GitHub Issues / Projects へ同期し、ロードマップ（ガント）で進捗を見る

## 背景

現在、進捗は `wbs/wbs.yml`（正本）を Claude Code が差分計算→確認ゲート→**Google スプレッドシート**へ片方向同期する仕組みで管理している（`/wbs-sync`）。

ここに「GitHub だけでタスク分解とガント風の進捗可視化ができる」という選択肢を加えたい。
検討の結果、以下の方針を採用する（本ステアリングの前提・確定事項）。

- **方式A：`wbs.yml` を正本のまま維持し、GitHub Issues / Projects へ片方向同期する。**
  既存の同期アーキテクチャ（差分→確認→冪等反映）を再利用し、出力先に GitHub を追加する。
- **ガントの水準：GitHub Projects の「ロードマップ・ビュー」で十分とする。**
  開始日／目標日でバーを時間軸に並べる表示までを目標とし、依存の矢印・クリティカルパス・自動スケジューリングは本作業の対象外。

## やること（スコープ）

1. `wbs.yml` の各タスクを GitHub の **Issue** として作成・更新する（冪等）。
   - 親子（`level:1` / `level:2`）を **Sub-issues（親子課題）** として表現する。
2. GitHub の **Project（v2）** を用意し、各 Issue を項目として登録する。
   - 進捗率・状態・フェーズ・開始日／目標日を **カスタムフィールド** として持たせる。
   - **ロードマップ・ビュー**で時間軸表示できる状態にする。
3. 同期は **片方向（`wbs.yml` → GitHub）** とし、`wbs.yml` を唯一の正本とする。
4. 反映前に **確認ゲート**（差分提示→ユーザー承認）を必ず通す。現行 `/wbs-sync` と同じ操作感にする。
5. 既存の Google スプレッドシート同期は**残す**（GitHub は同期先の追加であって置き換えではない）。

## やらないこと（スコープ外）

- GitHub 側で編集した内容を `wbs.yml` へ戻す双方向同期（将来のハイブリッド案として別途検討）。
- 依存関係の矢印・クリティカルパス・自動日程計算などの本格ガント機能。
- PR / Issue クローズからの完了状態の自動逆流（将来検討）。
- エンドユーザーの実データ（PII・機微データ）の取り扱い（本作業は開発タスクのメタ情報のみ）。

## ユーザーストーリー

- 開発者として、`wbs.yml` を編集して同期するだけで、GitHub 上にタスク階層とロードマップが反映され、コードと同じ場所で進捗を確認したい。
- 開発者として、人手で Issue を作り直さなくても、ID をキーに**冪等**に更新（重複作成なし）したい。
- 開発者として、反映前に**追加・更新・件数と対象**を確認してから承認したい（誤反映を防ぐ）。

## 受け入れ条件（AC）

- **AC-1**：`wbs.yml` の全タスクに対応する Issue が GitHub 上に存在する（`level:2` は対応する `level:1` の Sub-issue になっている）。
- **AC-2**：再実行しても Issue が重複作成されない（WBS ID をキーに冪等更新される）。
- **AC-3**：Project のロードマップ・ビューで、開始日／目標日を持つタスクが時間軸上にバー表示される。
- **AC-4**：進捗率・状態・フェーズが Project のフィールドに反映され、ビューで絞り込み／並べ替えできる。
- **AC-5**：反映前に差分（追加／更新／件数・主な対象）が提示され、ユーザー承認なしに書き込みが行われない。
- **AC-6**：同期は片方向で、GitHub 側を直接編集しても次回同期で `wbs.yml` の内容に整流される（正本は yml）。
- **AC-7**：実行に必要な認証情報・トークンがリポジトリにコミットされない（`.gitignore` で除外、PII・秘密の非混入）。
- **AC-8**：既存の Google スプレッドシート同期（`/wbs-sync`）が引き続き動作する。

## 制約事項

- GitHub Projects（v2）と Sub-issues の操作は主に **GraphQL API** が必要（REST だけでは完結しない箇所がある）。
- 認証は既存の `gh` CLI（認証済み: `Shintaro-Abe`）を基本とし、必要なスコープ（`project` 等）を満たすこと。
- 既存方針を最優先する：ローカルファースト、PII・機微データの非混入、しきい値・設定の外出し。
- 既存の同期コード（`wbs/` 配下の `diff.ts` / `sheets.ts` / `sync.ts`）の構造を尊重し、GitHub 用アダプタとして無理なく追加できる形にする。
