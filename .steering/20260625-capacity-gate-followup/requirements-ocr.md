## 1. 背景と位置づけ

容量ゲートの使用容量は現状**手入力**で完結している（プラン容量＝選択式、使用容量＝数値）。
ただし「設定 ＞ iCloud ＞ ストレージ」の数値を毎回手で写すのは面倒で、入力の鮮度（＝値の新しさ）が落ちる主因になる。
そこで **iCloud ストレージ画面のスクリーンショットを撮るだけで使用容量を埋める**OCR（＝画像から文字を読み取る）補助を検討する。

**位置づけ＝あくまで補助**。OCR は手入力を置き換えず、入力コストを下げて鮮度を保つための入口。OCR が使えない/外したときは手入力にいつでも戻れる。

---

## 2. ゴールと非ゴール

| ゴール | 非ゴール |
|---|---|
| ストレージ画面のスクショから「使用容量」「プラン容量」の**候補**を抽出する | スクショ無しで自動取得すること（iCloud に正規 API は無い＝先行調査 §2 で確定） |
| 抽出値を**ユーザーが確認してから**保存する | OCR 値をユーザー確認なしで判定に使うこと（禁止） |
| 手入力と同じデータ経路（容量ゲート）に流す | 画像そのものを保存・外部送信すること（PII・ローカルファースト方針に反する） |

---

## 3. 必須要件（先行調査の反証反映）

1. **3段フロー固定：候補抽出 → ユーザー確認 → 保存**。OCR の**自動確定は禁止**（§6）。誤読を判定に流さない安全弁。
2. **確認前は判定に使わない**：保存時に `confirmedByUser=true` を立てたものだけが容量ゲートの入力になる（[[design-family-scope]] の `CapacityUsageSnapshot.source=ocr` / `confirmedByUser` と整合）。
3. **画像を残さない**：OCR は端末内（iPhone Vision 想定）で完結し、抽出後の画像は保存も送信もしない。アプリに渡すのは**数値とスコープだけ**（集計値のみ送る設計＝CLAUDE.md PII 方針と一致）。
4. **単位の正規化**：「1.2 TB」「850 GB」「12,000 GB」等の表記ゆれを GB に正規化（純粋関数。`domain/capacity/` に置く）。誤った桁・単位は確認画面で必ず人が直せる。
5. **取得源と確認状態を分離**：`source=ocr` でも `confirmedByUser` は別フラグ。判定の重み付けに使う。

---

## 4. 想定フロー（iPhone Vision 主／Web フォールバック）

```
[iPhone 側（本命）]
  1. ショートカット/アプリで「設定＞iCloud＞ストレージ」画面のスクショを選ぶ
  2. 端末内 Vision（OCR）でテキスト抽出 → 数値候補（使用/プラン容量）を切り出す
  3. 候補を画面に出し、ユーザーが確認・修正
  4. 数値＋スコープ（個人/家族全体）だけを Web に送る（画像は送らない）
  5. 容量ゲートの保存経路（PUT → 再計算）に合流

[Web 側（フォールバック・任意）]
  - 画像アップロード→サーバOCRは PII・送信を増やすため初期は不採用
  - どうしても要るなら「ブラウザ内 OCR（端末内処理）」に限定して検討
```

> 本命は iOS ショートカット連携（[[ios-shortcut-launch-signal-research]] の送信経路・`USAGE_SYNC_TOKEN` 認証と同じ口を再利用できる見込み）。実機 HTTP ゲートの検証状況に依存する。

---

## 5. 確認画面の要件（誤読対策の主戦場）

- 抽出した「使用容量」「プラン容量」を**編集可能な状態**で提示（OCR は初期値、最終確定は人）。
- プラン容量は手入力UXと同じ**選択式**に寄せる（50/200/2000/6000/12000 GB）。OCR が外しても選び直せる。
- 桁・単位の誤りが起きやすい点を明示（例：TB↔GB、カンマ区切り）。
- スコープ選択（個人/家族全体）をここで取る（[[design-family-scope]] §6）。
- 確定で `capturedAt=今`・`source=ocr`・`confirmedByUser=true` を立てて保存。

---

## 6. スコープ境界

| やる（このスパイクで要件化） | やらない（別） |
|---|---|
| 3段フロー・確認必須・画像非保持の要件定義 | OCR エンジンの実装（Vision 連携コード） |
| 単位正規化の純粋関数の責務定義 | サーバ側OCR・画像アップロード基盤 |
| スナップショット連携（source/confirmed）の整合 | 請求メール逆引き（`billing_email` source）の取得実装 |

---

## 7. 着手前に潰す不確実性

- **iPhone Vision でストレージ画面の数値が安定して読めるか**（フォント・背景・端末差）。実機スクショ数枚で精度を当たってから実装判断。
- **送信経路の再利用可否**：既存の利用量同期（`USAGE_SYNC_TOKEN`）の口に容量値を相乗りできるか。実機 HTTP ゲートの検証待ち（[[ios-shortcut-launch-signal-research]] と同じ前提）。
- **手入力で足りているか**：OCR の投資対効果。実利用で「手入力が面倒で鮮度が落ちる」事象が実際に起きるかを見てから着手しても遅くない（容量ゲートは手入力で既に成立）。
