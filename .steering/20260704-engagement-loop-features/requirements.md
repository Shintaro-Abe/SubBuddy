# 要求内容 — 継続利用ループの機能群（配信後ロードマップ）

> ステアリング：`20260704-engagement-loop-features`
> 作成日：2026-07-04
> 位置づけ：市場調査（国内利用実態・海外消費者調査・競合）から導いた「役立つ・長く使われる・選ばれる」の要因分析を、機能ロードマップに落とす親ステアリング。個別機能の実装は本書から子ステアリングを切る。

---

## 1. 背景（調査結論の要約）

- ユーザーの継続/解約の判断材料は「総額の実感 → 利用実態と単価 → 代替 → タイミング → やめ方」の5点セット。SubBuddy はパターン判定でこの中核（利用実態・代替・更新時期）を既に持つ。
- ただし「判断材料の質」は3目標のうち「役立つ」にしか効かない。見直しは年数回の低頻度イベントで、見直しアプリは成功するほど用が済む自己解消構造を持つ。長く使われるかは「放置しても価値が維持されるか」で決まる。
- 使われ続ける構造はループで表せる：**検知（イベント）→ 開く（通知）→ 納得する（判断材料）→ 動く（解約導線）→ 記録する（成果台帳）→ 語る（口コミ）**。既存機能は「納得」「動く」に集中しており、両端（検知・通知／記録・成果）が欠けている。
- 選ばれる要因は初回体験（総支出は自己推定の約2.5倍という認知ギャップの提示）・信頼（集計値のみ・資格情報を保存しない設計の訴求）・共有できる成果。

## 2. 機能一覧（ループ工程別）

内部整理用に N 番号を振る。共有用ドキュメント・UI には番号を出さず内容ベースの名前を使う。

| 工程 | 名前（番号） | 内容 | 新規/強化 | フェーズ |
|---|---|---|---|---|
| 検知 | 更新前通知（N-1） | 更新日管理を画面表示からプッシュ通知へ昇格。年額7日前を既定、月額は選択制 | 強化 | v1.1 |
| 検知 | トライアル管理（N-2） | 契約タイプに「トライアル」を追加し終了 N 日前に通知 | 新規 | v1.1 |
| 検知 | 値上げ検知（N-3） | カタログ現行額と登録金額の乖離検知 | 新規 | フェーズ2（優先度低。§4 参照） |
| 開く | 月次ダイジェスト通知（N-4） | 月1回「合計・見直し候補件数・更新予定件数」を通知 | 新規 | v1.2 |
| 納得 | 総額ギャップの提示（N-5） | 棚卸し前に月額の自己推定を1問聞き、棚卸し後に実額と中立に並べる | 強化 | v1.2（計測のみ v1 任意） |
| 納得 | 単価・保険価値の判定根拠組込み（N-6） | 実装済みの1利用日あたり単価と初回1問を判定根拠の文面へ統合。カタログに再契約の容易さを追加 | 強化 | v1.2 |
| 動く | 解約手順ガイド（N-7） | 既存の解約導線 URL に手順ステップ・所要目安を追加。実行は本人（恒久方針） | 強化 | v1.2 |
| 記録 | 行動記録（N-8） | 「解約した／下げた／継続する」をユーザーがマークし、判定→行動をトレース可能にする | 新規 | v1.1 |
| 記録 | 節約台帳（N-9） | 行動記録から累計節約額を算出しダッシュボード常設。事実として控えめに表示 | 新規 | v1.2 |
| 語る | 成果の共有カード（N-10） | 年間実績の共有画像。煽らない原則の範囲で事実のみ | 新規 | フェーズ2 |

## 3. やらないこと

- 銀行・クレジットカード API 連携による自動検出（恒久除外）。発見の手間への対応は領収書メール抽出（フェーズ2既存計画）と棚卸し支援（`20260704-product-foundation`）が担う。
- 解約代行・自動解約（恒久除外）。N-7 は案内まで。
- 不安を煽る「総額ショック」演出。N-5 は推定と実額を中立に並べる実装とする（中立表現の原則）。

## 4. 過去合議との整合（訂正を含む）

- 値上げ検知（N-3）は過去の合議で「入力源がなく形骸化する＝売りにしない」と判定済み。本書でもフェーズ2・優先度低に置く。カタログ料金の月次チェック運用（別ステアリング）が実働してから再評価する。
- トライアル終了日は同じ合議で「手動でも成立＝優先」と判定済み。N-2 を v1.1 に置く根拠。

## 5. フェーズ方針

- v1（現行スプリント）：スコープ凍結。機能は足さず計測3点のみ（ロードマップ・ステアリング参照）。
- v1.1（審査待ち〜配信直後）：N-1・N-2・N-8。配信中の検証データ（判定→行動の転換率）の計測装置を優先する。
- v1.2 以降：N-9 → N-4 の依存順（行動記録が貯まってから台帳、台帳ができてからダイジェスト）。
- v1.1 候補は `20260704-product-foundation` の機能と合わせると膨らむため、配信後の計測3指標で半分に絞る前提とする。
