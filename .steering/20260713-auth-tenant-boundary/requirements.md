# 要求内容 - Apple認証・テナント境界

## 背景・目的

現在のAppleサインインAPIはAppleの本人確認と`users`への登録までは行うが、通常APIで使うセッションを発行していない。契約、支出、見直し、更新の主要APIは`getCurrentUserId()`から固定ユーザー`user_local`を使うため、クラウド配布すると別ユーザーのデータを同じ利用者として扱う危険がある。

この作業では、iPhoneとWebが同じAppleアカウントから同じ内部ユーザーを解決し、すべてのユーザーデータAPIが認証済み`userId`だけを使う境界を作る。`cloud-testflight`と`production`では固定ユーザーへ戻らず、認証設定不備も起動失敗として扱う。

## 変更・追加する機能

### やること

- Apple identity tokenを初回・再認証時だけ検証し、通常API用の短期アクセストークンと更新トークンへ交換する。
- iPhoneとWebで同じApple subjectから同じ内部`userId`を解決する。
- Webは安全なCookie、iPhoneはKeychainでセッションを保持する。
- 更新トークンをローテーションし、再利用検知時は同じトークン系列を失効する。
- 契約、支出、見直し、更新、端末、利用量、アカウント関連APIを認証済み利用者へ限定する。
- セッション一覧、個別失効、全サインアウトと、既存の端末同期トークン失効をAPIとして提供する。
- `local`だけ固定ユーザーを許可し、配布モードでは`user_local`と認証フォールバックを禁止する。
- 未認証、期限切れ、失効、別テナント、トークン再利用を合成データで検証する。

### やらないこと

- Apple以外のログイン、メール・パスワード、復旧用の代替ログインは追加しない。
- オフラインキャッシュ、競合解決、Face IDロック、完全退会の7日待機コードは別作業とする。
- 端末・セッション管理の完成版UIはiPhone全機能移植と設定画面の作業で実装する。本作業ではAPIと最小クライアント接続までを扱う。
- TestFlight参加者の手動管理、管理者パスキー、一般公開環境へのデータ移行は別作業とする。

## ユーザーストーリー

- 利用者として、iPhoneとWebで同じAppleアカウントの契約を見たい。端末ごとに別データになると管理できないから。
- 利用者として、紛失した端末や使い終えたブラウザをサインアウトさせたい。支出情報へのアクセスを止めたいから。
- 利用者として、セッション更新が切れたときに勝手に別アカウントや固定ユーザーへ切り替わってほしくない。別人の情報を表示する危険があるから。

## 受け入れ条件

- [ ] AC-1: iOS Bundle IDとWeb Services IDのどちらで発行された有効なApple identity tokenでも、同じApple subjectは同じ内部`userId`へ解決される。
- [ ] AC-2: Apple認証成功後、15分有効のアクセストークンと一度だけ使える更新トークンが発行され、更新のたびに更新トークンがローテーションする。
- [ ] AC-3: 使用済み更新トークンの再利用を検知すると、同じトークン系列の有効セッションが失効し、以後の更新が拒否される。
- [ ] AC-4: Webの認証情報は`HttpOnly`、`Secure`、`SameSite=Lax` Cookieへ保存され、`localStorage`やJavaScriptから読めるCookieへ保存されない。状態変更APIはOrigin確認とCSRF token検証を通る。
- [ ] AC-5: Webの「このブラウザで保持」は初期オフである。オフではブラウザセッションCookie、オンでは30日未使用または発行から90日の早い方で再認証になる。
- [ ] AC-6: iPhoneの更新トークンはKeychainへ保存され、アクセストークン期限切れ時に1回だけ更新して元のAPI操作を再試行する。更新不能時はApple再認証を求め、固定ユーザーへ切り替えない。
- [ ] AC-7: ユーザーデータを扱うすべてのAPIは認証済みactorの`userId`を使い、未認証・期限切れ・失効済みアクセスを`401`で拒否する。
- [ ] AC-8: URLやbodyで別ユーザー所有のIDを指定しても、取得・更新・削除・再計算・利用量登録ができず、存在を推測させない`404`を返す。
- [ ] AC-9: 利用量同期は有効な端末同期トークンと所有ユーザーを検証し、別ユーザーの契約IDを含むバッチ全体を保存せず拒否する。
- [ ] AC-10: 利用者は有効セッションを最大10件まで作成でき、一覧、個別失効、全サインアウトを行える。上限時は新規セッションだけを拒否し、既存データの閲覧・編集・削除・出力・退会を妨げない。
- [ ] AC-11: 端末同期トークンと更新トークンはサーバーへハッシュだけを保存し、API応答時以外のログ、DB、例外、テスト出力へ平文を残さない。
- [ ] AC-12: `local`モードだけが`user_local`を使用できる。`cloud-testflight`と`production`は認証設定不足、認証情報不足、不正な実行モードで起動または要求をフェイルクローズし、`user_local`へ戻らない。
- [ ] AC-13: サインアウト、個別失効、全サインアウト、アカウント削除により対象セッションと端末トークンが失効し、失効済みトークンの再利用が拒否される。
- [ ] AC-14: Appleサインイン障害時は、障害前から有効な既存セッションだけを最長72時間の設定範囲内で継続できる。新規ログイン、再認証、期限切れ・失効済みセッションは迂回できない。
- [ ] AC-15: 認証・認可テストは2人以上の合成ユーザーを使い、未認証、別テナント、期限切れ、失効、ローテーション競合、再利用、並行更新を自動テストする。
- [ ] AC-16: 既存localモードの主要操作を維持し、cloud-testflightではiPhoneのAppleサインインから契約API呼び出しまでを実環境で確認できる。

## 制約事項

- 実在のApple subject、メール、契約、支出、利用量、トークンをテスト・ログ・証跡へ保存しない。
- iPhoneからサーバーへ送る利用量は日別・バケット単位の集計値に限定する。
- Appleサインインだけを一般利用者の認証方式とし、代替ログインを追加しない。
- TestFlightとproductionのDB、署名鍵、Cookie名、トークン、Apple設定を分離する。
- 認証期限、Cookie名、許可Origin、障害時継続時間は環境設定へ出し、コード内で環境共通の固定値にしない。
