# 要求内容 — iPhone利用者向けメインUI

## 背景・目的

現行iOSアプリは、API URL、内部の契約ID、Device ID、計測・同期操作を直接扱う開発者向け検証画面である。外部TestFlightでは、iPhoneを主製品として、利用者が契約の棚卸しから支出把握、見直し、計測、設定、削除までをMacやWebなしで進められるUIが必要である。

本作業は、承認済みの親ステアリング `.steering/20260711-release-roadmap-rebaseline/` を実装可能なSwiftUI設計へ具体化する。製品スコープや見直し判断の意味は再定義しない。

## 変更・追加する機能の説明

### やること

- 開発者向け `ContentView` を、オンボーディングと「ホーム・契約・見直し」の3タブを持つ利用者向けSwiftUI構成へ置き換える。
- ホーム右上から設定を開き、データの扱い、Screen Time、同期、通知、端末・セッション、問い合わせ、出力、完全退会へ到達できるようにする。
- 契約の一覧、検索、カタログ選択、手動登録、詳細、編集、削除、金額未入力、更新日不足、iCloud+容量、計測設定をiPhoneへ移す。
- 年間・月間支出、カテゴリ別内訳、月次推移、集計から除外した金額未入力契約を表示する。
- 見直し一覧と詳細に、確認優先度、事実、対象期間、データ鮮度、根拠、分からないこと、単価、選択肢、節約額の条件、公式導線を表示する。
- 初回起動からAppleサインイン、最初の契約1件、支出表示、棚卸し継続、必要時のScreen Time説明、最初の見直しまでの一本道と再開導線を作る。
- 空、読込中、通信失敗、権限拒否、同期失敗、情報不足、保存中、削除確認、再認証要求の各状態を利用者の言葉で示す。
- Web版のブランド要素をiOS用トークンへ対応付け、SwiftUI標準操作、Dynamic Type、VoiceOver、動きの軽減を優先する。
- 既存Web/API/iOS機能との対応表と、契約フィールド・見直し出力・再計算契機の移植テスト表を実装中に完成させる。

### やらないこと

- 見直しエンジンの計算ロジック、スナップショット構造、料金カタログ内容そのものの修復。親WBS T-21で扱う。
- 暗号化オフラインキャッシュ、項目単位競合解決、端末・セッション失効基盤の新規実装。親WBS T-23/T-56で扱う。本作業では接続境界と表示状態を用意する。
- 完全退会、Apple認可取消、削除専用コード、データ出力のサーバー基盤。親WBS T-24で扱う。本作業では入口、説明、確認画面を用意し、未接続機能を出荷可能として見せない。
- Face IDアプリロックの基盤実装。親WBS T-58で扱う。本作業ではアプリルートのロック状態を受け取れる構造にする。
- 正式アプリアイコン、App Storeスクリーンショット、掲載文の制作。親WBS T-15で扱う。
- 判断記録、自動解約、課金、外貨換算、請求情報の自動抽出、AI補助。
- Web版の画面・CSSの変更。

## ユーザーストーリー

- 初めて使う人として、何を保存し何を送らないか理解してからサインインしたい。安心して契約情報を登録するため。
- 契約を見直す人として、最初の1件を登録した直後に年間支出の目安を知りたい。入力作業の価値を早く得るため。
- 契約を管理する人として、iPhoneだけで登録・確認・編集・削除したい。MacやWebを利用条件にしたくないため。
- 支出を把握したい人として、合計、内訳、推移、集計に含まれない情報を確認したい。数字の前提を誤解しないため。
- 見直しを考える人として、断定ではなく、分かっている事実と分からない点、選択肢を確認したい。自分で判断するため。
- Screen Timeを許可しない人としても、料金、更新日、重複、プラン情報による見直しを続けたい。権限を利用条件にしたくないため。
- 途中で中断した人として、次回は完了地点から再開したい。同じ入力や説明を繰り返さないため。
- 支援技術を使う人として、文字を最大まで拡大しVoiceOverを使っても主要操作を完了したい。

## 受け入れ条件

- [ ] AC-1: 未サインイン、オンボーディング途中、利用中、再認証要求、将来のアプリロックをルート状態として分離し、利用中は「ホーム・契約・見直し」の3タブ、設定はホーム右上から開ける。
- [ ] AC-2: 初回起動からデータ説明、Appleサインイン、最初の契約1件、年間支出表示、棚卸し継続、必要時だけのScreen Time説明、最初の見直しまで進め、中断後は保存済み地点から再開できる。
- [ ] AC-3: ホームに年間支出、登録件数、更新時刻、次に確認する1件、更新間近、支出の内訳、棚卸し再開を表示し、金額不足時は集計範囲を明示する。
- [ ] AC-4: 契約の一覧、検索、絞り込み、カタログ選択、手動登録、詳細、編集、削除をiPhoneで完了でき、内部IDとAPI URLを利用者へ入力・表示しない。
- [ ] AC-5: 契約詳細で料金、請求周期、更新日、カテゴリ、契約状態、単価、iCloud+容量、計測設定を該当時だけ表示し、情報不足を後から補える。
- [ ] AC-6: 支出の内訳で月額・年額合計、カテゴリ別内訳、月次推移、集計期間、更新時刻、金額未入力による除外を確認できる。
- [ ] AC-7: 見直し一覧と詳細で、確認優先度、根拠、対象期間、鮮度、分からないこと、誤判定の可能性、単価、節約額の条件、代替案、公式導線を表示し、継続・解約を断定しない。
- [ ] AC-8: Screen Timeは価値と保存範囲を説明した後だけ要求し、拒否・未決定・制限・失敗時にも非計測の見直し導線を維持する。契約との対応付けに内部契約IDを入力させない。
- [ ] AC-9: 設定からデータの扱い、Screen Time、同期、通知、端末・セッション、問い合わせ、データ出力、サインアウト、完全退会へ到達でき、未接続機能は有効な操作として表示しない。
- [ ] AC-10: 主要画面ごとに空、読込中、通信失敗、権限拒否、同期失敗、情報不足、保存中、削除確認、再認証要求を定義し、再試行または別経路を示し、入力中・未同期の内容を失わない。
- [ ] AC-11: Web版の色・書体の役割・余白・情報階層・中立トーンがiOSトークンへ対応し、ライト/ダーク、高コントラスト、VoiceOver、最大Dynamic Type、44ポイント以上の操作領域、動きの軽減で主要導線を完了できる。
- [ ] AC-12: iOS 17.4以降、iPhone専用、縦向きで、標準画面・大画面に切れや重なりがなく、キャッシュ済み主要画面は2秒以内、操作への視覚応答は100ミリ秒以内を目標とする。
- [ ] AC-13: PRD・機能設計・Web画面/API・現行iOS機能と新UIの対応表、および全契約フィールド・見直し出力・再計算契機の移植テスト表に欠落または理由のない非対応がない。
- [ ] AC-14: 実装・プレビュー・テスト・視覚証跡は合成データだけを使い、契約名、支出、利用量、認証情報等の実データをリポジトリや証跡へ保存しない。

## 制約事項

- 親ステアリング `.steering/20260711-release-roadmap-rebaseline/` と `docs/` の確定事項を優先し、本作業だけで製品スコープを縮小しない。
- iPhoneからクラウドへ送る利用情報は日次の集計値だけとし、アプリ選択トークンは端末外へ送らない。
- 日本語、日本円、日本時間を初回版の前提とし、金額未入力の契約を登録可能にする。
- しきい値、期間、件数上限、機能提供可否はコードへ散在させず設定またはサーバー応答へ外出しする。
- SwiftUI標準のナビゲーション、フォーム、確認ダイアログ、FamilyActivityPickerを優先する。
- API・別子ステアリングの未完成機能は、合成データのプレビューでは確認してよいが、実ビルドで完了済みと誤認させない。
