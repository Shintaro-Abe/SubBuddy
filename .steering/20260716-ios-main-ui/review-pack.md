## 前提・未決事項

### 要ユーザー判断（承認前に解消）

- なし。画面構成と製品範囲は承認済みの親ステアリングを継承する。

### 設計上の前提（崩れると設計が変わるもの）

- iPhoneを主製品とし、主要操作にWeb/Macを必須としない。
- 主タブは「ホーム・契約・見直し」、設定はホーム右上から開く。
- 見直しは判断材料を示し、継続・解約を断定しない。
- Web版はブランドの正本だが、SwiftUI標準操作とアクセシビリティを優先する。
- 初回版はiOS 17.4以降、iPhone専用、縦向き、日本語、日本円、日本時間とする。
- iPhoneから送る利用情報は集計値だけで、アプリ選択トークンは端末内に残す。
- 開発・テスト・証跡には合成データだけを使う。

## 影響範囲

- `docs/` への影響 / 更新案: 実装後に `docs/functional-design.md` のiPhone画面、状態、API対応を確定内容へ更新する。PRD、architecture、ADR 0013の方針変更はない。
- 既存iOSへの影響: `ContentView` の開発用UIを利用者画面へ置換する。認証・計測・同期サービスは保持し、ViewModel/Repository境界から利用する。
- Web/APIへの影響: Web画面は変更しない。既存APIのDTO拡張や不足APIが必要な場合もWeb互換性とテナント境界を維持する。
- マイグレーション・後方互換: DB変更なし。既存Keychainセッションと計測選択を保持する。API URL手入力は利用者画面から除外する。
- 他ステアリングとの境界: 見直し計算修復、オフライン競合、完全退会基盤、上限、アプリロックは親WBSの別子ステアリング成果を接続する。

## セルフレビュー結果

| 観点 | 指摘 | 対応 |
|---|---|---|
| PM | UIだけ先に完成扱いするとiPhone単体完結を誤認する | API接続と機能移植表をAC-13/T-1/T-16に含め、未接続を完了扱いしない |
| UX | 全機能を3タブへ詰めると最初の行動が分散する | ホームは次に確認する1件を主役にし、深い機能は目的別の子画面へ置く |
| SwiftUI | 1つの巨大View/ObservableObjectは状態競合とテスト困難を招く | 機能別ViewModelとRepositoryプロトコルへ分割する |
| アーキテクチャ | UIからAPIClientを直呼びすると後続の暗号化キャッシュ・競合解決で再設計になる | Repository境界と独立したsave/sync状態を先に設ける |
| QA | 正常系プレビューだけでは権限拒否、同期失敗、再認証、情報不足が欠落する | 状態表と合成FixturesをAC-10/T-2/T-11へ追加する |
| セキュリティ/プライバシー | 読込中の疑似データ、スクリーンショット、ログから支出情報が漏れる | 読込中は機微情報を疑似表示せず、合成データのみをAC-14へ固定する |
| アクセシビリティ | Webフォントと色の再現を優先するとDynamic Typeやコントラストを損なう | システムテキストスタイルと意味色を優先し、実機手動確認をT-13/T-14へ追加する |
| 機能維持 | 支出、単価、容量、代替案、節約条件が主要4画面のモックだけでは漏れる | 画面カタログ、AC-5〜AC-7、移植表T-1/T-16へ明記する |

### 第二意見（フル区分は必須）

- 手段: QA・セキュリティ/プライバシー・保守運用の別観点によるアドバーサリアル・セルフレビュー
- 要点:
  - 見た目の完成とAPI/基盤の完成が混同される危険がある。
  - 通信失敗時に画面を空へ戻すと、入力や利用者の信頼を失う。
  - Screen Time拒否、金額未入力、見直し情報不足は例外ではなく主要経路である。
  - 200契約、最大文字、VoiceOver、ダーク/高コントラストを最後だけ確認すると構造変更が遅れる。
  - 設定に未接続の削除・出力を置く場合、押せる仮ボタンは安全上不適切である。
- 反映した内容:
  - 7段階すべてにゲートを置き、未接続機能を完了扱いしないと明記した。
  - 表示データと読込/保存/同期状態を分離し、失敗時も既存内容と下書きを保持する設計にした。
  - 権限拒否、金額未入力、情報不足を受け入れ条件とテストへ格上げした。
  - 合成プレビューとアクセシビリティ確認を土台から導入し、M6で端末・性能回帰を行う。
  - 未接続の重要操作は有効表示しないことをAC-9へ追加した。
