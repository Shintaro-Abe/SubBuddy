# 機能実装の調査・戦略立案 Skill — 設計（design）

> ステアリング：`20260603-feature-research-skill`
> ドキュメント種別：作業単位ドキュメント（`.claude/.steering/`）
> 作成日：2026-06-03
> 起用ペルソナ：世界一の AI開発オーケストレーター（主）／テックリード・アーキテクト（従）
> 前提：`requirements.md` 承認済み。codex（`codex@openai-codex` v1.0.4 / ログイン済み）実働可能。

---

## 1. 設計方針

- 本 Skill は **手順書（オーケストレーション指示）**であり、Claude Code 本体に「調査フローをどう回すか」を指示する Markdown（`SKILL.md`）として実装する。新規のコード・MCP サーバ・hook は作らない。
- **不変条件（調査・戦略のみ／実装・コミット禁止）を冒頭に置き**、全ステップがこれに従う。
- 既存の能力（`deep-research` / `Explore` / `Plan` / `codex:rescue` / `knowledge-scribe`）を**オーケストレーションするだけ**で、それらの内部実装には踏み込まない。
- 失敗・不在に強い設計（codex 未ログイン等での縮退）。

---

## 2. 成果物（ファイル構成）

```
.claude/skills/feature-research/
└── SKILL.md          # 本 Skill の本体（frontmatter + 手順）
```

- 追加ファイルは作らない（リファレンス類は当面不要）。
- 調査の**成果物（レポート）はプロジェクト直下の専用ディレクトリ `research/` に切り出して格納**する（§6）。Skill 本体（`.claude/skills/`）・ステアリング（`.steering/`）とは物理的に分離する。

---

## 3. SKILL.md の frontmatter 設計

```yaml
---
name: feature-research
description: >-
  アプリ機能の「実装方法」を、実装せずに調査・戦略立案するときに使う。
  外部仕様の出典付き調査（deep-research）→ 既存コード規約の把握（Explore）→
  このリポジトリ向け実装戦略（Plan）→ 反証（codex:rescue）→ 記録（knowledge-scribe）を
  定型フローで回す。コードの実装・変更・コミットは一切行わない。
  「実装方法を調べたい」「どう作るか戦略を立てたい」「/feature-research」のとき起動。
---
```

設計上の注意：
- `description` に**起動トリガー語**（「実装方法を調べたい」「どう作るか」「/feature-research」）と**非該当境界**（実装・コミットはしない）を含め、誤起動・過剰起動を防ぐ。
- 起動は明示（`/feature-research <機能名>`）を基本とし、文脈一致での自動起動も許容する。

---

## 4. 入力（調査対象の指定）

| ケース | 挙動 |
|---|---|
| `/feature-research DeviceActivity 集計値同期` のように**機能名あり** | その機能を調査対象として確定し、フロー開始 |
| 機能名なしで起動 | **対話で1問だけ確認**（「どの機能の実装方法を調査しますか？」）。`docs/functional-design.md` の未実装機能を選択肢として提示してよい |
| 対象が曖昧／広すぎる | スコープを1つに絞るよう促す（例：「サブスク管理全般」→「サブスク自動検出ロジック」へ） |

入力は「調査したい**単一機能のスコープ**」に正規化する。

---

## 5. 実行フロー設計（5ステップ）

> 各ステップは前段の出力を入力に取る逐次パイプライン。**実装・コミットは全ステップ禁止**。

### Step 0（事前ガード）
- 不変条件を自分に再宣言（実装しない）。
- PII ガード確認（§7）。
- codex 実働可否を確認（`codex login status` 相当）。未ログインなら §8 の縮退へ分岐。

### Step 1 — 外部仕様・先行事例の調査（`deep-research`）
- **目的**：対象機能の公式仕様・実装パターン・既知の落とし穴を、出典付きで収集・反証検証する。
- **入力**：正規化した機能スコープ＋着眼点（API 制約、ライブラリ選定、セキュリティ/PII 影響）。
- **出力**：出典付き要約（主要な実装アプローチ候補、制約、リスク）。
- **ガード**：検索クエリ・取得内容に実 PII を含めない。実データを外部送信しない。

### Step 2 — 既存コードの規約・類似実装の把握（`Explore`）
- **目的**：このリポジトリの命名規約・レイヤ構成・類似実装を把握し、「既にある書き方」に合わせる土台を作る。
- **入力**：機能スコープ＋Step1 で出た技術キーワード。
- **対象例**：`apps/web/`（App Router / Prisma / Zod / scoring）、`docs/`（functional-design・architecture・glossary）、`src/config/`（しきい値外出し方針）。
- **出力**：関連ファイル・既存パターン・準拠すべき規約の一覧（read-only）。

### Step 3 — このリポジトリ向けの実装戦略（`Plan`）
- **目的**：Step1（外部）× Step2（自コード）を統合し、**実装はせずに**実装戦略へ落とす。
- **出力（戦略ドキュメントの骨子）**：
  - 採用アプローチと**却下した代替案＋理由**（トレードオフ）。
  - 変更が想定される重要ファイル／新規モジュール（**提案のみ。差分は書かない**）。
  - データモデル・API・型・Zod 境界への影響。
  - PII / ローカルファースト / しきい値外出し方針との整合。
  - リスク・未確定事項・検証方法（後続で `verify`/`run` を使う想定だと明記、ただし本 Skill では実行しない）。
- **ペルソナ**：対象機能に応じて最適なペルソナで判断（例：DeviceActivity なら Swift/SwiftUI、同期 API なら Next.js + Prisma + Zod）。

### Step 4 — 反証（`codex:rescue`）
- **目的**：Plan の戦略を独立した第二視点で**反証・別案提示**し、一案偏重を防ぐ。
- **入力**：Step3 の戦略要約と主要な判断（採用アプローチ・前提）。
- **出力**：戦略の穴・別アプローチ・見落とし。妥当な指摘は戦略へ反映（差分ではなく戦略文の更新）。
- **縮退**：codex 未ログイン時はスキップし、レポートに「反証未実施（信頼度留保）」と明記（§8）。

### Step 5 — 記録（`knowledge-scribe`）
- **目的**：確定した実装パターン・採用理由・却下した代替案を記録し、次回調査で再利用可能にする。
- **出力**：調査レポート（§6 のフォーマット）。

## 6. 成果物（調査レポート）の保存先とフォーマット

### 保存先（調査結果専用ディレクトリに切り出す）
```
research/[YYYYMMDD]-[機能名スラッグ]/investigation.md
```
- **配置**：調査結果は**プロジェクト直下の専用ディレクトリ `research/` に切り出して格納**する。
- **理由**：調査の対象は**アプリ機能の実装方法に限らず、Claude Code の制御・運用設計など横断的**になりうるため、Claude Code 専用領域（`.claude/`）に閉じず、プロジェクト共通の成果物としてトップレベルに置く。
- **分離**：ステアリング（`.steering/` … requirements/design/tasklist の意思決定記録）とは別系統とし、「調査アウトプット」を性格ごとに分離して混在を防ぐ。
- `research/` 直下に「`[YYYYMMDD]-[テーマ/機能名スラッグ]`」単位でサブディレクトリを作成し、1調査=1ディレクトリとする（添付図・補足メモが増えても収まる）。
- 日付はユーザー提示の現在日付を用いる（Skill 実行時に確認）。
- 初回実行時、`research/` が無ければ作成する。

> 補足：将来、調査結果を実装に着手する段階では、別途 root `.steering/[YYYYMMDD]-[機能名]/` を正式に切り、この `investigation.md` を入力とする。本 Skill はそこまでは行わない。

### investigation.md フォーマット（雛形）
```markdown
# [機能名] 実装方法の調査・戦略（実装なし）
> 調査日 / 対象スコープ / 起用ペルソナ / codex反証: 実施 or スキップ(理由)

## 1. 調査サマリ（結論）
## 2. 外部仕様・先行事例（出典付き）   ← Step1
## 3. 既存コードの規約・類似実装        ← Step2
## 4. 実装戦略（採用案）                ← Step3
## 5. 却下した代替案とトレードオフ      ← Step3
## 6. 反証で出た論点と対応              ← Step4
## 7. リスク・未確定事項・検証方針      ← 後続作業への申し送り
## 8. 参考リンク（出典一覧）
```

---

## 7. PII・機微データガード（全ステップ共通）

- `CLAUDE.md` の PII 方針を Skill 冒頭で参照・宣言する。
- **実データ不参照**：本番 DB、`~/.codex`、`.env`、`secrets/`、実領収書メール、実利用ログを読まない／レポートに転記しない。
- **合成データのみ**：例示は合成データで行う。
- **外部送信の最小化**：`deep-research` の検索・`codex:rescue` への入力に、実 PII・資格情報・社外秘を含めない（一般的な技術情報のみ）。
- 触れざるを得ない機微情報が出たら**作業を止めてユーザー確認**。

---

## 8. 縮退動作（codex 未ログイン等）

```
Step0 で codex 実働不可を検知
  → Step4(codex:rescue) をスキップ
  → Step1–3,5 は通常どおり実行
  → investigation.md に「反証未実施（理由）／信頼度留保」を明記
  → 完了報告で「codex にログインすれば反証込みで再実行可能」と案内（! codex login）
```
- `deep-research` / `Explore` / `Plan` / `knowledge-scribe` のいずれかが不在の場合も、可能な範囲で実行し、欠落をレポートに明記（サイレントに省略しない）。

---

## 9. 完了時の振る舞い（実装しないことの明示）

Skill 完了時、ユーザーへ必ず以下を提示する：
- 「これは**調査・戦略立案**であり、**コード実装は行っていない**」旨。
- 成果物パス（`investigation.md`）。
- 実装に進む場合の次アクション（root `.steering/` を正式に切る／`verify`・`run` での実証）— ただし**着手はユーザー判断**。
