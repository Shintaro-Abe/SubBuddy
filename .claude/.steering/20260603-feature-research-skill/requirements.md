# 機能実装の調査・戦略立案 Skill — 要求内容（requirements）

> ステアリング：`20260603-feature-research-skill`
> ドキュメント種別：作業単位ドキュメント（`.claude/.steering/`）
> 作成日：2026-06-03
> 区分：**アプリ機能ではなく Claude Code の運用基盤（開発支援ツール）**。このためリポジトリ root の `.steering/` ではなく `.claude/` 配下に起票する。
> 起用ペルソナ：世界一の AI開発オーケストレーター（主）／テックリード・アーキテクト（従）
> 関連：`CLAUDE.md`（ペルソナ運用・PII方針・ステアリング規約）、`.claude/skills/`（既存 Skill 配置先）

---

## 1. 今回の作業の目的

「**アプリ機能の実装方法を、より詳細に調査・戦略立案する**」フローを、毎回ゼロから組み立てるのではなく、
**再利用可能な Claude Code の Skill として整備**し、好きなタイミングで一発呼び出せるようにする。

合意済みの調査フロー（実装は行わない）を Skill として定型化する。

```
deep-research（外部仕様・先行事例を出典付きで固める）
        │
        ▼
Explore（既存コードの規約・類似実装を把握）
        │
        ▼
Plan（このリポジトリ向けの実装戦略へ落とす）
        │
        ├─ codex:rescue（別案で反証し、設計を頑健化）
        ▼
knowledge-scribe（確定した実装パターン・採用理由・捨てた代替案を記録）
```

### 1.1 最重要の不変条件（Skill の中核ガード）

- **本 Skill は調査・戦略立案のみを行い、アプリのコード実装・変更・コミットは一切行わない。**
- 成果物は「出典付き調査レポート」＋「このリポジトリ向けの実装戦略（コード変更なし）」に限定する。
- `verify` / `run` のような実アプリ起動・動作検証ステップは本 Skill のスコープ外（必要なら別途ユーザーが起動）。

---

## 2. スコープ

### 2.1 今回やること（In Scope）

1. **Skill 本体の作成**：`.claude/skills/feature-research/SKILL.md`（手順・不変条件・PII ガードを記述した運用手順書）。
2. **起動方法の定義**：調査対象の機能名を入力として受け取り、`/feature-research <機能名>` のように明示起動できるようにする。
3. **フロー定義**：上記 5 ステップ（deep-research → Explore → Plan → codex:rescue → knowledge-scribe）を、各ステップの目的・入力・期待アウトプットとして SKILL.md に明文化する。
4. **codex の導入（セットアップ）手順のドキュメント化**：本フローは `codex:rescue`（反証ステップ）に依存するが、現状 **Codex は未ログイン**で実働しない。導入手順・前提・確認方法をドキュメント化し、Skill 実行前のセットアップとして整備する（§7 参照）。
5. **PII・機微データガードの組み込み**：`CLAUDE.md` の PII 方針（実データ不参照・合成データのみ・外部送信禁止対象）を Skill 内で必ず参照・遵守させる。
6. **成果物の保存先ルール**：調査レポートの出力先（例：`.claude/.steering/[YYYYMMDD]-[機能名]-research/` への investigation.md 等）を定義する。
7. **「実装しない」ことの明示**：Skill 完了時に「これは調査・戦略であり実装は別作業」である旨をユーザーへ提示する。

### 2.2 今回やらないこと（Out of Scope）

- **調査フロー自体の実行**（特定機能の実際の調査）— Skill 整備が目的。実行は Skill 完成後に別途行う。
- **アプリ機能のコード実装・変更**（不変条件のとおり）。
- **Plugin 化・外部配布**（`.claude-plugin/` 等でのパッケージング）— 将来検討。今回は単一リポジトリ内の Skill に留める。
- **スラッシュコマンド（`.claude/commands/`）版の併設** — まず Skill 版のみ。必要なら別ステアリングで追加。
- **CI・hook との連携**（自動起動の hook 化など）— 今回は手動起動のみ。

## 3. ユーザーストーリー

- US-1：開発者として、「この機能どう実装すべきか」を調べたいとき、`/feature-research <機能名>` を呼ぶだけで、合意済みの調査フローが毎回同じ品質で再現されてほしい。
- US-2：開発者として、調査の過程で**誤って実装やコミットが始まらない**ことを保証してほしい（調査・戦略立案のみ）。
- US-3：開発者として、調査結果が出典付きで、かつ「このリポジトリの既存コード規約に当てはめた実装戦略」まで落ちた状態で得たい。
- US-4：開発者として、確定した実装パターン・採用理由・捨てた代替案が記録され、次の機能調査で再利用できるようにしたい。
- US-5：開発者として、PII・機微データ方針が調査中も自動的に守られることを前提にしたい。

## 4. 受け入れ条件（Acceptance Criteria）

- AC-1：`.claude/skills/feature-research/SKILL.md` が存在し、Claude Code の Skill として認識・起動できる。
- AC-2：SKILL.md 冒頭に「**調査・戦略立案のみ／実装・コミット禁止**」の不変条件が明記されている。
- AC-3：SKILL.md に 5 ステップのフロー（deep-research → Explore → Plan → codex:rescue → knowledge-scribe）と、各ステップの目的・入力・期待アウトプットが定義されている。
- AC-4：SKILL.md が `CLAUDE.md` の PII・機微データ方針を参照し、実データ不参照・合成データのみ・外部送信禁止を遵守させる記述を含む。
- AC-5：調査対象の機能名を入力として受け取れる（引数または対話での確認）。
- AC-6：成果物（調査レポート）の保存先ルールが定義されている。
- AC-7：本ステアリングの作成・Skill 整備の過程で、アプリ機能コードへの変更が一切発生しない。
- AC-8：codex の導入（セットアップ）手順がドキュメント化され、未ログイン状態から `codex:rescue` が実働可能になるまでの前提・手順・確認方法が明記されている。
- AC-9：codex 未ログインのまま Skill を実行した場合の縮退動作（警告のうえ反証ステップをスキップ、または導入を促す）が定義されている。

## 5. 制約事項

- **配置**：アプリ機能ではないため、ステアリングは `.claude/.steering/` 配下に置く（root `.steering/` と混在させない）。Skill 本体は `.claude/skills/feature-research/`。
- **規約遵守**：`CLAUDE.md` のステアリング運用（requirements → design → tasklist を 1 ファイルずつ承認）に従う。
- **既存 Skill との非干渉**：`aws-architecture-review` / `drawio` / `pre-commit-secret-scan` 等、既存 `.claude/skills/` に影響を与えない。
- **依存**：参照する `deep-research` / `Explore` / `Plan` / `codex:rescue` / `knowledge-scribe` は現環境で利用可能であること（不在時の縮退動作も design で検討）。
- **PII**：本ファイル含め、実在の個人データ・資格情報を一切記載しない。

## 7. codex の導入（セットアップ）要件

本フローの反証ステップ（`codex:rescue`）が依存する Codex を、未導入相当（未ログイン）状態から実働可能にするための要件。

### 7.1 現状（2026-06-03 実測 → セットアップ完了）

| 構成要素 | 状態 | 備考 |
|---|---|---|
| マーケットプレイス `openai-codex` | ✅ 登録済み・最新 | Source: GitHub `openai/codex-plugin-cc`。キャッシュ更新済み |
| Claude Code プラグイン `codex@openai-codex` | ✅ 導入済み・有効 | v1.0.4 / project scope。Skills 10 / Agent 1（codex-rescue）/ Hooks 3 |
| Codex CLI 本体 | ✅ 導入済み | `codex-cli 0.135.0`（PATH 上に存在） |
| Codex 認証（ログイン） | ✅ **ログイン済み** | `Logged in using ChatGPT`（2026-06-03 完了）。`codex:rescue` 実働可能 |

> 結論：`codex-plugin-cc` 由来のプラグイン・CLI・認証はすべて整い、**`codex:rescue`（反証ステップ）は実働可能**。
> §7.3 の縮退動作は「将来、別環境で未ログインの場合のフォールバック仕様」として保持する。

### 7.2 導入手順（要件）

1. **ログイン（ユーザー操作）**：`codex login` は対話認証のため Claude Code 側からは実行しない。ユーザーがセッションで `! codex login` を実行する前提とし、その手順をドキュメント化する。
2. **疎通確認**：`codex:setup` Skill（ローカル Codex CLI の準備状況を確認）で ready 状態を検証する手順を明記する。
3. **レビューゲート方針**：`codex:setup` が持つ stop-time review gate の有無を、本プロジェクトでどう扱うか（任意）を記載する。
4. ログイン情報・トークン等の**資格情報は一切リポジトリに記載・コミットしない**（`CLAUDE.md` PII 方針）。`~/.codex` は実データ扱いとし参照・コミットしない。

### 7.3 未導入（未ログイン）時の縮退（要件）

- Skill 実行時に codex 未ログインを検知した場合、**反証ステップ（`codex:rescue`）をスキップ**し、その旨と「導入手順（§7.2）」をユーザーへ提示する。
- 反証スキップでも、残り 4 ステップ（deep-research → Explore → Plan → knowledge-scribe）は成立させ、調査・戦略立案を完了できること。
- 縮退で反証を欠いた場合は、レポートにその旨（信頼度の留保）を明記する。
