# ターミナルの出力をマウス選択コピーすると日本語が文字化けする問題を回避する

> 現行性注記（2026-07-15）：本書はClaude Codeで観測した履歴であり、現在利用するCodex CLIでの再現確認ではない。同じ症状が出た場合の切り分け手順としてのみ使う。

## TL;DR

- ターミナルで動くフルスクリーン型のCLIツール（画面全体を描き換えるアプリ）の出力を、マウスドラッグで選択コピーすると日本語が化けることがある。原因はツールがマウス操作を横取りし、OSC 52（＝ターミナルがクリップボードへ文字を渡す仕組み）経由でコピーする際にUTF-8を取り違えるため。
- 「どこに貼っても化ける」ならクリップボードの中身が壊れている証拠。貼り付け先の設定ではなくコピー時点の問題。
- 回避策は「マウスの横取りを避けてターミナルのネイティブ選択でコピーする」（macOSのVS Codeなら Option を押しながらドラッグ → コピー）。確実版は出力をファイルに書いてエディタ画面でコピーする。

## 症状 / Symptoms

VS Code の統合ターミナル（＝VS Codeの画面下部に出るターミナル。ここではDevContainer〔開発用のコンテナ環境〕に接続し、ホストはmacOS）で、ターミナル内で動くCLIツール（このプロジェクトの例ではコーディング支援エージェントの Claude Code）の出力をマウスでドラッグ選択してコピーすると、日本語が次のように化ける。

```
（元）これで → （化け）æ ã ã ã ã » æ  のような Latin-1 風の並び
```

特徴的なのは、**貼り付け先がブラウザ・メモ帳・チャットツールのどれであっても、一律に同じように化ける**こと。半角英数字だけのときは問題が起きず、日本語などマルチバイト文字（＝1文字を複数バイトで表す文字）を含むときだけ発生する。

## 再現条件 / Reproduction

- 環境（このナレッジの観測時）
  - ホスト OS: macOS
  - エディタ: VS Code（統合ターミナル、Remote/DevContainer 接続）
  - コンテナ内ロケール（＝文字コードの地域設定）: `en_US.UTF-8`
  - ターミナル内で動くCLIツール: Claude Code（フルスクリーン描画モードで動作）
- 手順
  1. フルスクリーン描画モードのCLIツールを統合ターミナルで起動し、日本語を含む出力を表示させる。
  2. その日本語部分をマウスでドラッグ選択する。
  3. コピーして任意の場所に貼り付ける。
  4. 日本語が `æ…` のような並びに化ける。

> 注: 「フルスクリーン描画モード」とは、画面全体を1枚の絵のように描き換える表示方式（＝普通のスクロールではなく専用画面に描く）。多くのフルスクリーン型ツールはこのときマウス操作を自分で受け取る（横取りする）。

## 影響範囲 / Impact

- 日本語などマルチバイト文字を含む出力のコピー＆ペーストが実質的に使えなくなる（化けた文字を毎回手直しする必要が生じる）。
- データそのものが壊れるわけではない（ターミナルの表示や元のファイルは正常）。あくまで「マウス選択コピーという経路」だけの問題。
- 半角英数字のみの出力には影響しない。

## 調査の流れ / Investigation

症状から原因を絞り込んだ考え方を残す（同種のトラブルに応用できるため）。

1. 仮説: コンテナ／サーバ側の文字コード設定が悪いのでは？
   - 検証: コンテナ内で `locale` コマンドを実行。`LANG=en_US.UTF-8` 等、UTF-8で正常だった。
   - 判定: 棄却。サーバ側の文字コード設定は問題なし。

2. 仮説: 貼り付け先（ブラウザやチャット）の文字コード設定の問題では？
   - 検証: 複数の貼り付け先で試したが、**どこに貼っても同じように化ける**。
   - 判定: 棄却。貼り付け先ごとに違うなら貼り付け先の問題だが、一律に化けるということは、**クリップボード（＝コピー内容を一時的に保持する場所）に入る時点で、すでにバイト列〔＝文字を表す数値の並び〕が壊れている**と判断できる。これがこの調査の決め手。

3. 仮説: コピー経路（マウス選択）が壊れたバイト列を渡しているのでは？
   - 検証: 化けパターン `æ…` は、UTF-8のバイト列を Latin-1（＝1バイト1文字の別の文字コード。ISO-8859-1）として読んでしまったときの典型的な見た目。つまり「正しいUTF-8を、別の文字コードで読み直した」二重の取り違え（＝mojibake〔文字化け〕）が起きている。
   - 判定: 採用。コピー経路でエンコード（＝文字コードの変換）を誤っている。

## 根本原因 / Root Cause

フルスクリーン描画モードのCLIツールは、画面全体を描き換えるためにマウス操作をアプリ側が横取り（キャプチャ）する。これはVS Code公式も「ターミナル内のアプリがマウスイベントモードを有効にすると、マウス操作はターミナルではなくアプリ側へ送られる」と説明している（[VS Code: Terminal Basics](https://code.visualstudio.com/docs/terminal/basics)・参照日 2026-06-07）。

この状態でマウスドラッグ選択すると、選択テキストは **OSC 52**（＝ターミナルにクリップボードへ文字列を渡させるエスケープシーケンス〔＝制御用の特殊な文字列〕の一種）を通ってクリップボードに入る。Claude Code のこのOSC 52経路にUTF-8を取り違える不具合があり、壊れたバイト列がクリップボードに入る。

これは Claude Code 本体側の既知の不具合で、公式 issue #42417 で報告され「not planned（修正予定なし）」として閉じられている（[GitHub Issue #42417](https://github.com/anthropics/claude-code/issues/42417)・参照日 2026-06-07）。ユーザーの設定ミスではない。

> プラットフォーム差（理解の助けとして）: 上記issueの報告はWindows環境で、UTF-8がCP932（＝Windowsの日本語コードページ。Shift-JIS系）として読まれ `これで → 縺薙ｌ縺ｧ` のように化ける例。macOSのVS Code統合ターミナルでは同じOSC 52原因でも、UTF-8がLatin-1として読まれ `æ…` 系の見た目になる。**化け方は環境で変わるが、原因の経路（OSC 52でのエンコード取り違え）は共通**。なお、本ナレッジで観測したmacOS側の `æ…` 化けは観測事実であり、それがissue #42417と同一の不具合に由来する点は同一経路・同一回避策が効くことからの推定（confidence: high）。

## 解決策 / Solution

### 対処1（手軽・推奨）: マウスの横取りを避けてネイティブ選択でコピーする

アプリのマウス横取りを回避し、ターミナル（VS Code）側のネイティブな選択でコピーすると、正しいUTF-8でコピーできる。VS Code公式は「ターミナルの選択は Alt キー（macOSでは Option キー）を押しながら操作することで強制できる」と説明している（[VS Code: Terminal Basics](https://code.visualstudio.com/docs/terminal/basics)・参照日 2026-06-07）。

- macOS の VS Code 統合ターミナル: **Option（⌥）を押しながらドラッグして選択 → Cmd（⌘）+ C でコピー**。
- これを安定させる設定（VS Code側）:

```json
// VS Code の settings.json
{
  "terminal.integrated.macOptionClickForcesSelection": true
}
```

  この設定を有効にすると、Option+ドラッグが常に「選択」として扱われる（macOSでこの設定を有効にして初めてOption強制選択が使えるようになる、と公式に明記）。
- 一般的なターミナル（Windows Terminal など、Windows/Linux）では **Shift を押しながらドラッグ**でネイティブ選択になることが多い（issue #42417 の回避策としても「Shift+ドラッグ」が案内されている）。

#### 設定をどこに書くか（VS Code Remote / DevContainer 特有の重要ポイント）

`terminal.integrated.*` は **VS Code 本体（UIを描くホスト側）の機能**。VS Code の設定は次の優先順位で後のものが勝つ。

```
既定 < ユーザー設定（ホスト） < リモート(Machine)設定 < ワークスペース設定
```

- DevContainer に接続中は、コンテナ内の **リモート(Machine)設定** `~/.vscode-server/data/Machine/settings.json` に書くと、**コンテナの中（＝CLIツールやエージェント）から編集でき、接続中ウィンドウのターミナルに即反映**できる。`terminal.integrated.*` は Machine スコープで上書きできるため有効。
- ただし **DevContainer をリビルド（再構築）すると、この Machine 設定は再生成されて消えることがある**。恒久的にしたいなら次のどちらかが確実。
  - ホスト mac のユーザー設定（コマンドパレットで `Preferences: Open User Settings (JSON)`）に書く。
  - リポジトリのワークスペース設定 `.vscode/settings.json` に書く。ただし**コミットするとチーム全員に適用される**点に注意（個人の好みかチーム共通かを判断する）。
- 設定変更後はターミナルを開き直すか、コマンドパレットで `Developer: Reload Window` を実行して反映する。

### 対処2（確実・保険）: ターミナルの描画を介さずコピーする

ターミナルの描画とOSC 52を一切通さなければ、化けは100%起きない。

- CLIツールの出力をファイルに書き出し、そのファイルを **VS Code のエディタ画面で開いてコピー**する。エディタ経由のコピーは常に正しいUTF-8になる。
- 大量の出力をやり取りしたいときにも有効（公式も大きな入出力はファイル経由を推奨している）。

## 恒久対策 / Prevention

- マルチバイト文字を含む出力を頻繁にコピーする運用なら、対処1の設定（`macOptionClickForcesSelection: true`）を**ホストのユーザー設定に入れておく**と、リビルドで消えず安定する。
- チームでDevContainerを共有していて全員に効かせたい場合は、`.vscode/settings.json` にコミットする（個人差が出ない）。
- 「コピーは Option+ドラッグ（または Shift+ドラッグ）」を開発メモ／オンボーディング資料に1行残しておくと、新メンバーが同じ罠を踏まない。
