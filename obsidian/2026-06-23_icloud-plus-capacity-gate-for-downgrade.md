# iCloud+ の容量を「安いプランがある」判定の安全ゲートにする設計判断

> 現行性注記（2026-07-15）：容量ゲートの決定と実装は有効である。本文のコード行番号は作成時点の目安で、現在はシンボル名で参照する。外部TestFlightの利用者向け表示は「ダウングレード推奨」ではなく、事実・不足情報・選択肢を含む確認優先度へ移行する。

## TL;DR

- SubBuddy はサブスクを「利用の性質（usage_type）」で分類し、性質ごとに判定方法を変える。iCloud+ は時間軸（使っていない判定）が効かない `capacity`（容量ベース）型である。
- iCloud+ の使用容量は全サブスク共通の必須情報ではないが、「安いプランがある（下位プランへのダウングレード）」判定を**安全に**出すための**ゲート条件**としてのみ必要になる。
- 新しい判定パターンは作らない。既存の P3「安いプランがある」判定に、iCloud+ のときだけ容量チェックの安全弁を足す。容量は新しい主張ではなく、判定の確度を上げる「証拠」として使う。

## 背景・文脈 / Context

- **プロジェクト**：SubBuddy（サブスク見直し支援、ローカルファースト）。判定エンジンは P1〜P6 のパターン判定方式（純関数）。実装は `apps/web/src/domain/scoring/computeRecommendation.ts`。
- **検証日**：2026-06-23。
- **意思決定の母体**：Claude Code × Codex の 2 者議論が 2026-06-23 に収束した結論。先行調査は `research/20260621-icloud-capacity-acquisition/investigation.md`、本件ステアリングは `.steering/20260623-icloud-plus-capacity-gate/requirements.md`。
- **PM 市場調査の確定事項**：iCloud+ の固有価値は「Apple 自身が助けてくれない『下位プランで足りる／足りないの安全な適正化判断』」にある。利用量だけでの解約判定は業界前例がなく、iCloud+ では時間軸そのものが判定軸にならない。

### 利用の性質（usage_type）と判定方法の対応（前提知識）

SubBuddy はサブスクを「どう使われるか」で分け、判定方法を変える。iCloud+ がなぜ時間軸で測れないかは、この分類を押さえないと誤る。

| usage_type | 例 | 判定方法 |
|---|---|---|
| 能動利用（前面）`active_foreground` | 画面で操作するアプリ | 利用日数＋最終利用日で「使っていない」を判定（時間軸） |
| 能動利用（背景）`active_background` | 音楽再生 | 起動のみ検出 |
| 能動利用（別端末）`active_other_device` | PC / TV 専用 | 検出できない |
| 受動利用 `passive` | クラウド同期・VPN | 稼働している限り利用中 |
| 権利保有 `entitlement` | Amazon Prime 等 | 時間では測らない |
| **容量ベース `capacity`** | **iCloud+ 専用** | **容量という別軸で判定。時間軸の「使っていない」判定は適用しない** |

> 罠：iCloud+ を「能動利用」と取り違えると、「最近開いていない＝使っていない」と誤判定する。iCloud+ は裏で同期し続けるサービスであり、時間軸の不使用判定は意味をなさない。判定軸は「容量に余剰があるか」である。`computeRecommendation.ts` は `usageType === "capacity"` のとき時間軸パターン P1 を適用しない分岐を既に持つ（コード参照：型定義 16〜22 行）。

## 検討した選択肢 / Options Considered

| 選択肢 | 概要 | Pros | Cons | 採否 |
|---|---|---|---|---|
| A. 容量専用の新パターンを追加 | `capacity` 用に独立した判定パターンを新設（2026-06-21 調査の初期案） | 容量ロジックが 1 か所に集まる | 分類体系が複雑化。「安いプランがある」という主張は P3 と重複。判定パターンの増殖 | 却下（後述の決定で上書き） |
| B. P3 にゲート条件を足す | 既存「安いプランがある（P3）」に、iCloud+ のときだけ容量の安全弁を追加 | 新パターン不要。容量＝「主張」ではなく「確度を上げる証拠」として正しく位置づく | P3 の分岐に usage_type 依存が入る | **採用** |
| C. 容量を全サブスク必須入力にして単純比較 | 使用容量 < 下位プラン容量 で機械的に提案 | 実装が単純 | 取得摩擦が高く登録離脱。楽観的すぎて同期停止・後悔を招く | 却下 |

## 決定 / Decision

**新しい判定パターンは作らず、既存の P3「安いプランがある」判定に、iCloud+（`capacity` 型）のときだけ容量チェックという安全弁（ゲート条件）を足す。** 容量は新しい主張を生むのではなく、既存のダウングレード提案を「安全に断定できるか」を判断するための証拠として使う。

> 設計の進化の記録：先行調査 `research/20260621-icloud-capacity-acquisition/investigation.md` §4.3 では「容量専用パターンの追加」を提案していた。2026-06-23 の 2 者議論で「P3 への安全弁」へ収束し、容量専用パターン案は撤回された（選択肢 A → B）。本ドキュメントの決定が最新であり、調査ファイルの当該記述に優先する。

### 出し分けは 3 段（容量データの有無・鮮度で変える）

| 容量データの状態 | 提示 | 強さ |
|---|---|---|
| なし | 「もっと安い下位プランがあります。使用容量を確認すると、下げられるか判定できます」 | 断定しない発見プロンプト（容量確認を促す） |
| あり＋鮮度OK＋安全バッファ込みで収まる | 「現在の使用容量なら下位プランで足ります。月額／年換算で見直し余地」 | ダウングレード提案（断定） |
| あり＋鮮度切れ（古い） | 「前回確認時点では足りそうでした。今の使用容量を再確認しましょう」 | 弱める（再確認を促す） |

「安全に収まる」の判定式（しきい値は設定外出し）：

```
使用容量 + 安全バッファ ≤ 下位プラン容量
安全バッファ = max(固定GB, 下位プラン容量 × 一定割合)
```

> 安全バッファの考え方（`max(5GB, 下位容量 × 10%)` 程度）は**設計判断・経験則**であり、確定値ではない。反証条件：実利用で「バッファが厚すぎてダウングレード可能なケースを取りこぼす／薄すぎて同期停止を招く」が観測されたら見直す。具体値はコードに直書きせず `apps/web/src/config/scoring.ts` に追加する（既存方針との整合。同ファイルは Zod 検証つきでテストから差し替え可能）。

## 根拠 / Rationale

- **なぜゲートにするのか（Why）**：容量を知らずにダウングレードを勧めると、下位プランに収まらない場合に同期・バックアップ・新規保存が止まり、ユーザーが後悔する。Apple 自身はこの「下位プランで足りるか」を教えてくれない。ここに SubBuddy の固有価値があるため、容量を「安全確認の関門」に据える。
- **なぜ新パターンを作らないのか**：ユーザーへの主張は「安いプランがある」で P3 と同一。新パターンを作ると同じ主張が二重になり、分類体系が複雑化する。容量は「新しい主張」ではなく「既存主張を断定してよいかを決める証拠」なので、P3 の確度を上げる安全弁として組み込むのが正しい責務分担（単一責任・分類体系の単純さの維持）。
- **なぜ必須入力にしないのか**：使用容量の取得摩擦は高く、オンボーディング必須にすると登録離脱を招く。下位プラン候補があり、ユーザーの動機が立った瞬間（just-in-time）に取得を促す。
- **設計原則との接続**：判定軸を usage_type ごとに独立させる（関心の分離）。しきい値は設定外出し（CLAUDE.md 方針）。証拠が無い場合は断定せず「発見プロンプト」に落とす（過信を避ける＝安全側に倒す設計）。

### 正確性の罠（必ず守る）

判定・文言で誤った断定をしないためのルール。いずれも調査で確定済み。

| ルール | 理由 |
|---|---|
| 容量から価格を一意に決めつけない | 値上げ・家族共有の割り勘・Apple One 内包・既存契約で実効額が変わる。プラン名だけで月額を断定しない |
| 年額は「月額×12の目安」と明示する | iCloud+ は月額課金のみ。実在の年額プランのように見せない（※ 2026-06-23 時点、Apple 公式は月額請求のみと記載） |
| 「データが消える」と断定しない | Apple 公式の表現は「同期・バックアップ・新規保存が止まる可能性」。データ削除ではない。変更前に Apple の画面で確認、と案内する |
| 無料枠 5GB は定数でコードに保持し、ユーザーに入力させない | 誤入力源を断つ。5GB は無料枠、最小の有料プランは 50GB（※ 2026-06-23 時点） |

## 影響と副作用 / Consequences

- **良い影響**：iCloud+ のダウングレード提案を、同期停止リスクを避けつつ安全に出せる。容量未入力でも「確認を促す」発見プロンプトで見直しのきっかけは提供できる。
- **悪い影響・コスト**：P3 の分岐に usage_type 依存と鮮度判定が入り、テスト観点が増える（収まる／収まらない／鮮度切れの境界、安全バッファのしきい値差し替え）。
- **今後の監視点**：
  - 容量の取得は手動入力が主。正規 API は存在しない（調査で確定）ため、判定品質は信頼度（ユーザー確認済みか）・鮮度（いつの値か）・スコープ（個人／家族全体）に依存する。鮮度管理が品質の主戦場。
  - 家族共有時の「本人分／家族全体」の取り違えは他メンバーを容量不足にし得るため、usageScope / costScope を分離して扱う（本ドキュメントのスコープ外。後続スパイク）。

## How to apply（実装時にどう使うか）

1. **新パターンは追加しない。** `MatchedPattern` の列挙（`computeRecommendation.ts` 33〜38 行：`P1`〜`P6`）に `P7` 等を増やさない。
2. **P3 の発火箇所にゲートを足す。** P3 は `cheaperPlan` があると発火する（`computeRecommendation.ts` 241〜248 行付近、`Decision.consider_downgrade` を出す）。iCloud+（`usageType === "capacity"`）かつ下位プラン候補があるときだけ、容量の安全弁を通す：
   - 容量なし → P3 を断定せず「容量確認を促す発見プロンプト」に弱める。
   - 容量あり＋鮮度OK＋`使用容量 + 安全バッファ ≤ 下位プラン容量` → P3 を断定（ダウングレード提案）。
   - 鮮度切れ → P3 を弱め「再確認を促す」。
3. **しきい値（鮮度日数・安全バッファ）は `apps/web/src/config/scoring.ts` に追加。** 直書きしない。テストは config 差し替えで境界・鮮度・バッファを検証（既存 `config/scoring.test.ts` のパターンに倣う）。
4. **時間軸 P1〜P6 には干渉させない。** `capacity` 型は時間軸判定（P1）を既に適用除外している。容量ゲートは P3 の出し分けに閉じる。

## 既知の限界 / Open Questions

- 容量→価格の逆引きは税込み・通貨・国別階層差で曖昧になり得る。曖昧なときは確定させずユーザー確認に回す。
- 家族共有スコープの本格モデリング、スクリーンショット OCR、値上げアラートは本決定のスコープ外（別スパイク）。
- 「正規 API が将来も無い」ことは不在証明であり論理的には証明できない。実装着手前に StoreKit 2 等で再確認の余地（先行調査の申し送り）。
