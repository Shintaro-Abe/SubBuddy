## 0. SubBuddy の本質（この文書全体の前提）

SubBuddy は、登録されたサブスクリプションに対して「継続／様子見／解約」のレコメンドを **SubBuddy 自身が定量的に判断して出す**アプリ。ユーザーに判断を委ねるのではなく、数値に基づいて判定を出すことが価値。

## 2. レコメンドの判定に使う5つの手段

| # | 手段 | 役割 | SubBuddy が出せるレコメンドの例 |
|---|---|---|---|
| 1 | **契約事実**（料金・更新日・重複・カテゴリ） | 全サブスクの基盤スコア | 「同カテゴリに重複があります → 解約検討」 |
| 2 | **Shortcuts 自動記録**（アプリ起動回数・最終起動日） | 利用有無の唯一の客観データ | 「30日間起動ゼロです → 解約検討」 |
| 3 | **サービス知識ベース**（代替案・無料プラン・ダウングレード先） | 代替・ダウングレードの具体提案 | 「無料プランがあります → ダウングレード検討」 |
| 4 | **初回登録時の1問**（「なくなったら困るか」） | ユーザーの価値判断の初期信号 | 「困らないと回答＋高額 → 解約検討」 |
| 5 | **契約期間の長さ**（登録日から自動算出） | 惰性更新リスクの検出 | 「12ヶ月以上継続＋他の証拠あり → 惰性リスク加点」 |

## 3. 定量スコアリング公式（Gemini × Codex 合議で設計）

各サブスクに 0〜100 点の「解約スコア（cancel_score）」を付与する。スコアが高いほど解約推薦に近い。

### 3.1 サブスコアの定義

**A. コスト効率スコア（0〜30点）**

代替サービスとの料金比較から算出。

```
alt_ratio = 月額料金 / 代替サービス月額料金

cost_score = min(30, max(0, (alt_ratio - 1.0) × 30))

例：
  代替より50%割高（alt_ratio = 1.5） → 15点
  代替の2倍以上（alt_ratio = 2.0）   → 30点（上限）
  代替データなし                       → 0点
```

**B. 重複スコア（0〜25点）**

同カテゴリ内の重複件数から算出。

```
duplicate_score = min(25, (duplicate_count - 1) × 12)

例：
  重複なし（自分だけ）       → 0点
  同カテゴリ2本（重複1件）   → 12点
  同カテゴリ3本（重複2件）   → 24点
  同カテゴリ4本以上          → 25点（上限）
```

**C. 契約の積み重なりスコア（0〜20点）**

長期継続と累計支出から算出。

```
duration_sub = min(10, months / 3)
  例：6ヶ月以上 → 10点（上限）、3ヶ月 → 5点

spend_sub = min(10, cumulative_spend / 10000)
  例：累計¥50,000 → 5点、累計¥100,000 → 10点（上限）

longevity_score = duration_sub + spend_sub
```

**D. 利用頻度スコア（0〜25点、Shortcuts データあり時のみ）**

```
freq_sub = min(15, max(0, (30 - opens_30d) / 2))
  例：月30回起動（毎日） → 0点、月0回 → 15点

recency_sub = min(10, days_since_open / 3)
  例：30日未起動 → 10点、15日 → 5点、当日 → 0点

usage_score = freq_sub + recency_sub
```

Shortcuts データ不在時：usage_score = 0。合計を75点満点→100点に正規化。

### 3.2 総合スコアと判定

```
cancel_score = cost_score + duplicate_score + longevity_score + usage_score

── 更新日による緊急度補正 ──
更新7日以内   → cancel_score × 1.5
更新14日以内  → cancel_score × 1.2

── 判定 ──
65点以上  → 解約検討
40〜64点  → 様子見
39点以下  → 継続
```

**安全弁**：Shortcuts で直近14日以内に利用確認済みなら、スコアが65点を超えても「解約検討」ではなく「様子見」に留める（使っている証拠がある以上、解約を断定しない）。

### 3.3 具体例（判定テーブル）

| サブスク例 | 月額 | 代替比 | 重複 | 継続月 | 起動/月 | 最終起動 | スコア | 判定 |
|---|---|---|---|---|---|---|---|---|
| Spotify（毎日使用） | ¥980 | 1.0 | 0 | 3ヶ月 | 25回 | 2日前 | 6点 | 継続 |
| Netflix（放置） | ¥1,490 | 1.5 | 1件 | 12ヶ月 | 0回 | 45日前 | 82点 | 解約検討 |
| 動画B（たまに） | ¥500 | 0.8 | 1件 | 6ヶ月 | 5回 | 10日前 | 30点 | 継続 |
| SaaS（高額・重複） | ¥3,000 | 1.8 | 2件 | 18ヶ月 | 3回 | 25日前 | 73点 | 解約検討 |
| クラウド（Shortcuts無） | ¥1,300 | 1.5 | 1件 | 24ヶ月 | — | — | 68点 | 解約検討 |

## 4. Shortcuts データ不在時に「解約検討」を出せる条件

Shortcuts データがない場合でも、以下の**全条件が揃えば**「解約検討」を出せる（Gemini × Codex 合意）。

```
AND条件（すべて満たす）：
  ① 代替より50%以上割高（alt_ratio ≧ 1.5）
  ② 同カテゴリ重複2件以上
  ③ 契約12ヶ月以上
  ④ 年間節約額 ¥10,000 以上
```

1つでも欠ければ「様子見」止まり。証拠不足で解約を断定しない。

---

## 5. SubBuddy が出すレコメンドの形式

単に「解約」と出すだけでは不十分。**根拠となった数値を必ず添える**。

```
推奨：解約検討
根拠：
  ・同カテゴリに重複する契約があります（Disney+）
  ・代替より50%割高です
  ・30日間の起動が0回です
  ・契約12ヶ月で累計¥17,880支払い済みです
不足している情報：
  ・利用データは未連携です（Shortcuts設定で精度が上がります）
具体アクション：
  ・解約する → 年間¥17,880 の節約
  ・ダウングレード → 年間¥8,400 の節約
```

Codex の核心指摘：**スコアの数値より「節約の具体額」の方がユーザーの行動を起こさせる**。「解約スコア: 72点」より「解約すれば年間¥17,880節約」の方が行動喚起力が高い。

## 6. Shortcuts 自動記録の実装方式

### 6.1 概要

iPhone の標準アプリ「ショートカット」のオートメーション機能を使い、特定のアプリを開いた事実を SubBuddy のAPIサーバに自動送信する。**iOS アプリの開発は不要**。

### 6.2 構成

```
iPhone（ショートカット）
  → インターネット経由で POST
    → SubBuddy の API サーバ（クラウドまたはMac）
      → DB に保存
```

- iPhone と Mac が同じ Wi-Fi にいる必要はない。API サーバをインターネット上に立てれば、外出先からでも送信できる。
- ポストMVP でクラウド化（SaaS）する方針と整合する。

### 6.3 送信するデータ

| パラメータ | 型 | 値の例 | 説明 |
|---|---|---|---|
| `subscriptionId` | string | `sub_abc123` | SubBuddy がサブスク登録時に発行する ID |
| `date` | string | `2026-06-08` | ショートカット実行時の日付（YYYY-MM-DD） |
| `used` | boolean | `true` | 固定値。起動した＝使った事実 |
| `usageBucket` | string | `1m_plus` | 固定値。起動の事実のみで時間は不明なため最小バケット |
| `source` | string | `ios_shortcut` | データの取得元を示す識別子 |

リクエスト例：

```json
POST /api/usage/daily

{
  "items": [
    {
      "subscriptionId": "sub_abc123",
      "date": "2026-06-08",
      "used": true,
      "usageBucket": "1m_plus",
      "source": "ios_shortcut"
    }
  ]
}
```

この形式は既存の `usageDailyBatchSchema`（`apps/web/src/schemas/usage.ts`）にそのまま適合する。`source` に `ios_shortcut` を追加するだけで Screen Time 経由のデータと区別できる。

### 6.4 ID とサブスクリプション名の紐付け

ショートカットから送られるのは `sub_abc123` という ID だけ。SubBuddy は DB の `subscriptions` テーブルで照合する。

```
subscriptions テーブル（既存）
┌────────────┬──────────┬────────┐
│ id         │ name     │ ...    │
├────────────┼──────────┼────────┤
│ sub_abc123 │ Netflix  │ ...    │
│ sub_def456 │ Spotify  │ ...    │
└────────────┴──────────┴────────┘
```

- このテーブルは外部からダウンロードするものではない。**ユーザーがサブスクを登録したときに1件ずつ作られる**。
- サブスクリプション名はユーザーが入力する。
- ID は SubBuddy が自動発行する。
- 紐付けのために新しいテーブルは不要。既存の `subscriptions.id` をそのまま使う。

### 6.5 ユーザーが行う設定手順（1サブスクにつき1回）

**推奨方式：QR コード**

① SubBuddy の Mac 画面（ブラウザ）でサブスク詳細を開く
② 「利用記録を自動化する」ボタンを押すと QR コードが表示される
③ iPhone の標準カメラで QR コードを読み取る
④ ショートカットアプリが開き、API の URL と ID が入った状態で設定画面が表示される
⑤ 対象アプリ（例：Netflix）を選んで「完了」を押す

以後、Netflix を開くたびに**バックグラウンドで自動的に**記録が送信される。ユーザーの追加操作は不要。

**QR コード方式に iOS アプリは不要**。QR コードは Mac のブラウザに表示し、iPhone の標準カメラで読み取る。

### 6.6 制約

| 制約 | 内容 |
|---|---|
| サブスクごとに設定が必要 | 全アプリを一括設定する方法はない |
| 記録できるのは「開いた事実」だけ | 何分使ったか、何を見たかは記録されない |
| iOS 15.4 以降が必要 | それ以前は実行のたびに確認ダイアログが出る |
| API サーバが必要 | ショートカットからの HTTP リクエストを受ける API サーバがインターネット上に必要 |

## 7. カテゴリ別の判定閾値（Codex 提案）

利用頻度が低くても価値が高いサービス（バックアップ・保険等）があるため、カテゴリ別に閾値を変える。

| カテゴリ | 様子見にする起動なし日数 | 解約検討にする起動なし日数 |
|---|---|---|
| 動画・音楽・ニュース | 15日 | 30日 |
| フィットネス・学習 | 21日 | 45日 |
| クラウド・バックアップ | 60日 | 120日＋代替あり |
| 業務ツール | 30日 | 60日＋低価値回答 |
| 季節性サービス | 90日 | 180日 |

## 8. サービス知識ベースの設計（Gemini × Codex 合議）

### 8.1 概要

サブスクの代替サービス・料金プランの情報を管理する仕組み。スコアリングの「コスト効率スコア（A）」の入力となる。

### 8.2 情報の取得元

**手動テーブルが主軸**。LLM は補助（候補提案のみ。スコアには弱く反映）。

理由：料金の正確性が最優先。LLM のハルシネーション（＝事実と異なる情報の生成）は、ユーザーの金銭的判断を誤らせるため致命的。

MVP では主要50件のサービスから開始し、段階的に拡充する。

### 8.3 データ構造

```json
{
  "id": "netflix",
  "display_name": "Netflix",
  "aliases": ["netflix", "ネットフリックス", "ネトフリ"],
  "category": "video_streaming",
  "plans": [
    {
      "name": "広告つきスタンダード",
      "monthly_price": 790,
      "verified_at": "2026-05-01"
    },
    {
      "name": "スタンダード",
      "monthly_price": 1590,
      "verified_at": "2026-05-01"
    }
  ],
  "alternatives": [
    { "service_id": "disney_plus" },
    { "service_id": "amazon_prime_video" }
  ]
}
```

### 8.4 ユーザー入力とカタログの紐付け

登録時に自由入力ではなく、**サービスカタログから選択させる**（あいまい検索付き）。

```
① ユーザーが「ネトフリ」と入力
② aliases であいまい検索 → 候補「Netflix」が表示される
③ ユーザーが選択 → matched_service_id = "netflix" として保存
```

カタログにない場合：
- 「○○を新しいサービスとして登録する」を選べるようにする
- `matched_service_id = null` で保存
- カテゴリだけ選んでもらい、カテゴリ中央値で `alt_ratio` を仮計算
- 代替サービス比較は対象外。他のスコア（重複・契約期間等）は通常通り計算

### 8.5 料金の鮮度管理（半自動：Gemini 自動チェック＋月1回承認）

#### 手動と自動の仕分け

| 分類 | 作業 | 頻度 |
|---|---|---|
| 完全自動 | 陳腐化の検出・信頼度低下の計算 | 常時 |
| **自動→人間が承認** | **料金の変更検知・新プランの検出** | **月1回** |
| 手動（初回のみ） | サービスの新規追加・カテゴリ設定・代替関係の定義 | 随時 |
| 手動（まれ） | カテゴリ体系の見直し・代替関係の変更 | 年数回 |

#### 月1回の更新フロー

```
① 自動：Gemini（Web検索付き）がカタログ全件の料金を調べる
     → 「Netflix 日本 現在の料金プラン一覧」等をサービスごとに検索
② 自動：取得した料金と DB の現在値を比較し、差分を検出
③ 自動：差分があれば開発者に通知（GitHub Issue 等）
     → 例：「Netflix スタンダード ¥1,590 → ¥1,790 の可能性。出典: netflix.com/signup」
④ 手動：開発者が通知を見て正しいか確認
⑤ 半自動：承認 → DB に反映（verified_at を更新）
```

| ステップ | 自動/手動 | 内容 |
|---|---|---|
| 料金の取得 | 自動 | Gemini が Web 検索で最新料金を取得 |
| 差分の検出 | 自動 | 現在の DB の値と比較 |
| 変更の通知 | 自動 | 差分があれば開発者に通知 |
| 内容の確認 | **手動** | 開発者が通知を見て正しいか判断 |
| DB への反映 | 半自動 | 承認操作で反映（`verified_at` を更新） |

#### 陳腐化の自動処理

各プランに `verified_at`（最終確認日）を持たせ、時間経過で信頼度を自動低下させる。

```
確認から6ヶ月以内   → そのまま使用（信頼度 100%）
確認から6ヶ月超過   → 信頼度を 70% に低下
確認から12ヶ月超過  → 信頼度を 50% に低下＋「情報が古い可能性があります」と表示
```

スコアへの反映：

```
effective_alt_ratio = alt_ratio × 信頼度
```

古い情報で誤った解約レコメンドを出すリスクを抑える。

#### 手動でしかできないこと

| 作業 | 理由 |
|---|---|
| 新しいサービスのカタログ追加 | サービスの存在・カテゴリ分類は人間の判断が必要 |
| 代替関係の定義（Netflix↔Disney+等） | 「何と何が代替か」は機械的に決められない |
| カテゴリ体系の見直し | ビジネス判断 |

## 9. 穴と対策（Gemini × Codex 指摘）


| 穴 | なぜ危険か | 対策 |
|---|---|---|
| Shortcuts はアプリ起動しか見ない | Web版・TV版・家族利用・バックグラウンド再生を見逃す | 証拠に「iPhoneアプリの起動のみ」と明記 |
| 家族共有サービス | 本人が使っていなくても家族が使っている | 登録時に「家族も利用」フラグを1回だけ追加 |
| 重複が必ず無駄とは限らない | Spotify と YouTube Premium は部分重複 | 知識ベースで「完全重複」と「部分重複」を分ける |
| 初回質問の回答が時間とともに変わる | 登録時は「困る」でも半年後には不要に | 年額更新前に再評価を促す |
| 解約を出しすぎるリスク | 節約アプリとして過剰に解約寄りになる | 「継続」にも証拠を出す。利用ありなら解約抑制 |
