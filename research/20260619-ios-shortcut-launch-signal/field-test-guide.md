# 実機検証ガイド：iOS Shortcut 起動シグナル連携

> 対象：`research/20260619-ios-shortcut-launch-signal/investigation.md` の §7.1「必須の実機検証ゲート」
> 実施者：ユーザー（Mac + iPhone）。この環境（Linux コンテナ）では実機検証ができないため、手順と記録欄だけ用意します。
> 目的：iPhone の Shortcut（＝iOS標準の自動化アプリ）から、自宅 Mac のローカル API へ「アプリを開いた」という利用シグナルが**確認タップなし・無音で**届くかを、本格実装の前に確かめる。
>
> 📖 読み方：専門用語は残しつつ、初出に「（＝平たい言い換え）」を添えています。
> ⚠️ この検証は**合成（ダミー）データのみ**で行います。実在の他人のデータやパスワードは使いません。

---

## 0. このガイドのゴール（go / no-go の基準）

下の **G1〜G4 がすべて OK なら go**（本格実装のステアリングに進む）。1つでも NG なら、原因を §7 のトラブルシュートで切り分け、それでも解決しなければ **no-go**（この自動化方式は見送り、別案を検討）。

| ゲート | 確かめること | 合格ライン |
|---|---|---|
| **G1** | Mac のローカル API が iPhone から見えるか | iPhone のブラウザ等から Mac の API に届く |
| **G2** | Shortcut から POST が通るか（最重要） | API が **HTTP 200** と `{"upserted":1}` を返す |
| **G3** | 無音で自動発火するか | 対象アプリを開くと、確認タップなし・通知バナーなしで Shortcut が動く |
| **G4** | 日付が端末ローカル日付で入るか | 送られた `date` が iPhone の「今日」と一致 |

## 1. 事前準備（Mac 側）

### 1-1. 開発サーバを「LAN から見える」状態で起動する
SubBuddy の Web（Next.js）は通常 `localhost` だけで待ち受けるため、**他の端末（iPhone）から届きません**。LAN（＝自宅の Wi-Fi 内）に公開して起動します。

```bash
# apps/web で実行（ポートは既定の 3000）
npm run dev -- -H 0.0.0.0
```

- `-H 0.0.0.0`（＝すべてのネットワーク面で待ち受ける指定）が肝。これを付けないと iPhone から届きません。
- 起動ログに `Network: http://192.168.x.x:3000` のような **LAN アドレス**が出ます。この IP を控える（以後 `<MAC_IP>` と表記）。
- PostgreSQL を手動起動する運用なら、先に DB を起動しておく（[[dev-env-quirks]] 参照）。

### 1-2. Mac の IP を確認する（ログに出ない場合）
```bash
ipconfig getifaddr en0   # Wi-Fi の場合。有線なら en1 等
```

### 1-3. 認証トークンを確認する
`apps/web/.env` の `USAGE_SYNC_TOKEN` の値を控える（以後 `<TOKEN>`）。
- 未設定だと API は**常に 401** を返します（仕様）。必ず実値を入れておく。
- このトークンは**秘密情報**。スクショやメモを他人に共有しない。コミットしない。

### 1-4. 送信先の実在サブスク ID を1つ用意する
`subscriptionId` は**実在するサブスクの UUID** が必要（存在しない ID だと保存に失敗します）。
- 取得方法：ブラウザで `http://localhost:3000/api/subscriptions` を開く、または Web 画面のサブスク一覧から1件選び、その `id`（UUID）を控える（以後 `<SUB_ID>`）。
- ダミー検証用に、捨ててよいテスト用サブスクを1件作っておくと安全。

### 1-5. Mac のファイアウォール
Mac のファイアウォールが有効なら、Node（开発サーバ）の着信を**許可**する。初回アクセス時に許可ダイアログが出たら「許可」。

## 2. まず手元で疎通確認（Shortcut の前に）

Shortcut を作る前に、**素の HTTP で通ることを先に確かめる**と切り分けが楽です。

### 2-1. Mac 自身から（localhost）
```bash
curl -i -X POST http://localhost:3000/api/usage/daily \
  -H "Authorization: Bearer <TOKEN>" \
  -H "Content-Type: application/json" \
  -d '{"items":[{"subscriptionId":"<SUB_ID>","date":"2026-06-19","used":true,"usageBucket":"none","source":"ios_shortcut"}]}'
```
→ **`HTTP/1.1 200` と `{"upserted":1}`** が返れば API 本体は正常。
- 401 → トークン不一致。`<TOKEN>` を再確認。
- 400 → JSON の形が違う。`date` は `YYYY-MM-DD`、`usageBucket` は `none` か。
- 500 → `<SUB_ID>` が実在しない可能性。手順 1-4 をやり直す。

### 2-2. iPhone から（LAN 越し）= G1 + G2 の先取り
iPhone を **Mac と同じ Wi-Fi** に繋ぎ、上の `curl` の `localhost` を `<MAC_IP>` に変えた URL に、後述の Shortcut で送る。まず iPhone の Safari で `http://<MAC_IP>:3000/api/subscriptions` を開いて**ページが表示されれば G1 OK**（LAN 到達確認）。

## 3. Shortcut（オートメーション）を作る

### 3-1. 送信用ショートカット本体を作る
Shortcuts アプリ →「新規ショートカット」→ アクション **「URLの内容を取得（Get Contents of URL）」** を追加し、次のように設定：

- **URL**：`http://<MAC_IP>:3000/api/usage/daily`
- **方法（Method）**：`POST`
- **ヘッダ（Headers）**：
  - `Authorization` = `Bearer <TOKEN>`
  - `Content-Type` = `application/json`
- **要求の本文（Request Body）**：`JSON` を選び、次の構造を作る
  - `items`（配列）→ 要素1つ（辞書）：
    - `subscriptionId`（テキスト）= `<SUB_ID>`
    - `date`（テキスト）= 後述の「今日の日付テキスト」
    - `used`（真偽値）= `true`（数値の 1 ではなく Boolean）
    - `usageBucket`（テキスト）= `none`
    - `source`（テキスト）= `ios_shortcut`

> 補足：本文の JSON が作りにくい場合、`テキスト` アクションに上の JSON を丸ごと書き、`URLの内容を取得` の Body を「ファイル＝そのテキスト」にする方法でも可。

### 3-2. 「今日の日付テキスト」を端末ローカル日付で作る（G4 対策）
URL取得アクションの前に：
1. **「現在の日付（Current Date）」** アクションを追加。
2. **「日付をフォーマット（Format Date）」** で、カスタム形式 `yyyy-MM-dd` を指定。
   - これで iPhone の現地時刻に基づく「今日」が `2026-06-19` 形式になる。
   - この結果を `date` に差し込む。Mac 側はこの日付をそのまま使い、解釈し直しません。

### 3-3. 動作テスト（手動実行）= G2
ショートカットを手動で1回実行 → **`{"upserted":1}` が返ればG2 OK**。
- Web 画面 or `GET /api/subscriptions/<SUB_ID>`（あるいは DB）で、対象サブスクに**今日の利用が1件**入ったことを確認。

## 4. 自動化（オートメーション）にして無音化する = G3

### 4-1. アプリ起動で発火させる
Shortcuts →「オートメーション」→「新規」→ **「App（アプリ）」** → 対象の背景再生アプリを選ぶ → **「開かれたとき（Is Opened）」** → 実行内容に 3 で作ったショートカットを指定。

### 4-2. 無音化（確認タップ・通知バナーを消す）
- **「すぐに実行（Run Immediately）」** を選ぶ（iOS 17 以降）。
- 「実行前に尋ねる（Ask Before Running）」を **OFF**（iOS 14 以降）。
- 出てくる **「実行時に通知（Notify When Run）」を OFF**（iOS 15.4 以降。Ask Before Running を OFF にすると現れる）。
- それでも届く**定期サマリ通知**は、通知設定の Shortcuts から別途 OFF。

### 4-3. 実機で確かめる = G3 + G4
1. 対象アプリを一度閉じてから**開く**。
2. **確認タップを求められない／通知バナーが出ない**ことを確認（= G3）。
3. Mac 側で、対象サブスクに**今日の利用が記録された**ことを確認（= G4：日付が今日と一致）。

## 5. 念のための追加確認（任意・推奨）

| 確認 | 方法 | 期待 |
|---|---|---|
| 冪等性（同日重複しない） | 対象アプリを同じ日に2〜3回開く | 利用は**1日1件**のまま（行が増えない） |
| recency 反映 | 1〜2日空けてから開く | 「最後に使った日」が更新される |
| ロック中の挙動 | 画面ロック状態から対象アプリを開く | 発火するか（しなくても用途上は許容。記録だけ残す） |
| Wi-Fi 外（モバイル回線） | 自宅 Wi-Fi を切って開く | 届かない想定（LAN前提）。挙動を記録 |

## 7. トラブルシュート（NG のときの切り分け）

- **iPhone から Mac に全く届かない（G1 NG）**
  - 同じ Wi-Fi か。`-H 0.0.0.0` 付きで起動したか。`<MAC_IP>` が合っているか。
  - **Wi-Fi のクライアント分離（AP isolation）**＝同じ Wi-Fi 内の端末同士を遮断する設定が ON だと届かない。ルーター設定を確認。
  - Mac のファイアウォールで Node の着信が許可されているか。
- **401 が返る**：`<TOKEN>` が `.env` の `USAGE_SYNC_TOKEN` と不一致。`Bearer ` の後ろにスペース1つ＋トークン。
- **400 が返る**：本文 JSON の形違い。`date`=`YYYY-MM-DD`、`used`=Boolean の true、`usageBucket`=`none`、`source`=`ios_shortcut` を確認。
- **500 が返る**：`<SUB_ID>` が実在しない。手順 1-4 で実在 UUID を取り直す。
- **手動実行は通るが自動で発火しない（G3 NG）**：オートメーションの「すぐに実行」が選べているか。iOS 18/26 系でアプリ起動オートメーションが不安定との報告あり（投稿レベル）。OS バージョンを記録し、別トリガ（後述）も検討。
- **プレーン HTTP が拒否される場合**：初回の**ローカルネットワーク許可**ダイアログを許可したか。どうしても通らなければ、暫定で Mac 側にローカル HTTPS リバースプロキシを立てる／Tailscale 経由にする案（investigation §5・§6B）。

## 8. この検証で分かること・分からないこと（前提の再確認）

- ✅ 分かる：**「アプリを前面で開いた」という起動シグナルが、無音で Mac に届くか**。
- ⚠️ 分からない／限界：この信号は**前面起動の検知のみ**。背景再生（ロック画面・CarPlay だけの音楽再生など）では発火しません（investigation §1・§6A）。
  - よって go でも、**「シグナルが来ない＝使っていない」と断定して解約候補に直結させない**運用が前提（recency の補助証拠として使う）。
