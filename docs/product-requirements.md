# プロダクト要求定義書（Product Requirements）

> プロジェクト名 / アプリ名：**SubBuddy**
> ドキュメント種別：永続的ドキュメント（`docs/`）
> 最終更新：2026-06-09（レコメンドエンジンをパターン判定方式に転換。steering: `.steering/20260609-quantitative-recommendation-engine/`）

---

## 1. プロダクトビジョン

インターネット経由で契約しているサブスクリプションについて、**料金・更新日・利用の性質に応じたパターン判定**をもとに、
**継続・様子見・ダウングレード検討・解約検討**を提案する、**ローカルファーストの個人用アプリ**。

中心価値は次の一点に集約される。

> **具体的な根拠（未利用・重複・安いプラン・安い競合・更新間近・高額長期）に基づいて見直すべき契約を判定し、後悔しない判断を支援する。**

**一言でいうと：自動更新に流されないための、ローカルな見直しの相棒（＝契約を見直す習慣を支える道具）。**

単なるサブスク一覧管理ツールではなく、サブスクの利用の性質（能動前面/能動背景/受動/権利保有等）を踏まえた
パターン判定で判断材料を提供し、ユーザー自身の意思決定を支援することがプロダクトの核である。

---

## 2. 解決したい課題

ユーザーは以下の問題を抱えている。

- 契約中のサブスクリプションを把握しきれない
- 使っていないサブスクに支払い続けている
- 無料体験後の課金や年額更新に気づきにくい
- 月額では安く見えても、年換算では大きな支出になっている
- 類似サービスが重複している
- 解約すべきか継続すべきか、判断材料が足りない
- iPhoneアプリの課金やWeb経由のサブスクを横断して見直したい

---

## 3. ターゲットユーザー

- 複数のサブスクリプションを契約している個人ユーザー
- iOS を利用しており、**iPhone と Mac（macOS PC）の両方を日常的に使う**ユーザー
- 支出・利用情報をクラウドに預けたくない、プライバシー意識の高いユーザー
- 自動解約のような「丸投げ」ではなく、自分で判断したいユーザー

初期 MVP は**単一ユーザーの個人利用（Mac mini をサーバとするローカル運用）**を前提とする（家族共有は対象外）。
ただし **ポストMVP では、クラウド上で多くのユーザーがアクセスできるアプリとして商品化・リリースすること**を目標とする
（段階ロードマップはセクション10.0）。すなわち **多ユーザー公開（SaaS）は MVP では作らないが、永久除外ではなくポストMVP の到達点**であり、
個人のローカル運用（Mac mini）と商品版（クラウド多ユーザー）の二段で育てる。「商品化レベルの開発技術を身につける」ことも開発目的に含む。

> 補足：ローカルサーバー（DB・API・Webダッシュボード・Worker）は Mac 上で動かす構成を想定する。
> 本書の後続および `architecture.md` 内で「Mac mini」と記載している箇所は、引き継ぎ書時点の具体的な検証環境を指すものであり、
> 要件としては **macOS が動作する Mac** であれば成立する（特定機種に限定しない）。

## 6. Appleサービス対象外リスト

ユーザーは iPhone / Mac mini を利用するが、以下の Apple サービスは**利用しないため設計から外す**。

### 6.1 明示的に対象外（設計・実装・サンプル・レコメンド対象に含めない）

- **Apple Music**（音楽は Amazon Music を利用するため）
- **Apple TV+**
- **Apple Arcade**
- **Apple One**

これに伴い、以下も不要とする。

- MusicKit 連携
- Apple TV+ 視聴履歴推定
- Apple Arcade 横断集計
- Apple One の分解評価

### 6.2 Apple関連で対象に残すもの

- App Store経由の個別アプリ課金
- iPhone上で利用しているサブスクアプリ
- **iCloud+**（ユーザーが実際に契約・利用中のため、対象に含める）
- Apple領収書メール（App Store個別アプリ課金の請求検出用途）

> 補足：ユーザーは **iCloud+ を利用しているが、Apple One は利用していない**。
> したがって iCloud+ は対象（容量ベース評価）、Apple One はセクション5・6.1のとおり対象外とする。

### 6.3 iCloud+ の扱い

iCloud+ は Screen Time では価値判断できないため、利用時間ではなく以下で判断する。

- 契約プラン
- 使用容量
- 無料枠との差分
- 家族共有の有無
- バックアップ利用の有無
- 下位プランで足りるか

iCloud+ は時間単位の利用量では評価できないため Screen Time 自動取得の対象外であり、
**容量という別軸**で評価する。容量・プラン情報の取得方法は、MVP では手動入力またはスクリーンショット読取を想定する
（時間単位の「利用量」の手動入力は行わないが、容量・プランは変動頻度が低い設定値のため例外的に手入力を許容する）。

## 7. 機能・スコープの除外事項

サブスクの「種別」ではなく、**機能・取得手法・公開範囲**として本プロダクトが行わない事項を一覧化する。
これらはプロダクト思想（セクション8）・セキュリティ方針（セクション14）に基づく除外であり、
非MVPスコープ（セクション10.2）でも重複して管理する。

- **自動解約の実行・解約代行**（更新前の判断提案は行うが、代理での解約はしない）
- **外部サービスへの自動ログイン／スクレイピング**（資格情報を保存して取得する方式）
- **外部サービスのID/パスワードの保存**
- **銀行API・クレジットカードAPIの本格連携**
- **Apple ID からの全サブスク横断取得**
- **App Store Server API によるユーザー全体の課金取得**
- **Amazon アカウントのスクレイピング**
- **詳細な iPhone Screen Time データの外部送信**（送信は集計値のみ）
- **家族全員のサブスク統合管理（家族共有）**

> 注（商品化スコープ）：「**クラウドでの多ユーザー公開（SaaS 化・商品化）**」は従来は除外事項だったが、
> 本改訂で **ポストMVP の目標**へ格上げした（セクション3・10.0）。MVP では作らないが永久除外ではない。
> ただし **自動解約・外部 ID/PW の保存・自動ログイン/スクレイピング**は、クラウド商品版でも引き続き永久に行わない。

> 注（利用量取得との関係）：利用量は iPhone Screen Time（DeviceActivity）から自動取得するため、
> サービスへのログインや資格情報の保存は不要であり、上記の「自動ログイン／スクレイピング」は**永久に対象外**とする。
> Screen Time で取得できないのは「サービス内部の詳細実績（再生回数・閲覧記事数など）」だが、
> SubBuddy の判断に必要なのは「時間単位の利用量」であり、これらは不要。
> なお、**公式 OAuth API によるトークンベースの読み取り**は「ID/パスワード保存型の自動ログイン」とは区別され、
> 将来の検討対象として残す（PW は保存しない）。

## 8. プロダクト思想（基本原則）

本プロダクトが守る 5 つの原則。詳細な根拠は `architecture.md` / `functional-design.md` に展開する。

### 8.1 ローカルファースト

支出・利用情報は機微性が高いため、まず個人利用のローカルファーストアプリとして設計する。
クラウドに支出・利用履歴を預けずに開始でき、必要になった部分だけ後から追加する。
**ポストMVP では、これを土台にクラウド多ユーザーの商品版へ拡張する**（商品化レベルの開発技術の習得も目的。セクション10.0）。

- **iPhone**：利用量を取得・集計するセンサー
- **Mac mini**：DB・API・Webダッシュボード・レコメンドWorker（個人運用の常時稼働サーバ）
- **クラウド**：MVP では使わない（**ポストMVP で多ユーザー商品版の基盤として導入**。セクション10.0）

### 8.2 半自動から始める（完全自動化しない）

全サブスクの完全自動取得は非現実的かつセキュリティ・規約・保守性の問題が大きいため行わない。
MVP は手動登録と利用量推定だけでも価値を出せる設計とする。

### 8.3 AIは判断の主役にしない

解約判断は**ルールベースのスコアリング**で行う。AIは以下の補助用途に限定し、
初期 MVP では AI 連携なしでも成立させる。

- レコメンド理由文の自然文生成
- 月次レポート文面の生成
- 領収書メール・スクリーンショットからの情報抽出
- サービス名の表記揺れ正規化

### 8.4 自動解約はしない

本アプリは解約代行アプリではない。

- **やること**：解約検討を提案する／公式の解約導線をメモ・表示する／更新日前に判断を促す
- **やらないこと**：ユーザーの代わりに自動解約する／外部アカウントのID・パスワードを保存する／自動ログインして解約処理を実行する

### 8.5 中立表現を守る（煽らない）

見直しを促す表現は中立に保ち、ユーザーを不安にさせて行動させる作り方をしない。

- **煽らない**：「今すぐ解約すべき」等の断定や、不安をあおる表現を使わない。
- **節約額を大げさに見せない**：年間節約額などは事実として控えめに示し、過大に演出しない。
- **断定せず根拠を示す**：解約を決めつけず、判定の根拠（未利用・重複・安い代替・更新間近・高額長期）を示して、判断は人に委ねる。

この原則は UI 文言・レポート文面・レコメンド理由文のすべてに適用する。

## 9. ユーザーストーリー

1. 契約中のサブスクを手動で登録し、一覧で把握したい。
2. 毎月・毎年いくら支払っているか合計額を知りたい。
3. 各サブスクの次回更新日を管理し、更新前に気づきたい。
4. 各サブスクの利用量（利用日数・時間）を、手で記録せず iPhone から自動で計測してほしい。
5. 1利用日あたりの単価を見て「割高かどうか」を判断したい。
6. 使っていないサブスクを解約候補として提示してほしい。
7. 継続／様子見／ダウングレード検討／解約検討／強い解約候補の判定理由を知りたい。
8. 解約を決めたら公式の解約導線（URL）をすぐ開きたい。
9. 利用していない Apple サービス（Music / TV+ / Arcade / One）を提案に出さないでほしい。
10. 音楽は Amazon Music を実利用サービスとして扱ってほしい。
11. 利用中の iCloud+ を容量ベースで見直したい。
12. 支出・利用情報をクラウドに預けず、手元（Mac mini）に保存したい。

## 10. MVPスコープ

### 10.0 フェーズの考え方（MVP と その後の違い）

**MVP（フェーズ1）でやること**

MVP は「価値が出る最小形」を作るフェーズである。
サブスクの**登録は手動**（見直したいものをユーザーが一度だけ登録）で行い、
料金・更新日・利用の性質（＝自分で操作して使うのか、流して使うのか等）をもとに、
アプリが状況ごとに見直しを提案する**パターン判定**（＝決まった条件に当てはめて
「継続／様子見／ダウングレード検討／解約検討／強い解約候補」を出すやり方）で判断材料を示す。
判定は「月いくら・年いくら払っているか」「同じカテゴリで重複していないか」
「もっと安いプラン・競合はないか」「更新が近いか」「高額で長く続けていないか」といった**根拠**に基づく。
ここまでを、**AI もクラウドも使わず、Mac ローカル保存**で完結させる。
**MVP の主役は「パターン判定 × ローカル保存」である。**

このうち「使っていない（＝ P1）」の判定だけは利用量の裏づけが要るため、
**iPhone の Screen Time（DeviceActivity）から利用量（利用有無・利用日数・概算時間）を自動取得し、
能動利用（前面）のサブスク（＝自分で操作して使うアプリ）に限って補強材料として使う**。
ここで取得するのは「サービスをどう使ったか（内部の中身）」ではなく、
「**どれだけ使ったか＝時間単位の利用量**」である。
利用量は手動入力しない（日々の記録を手で付けるのは現実的でないため）。
利用量がなくても、P1 以外（重複・割高・安い代替・更新間近・高額長期）のレコメンドは出る。
サービスへのログインや資格情報の保存も行わない。

**その後（フェーズ2以降）でやること**

その後のフェーズは「入力の手間をさらに減らし、判断を読みやすくする」フェーズである。
請求・契約情報の入力を、領収書メール・スクリーンショットからの情報抽出で**自動化・省力化**していく。
あわせて、ルールが出した判定に対して AI が「なぜこの判定か」の理由文や月次レポートを**自然な文章で補足**する。
基盤面では、**クラウド上で多くのユーザーがアクセスできる商品版としてのリリース**を目標に据え、
マルチテナント化・正式な認証・Worker 分離・クラウドデプロイ/運用など、**規模と利便性を広げる**拡張を行う。
これは「**商品化レベルの開発技術を身につける**」という開発目的も兼ねる。個人運用（Mac mini）と商品版（クラウド多ユーザー）は両立して育てる。

**両フェーズで一貫して変えないこと**

どのフェーズでも「**判断するのは人、アプリは材料を出すだけ**」という姿勢は変えない。
自動解約はせず、外部アカウントの ID/パスワードも持たない。iPhone から受け取るのも詳細ログではなく集計値だけ。
AI はあくまで補助で、判断の主役はルールベースのままとする。
また、どのフェーズでも、**最初の棚卸し（＝契約中のサブスクを一度すべて登録して見直す体験）を完了してもらうこと**を最重要の体験として設計する。
**クラウド商品版でも機微 PII の保護は最優先**とし、保存時暗号化や（可能なら）エンドツーエンド暗号化、
最小データ収集（iPhone は集計値のみ）でプライバシーの精神を維持する（技術詳細は `architecture.md`）。
要するに、**MVP で「パターン判定ベースの最小形」を作り、その後は「同じ価値をより楽に・より分かりやすく」**——
軸はそのままに、入力と説明を自動化していく、という違いである。

### 10.1 初期MVPで作るもの

- サブスク手動登録（見直したいサブスクの一度きりの登録）
- サブスク一覧
- 月額・年額合計
- 更新日管理
- **iPhone Screen Time（DeviceActivity）による利用量の自動取得**
  （対象アプリ/Webドメインの利用有無・利用バケット・概算利用時間を、時間単位の集計値として日別に取得）
- iOS アプリ（利用量センサー）：計測対象の選択、オンデバイス集計、Mac への集計値同期
- 解約候補スコア
- レコメンド結果表示
- 更新日前レビュー
- 1利用日あたり単価
- 対象外 Apple サービスを登録候補に出さない
- Amazon Music を音楽カテゴリの実利用サービスとして扱う
- App Store 個別アプリ課金を対象にする
- iCloud+ を容量ベースで管理（実利用中のため対象。容量は時間計測ではなくプラン/使用容量で評価）

### 10.2 初期MVPで作らないもの（非MVPスコープ）

- Apple Music 連携
- MusicKit 連携
- Apple TV+ 視聴履歴推定
- Apple Arcade 横断集計
- Apple One 分解評価
- 自動解約
- Apple ID からの全サブスク一覧取得
- Amazon 自動ログイン
- 銀行API・クレカAPI連携
- Gmail 本格連携
- 領収書メール／スクリーンショットからの情報抽出（フェーズ2）
- AI によるレコメンド理由文・月次レポート生成（フェーズ2）
- 家族共有
- クラウド多ユーザー基盤・SaaS 公開（※**ポストMVP の目標**。MVP では作らない／永久除外ではない。セクション3・10.0）

> 補足：レコメンドエンジンは**パターン判定方式**（P1〜P7）に転換した。
> iPhone Screen Time（DeviceActivity）による利用量取得は P1（使っていない）の判定手段であり、
> 能動利用（前面）のサブスクに限定して適用する。中核はパターン判定方式であり、利用量なしでも P2〜P6 のレコメンドは出る。
> 詳細は `.steering/20260609-quantitative-recommendation-engine/requirements.md` を参照。

### 10.3 iOS は Spike を経て P1 の判定手段として組み込む

iPhone Screen Time（DeviceActivity）連携は P1（使っていない）の判定手段だが、Family Controls entitlement の取得可否や
実機での認可・イベント発火など不確実性が高い。そのため、**実装前に必ず Spike（技術検証）を実施**し、
成立性を確認したうえで実装に進む。Spike は別ステアリングで管理する。

```text
.steering/[YYYYMMDD]-ios-screen-time-spike/
  requirements.md
  design.md
  tasklist.md
```

Spike で確認する主な項目：

- Family Controls entitlement の取得可否
- iPhone 実機での individual authorization
- FamilyActivityPicker による対象アプリ／Webドメインの選択
- DeviceActivityMonitor Extension のしきい値イベント発火（1分/5分/15分/30分 等）
- 端末・アプリ再起動後の挙動
- Mac へ送る集計値の粒度（詳細ログは送らない）
- App Review 上問題のない用途説明

> Spike の結果、MVP 期間内での成立が困難と判明した場合に限り、利用量取得方式の再検討（代替案の評価）を行う。

## 11. 成功条件

初期 MVP は、以下を満たしたとき成功とする。

- **初回の棚卸し（＝契約中のサブスクを一度すべて登録して見直す体験）を、最後までやり切れる**。これを最重要の成功体験に置く（MVP では完了率の数値目標は設けない）
- サブスクを手動登録できる
- 月額・年額の合計が見える
- 更新日が管理できる
- パターン判定方式（P1〜P7）でレコメンドが出る
- 継続・様子見・ダウングレード検討・解約検討・強い解約候補を、具体的な根拠付きで表示できる
- サブスクの利用の性質（`usage_type`）に応じて P1 の適用可否が切り替わる
- 受動利用・権利保有のサブスクに「使っていない」という誤ったレコメンドが出ない
- サービスカタログから選択式で登録でき、代替サービス・安いプランの情報が自動で利用される
- Apple Music / Apple TV+ / Apple Arcade / Apple One が対象外として扱われる
- Amazon Music が音楽カテゴリの実利用サービスとして扱われる
- App Store 個別アプリ課金が対象として扱われる
- iCloud+ が容量ベースで扱える（実利用中のため対象）
- データは Mac ローカルに保存される
- 利用量の手動入力を必要としない（パターン判定方式により利用量なしでもレコメンドが出る。能動前面のサブスクは DeviceActivity / Shortcuts で補強）
- 自動解約や外部アカウントのパスワード保存を行わない

## 12. 受け入れ条件（Acceptance Criteria）

| # | 条件 | 受け入れ基準 |
|---|------|-------------|
| AC-1 | サブスク登録 | 名称・料金・課金周期・更新日を入力して保存・一覧表示できる |
| AC-2 | 合計金額 | 全サブスクの月額合計・年額合計が自動計算され表示される |
| AC-3 | 利用量自動取得 | iPhone Screen Time（DeviceActivity）で対象アプリ/Webドメインの利用量（利用有無・利用バケット・概算時間）を自動取得し、日別サマリとして保存できる（手動入力を要しない） |
| AC-4 | 単価表示 | 「料金 ÷ 過去30日の利用日数」が1利用日あたり単価として表示される |
| AC-5 | スコアリング | ルールベースで解約候補スコアが計算される |
| AC-6 | 判定表示 | keep / review / consider_downgrade / consider_cancel / strong_cancel_candidate の5判定と理由が表示される |
| AC-6b | 段階的な情報提供 | 利用量集計は登録時点から開始し、登録直後でも利用に依存しない指摘（同カテゴリ重複・割高・更新間近）を表示する。利用ベース判定は観測が十分になるまで「観測中（あと N 日）」と表示し、確定後に判定へ切り替わる（`functional-design.md` §8.5） |
| AC-7 | 更新日前レビュー | 更新日が近いサブスクが抽出・提示される |
| AC-8 | Apple対象外 | Apple Music / TV+ / Arcade / One は登録候補・レコメンド対象に現れない |
| AC-9 | 解約導線 | 解約検討時に公式解約URLを表示・遷移できる（自動解約はしない） |
| AC-10 | データ保存 | すべてのデータが Mac ローカル（PostgreSQL）に保存される |
| AC-11 | 秘密情報非保存 | 外部サービスのID/パスワードを一切保存しない |
| AC-12 | 計測対象の選択と集計値送信 | ユーザーが計測対象アプリ/Webドメインを選択でき、iPhone から Mac へ送るのは詳細ログではなく集計値のみである |

## 13. 機能要件（概要）

詳細設計は `functional-design.md` に展開する。

- **サブスク管理**：CRUD、カテゴリ、重要度、解約URL、メモ
- **請求イベント管理**：請求履歴の記録（MVPは手動／フェーズ2で領収書メール・スクショ由来）
- **利用量管理（自動取得）**：iPhone Screen Time（DeviceActivity）による対象アプリ/Webドメインの
  利用量自動取得（利用有無・利用バケット・推定利用時間レンジ）と日別サマリ保存。利用量の手動入力は行わない。
- **iOS アプリ（利用量センサー）**：計測対象の選択（FamilyActivityPicker）、オンデバイス集計、Mac への集計値同期
- **集計**：月額・年額合計、利用日数集計、1利用日あたり単価
- **レコメンドエンジン**：ルールベースのスコアリングと5段階判定、判定理由
- **更新日前レビュー**：更新日が近いサブスクの抽出
- **サービスカタログ（正規化辞書）**：サービス名の表記揺れ正規化、対象/対象外フラグ、アプリBundle ID・ドメインの対応

## 14. 非機能要件（概要）

- **プライバシー/セキュリティ**：MVP はローカルファースト（支出・利用情報は Mac ローカル＝Mac mini に保存）。
  外部アカウントのID/パスワードは保存しない。iPhone から送るのは詳細ログではなく集計値のみ。
  **ポストMVP のクラウド多ユーザー版では、マルチテナント分離・正式な認証・TLS・保存時暗号化（可能なら E2E）・個情法/GDPR 配慮**で機微 PII を保護する（詳細は `architecture.md`）。
- **PII・機微データの扱い**：本プロダクトは PII を含む機微な個人データを扱うが、実データは実行時に Mac ローカルにのみ保存し、
  リポジトリには含めない。**開発時（Claude Code を含む）は実在の個人データを参照・生成・コミット・外部送信せず、合成データのみを用いる**
  （詳細は `CLAUDE.md`「PII・機微データの取り扱い」および `development-guidelines.md` を参照）。
- **可搬性**：将来の Worker 分離・クラウド連携を見据えた構造とする。
- **保守性**：ルールベースのスコアリングは設定値として外出しし、調整可能にする。
- **UI**：共通デザインシステム（Tailwind CSS）で統一感を保つ。
- **品質**：入力バリデーション、XSS 対策を行う。コード変更後はリント・型チェック・テストを実施。

詳細は `architecture.md` および `development-guidelines.md` に展開する。
