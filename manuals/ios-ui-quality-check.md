# iPhone UIの視覚・アクセシビリティ・性能確認手順

## 目的

利用者向けiPhone UIを、画面サイズ、表示設定、VoiceOver、動きの軽減、案内の開閉、共通操作色、Apple公式ボタン寸法、合成200契約の条件で確認する。

対象は`.steering/20260716-ios-main-ui/tasklist.md`の`T-13`〜`T-15`。実在の契約・金額・Apple識別子・利用実績は使わない。

## 合格基準

| 確認 | 合格条件 |
|---|---|
| 標準・大画面 | 文字切れ、重なり、画面外へ出て操作できない項目がない |
| ライト・ダーク・コントラスト | 情報が読め、状態を色だけで区別しない |
| 最大Dynamic Type | 横スクロールを要求せず、主要操作を完了できる |
| VoiceOver | 初回、ホーム、契約、見直し、設定の順序とラベルが理解できる |
| 画面説明 | 説明と`この画面について`が同時表示されず、`閉じる`をダークモードでも読める |
| Appleサインイン | 公式ボタンが高さ44pt・最大幅320ptで中央にあり、横幅を占有しすぎない |
| 44ポイント操作領域 | 主要ボタンと行を安定して選べる |
| 動きの軽減 | 有効時も情報や完了状態を失わない |
| 合成200契約 | 初期表示2秒以内、タップ・検索入力の視覚応答100ミリ秒以内を目標にする |

> ⚠ Simulatorだけで実機合格にしない。VoiceOver、操作領域、動きの軽減は実機でも確認する。

## 3. Xcodeで合成Previewを開く

○ 実在データを読み込まず、主要状態を確認する。

□ 操作

1. Xcodeで`apps/ios/SubBuddy.xcodeproj`を開く。
2. `PreviewFixtures.swift`を開く。
3. Canvasを表示する。
4. `ホーム・通常`、`ホーム・空`、`ホーム・通信失敗`を順に開く。
5. `契約詳細・最大文字`、`見直し詳細`、`契約一覧・合成200件`を順に開く。

> ✅ 通常、空、失敗、最大文字、200件の各状態が合成データだけで表示される。

## 4. 標準・大画面を確認する

○ 画面幅による切れ・重なりを確認する。

□ 操作

1. 標準画面はiPhone 17 Pro相当を選ぶ。
2. 大画面はiPhone 17 Pro Max相当を選ぶ。
3. 各端末でホーム、契約一覧・詳細・編集、支出、見直し一覧・詳細、設定を確認する。
4. 長い日本語の契約名でも、削除・保存・戻る操作が画面外へ出ないことを確認する。

> ✅ 2画面区分で、切れ・重なり・操作不能がない。

---

## 5. ライト・ダーク・高コントラストを確認する

○ 色と背景が変わっても情報を読めることを確認する。

□ 操作

1. Simulatorまたは実機をライト表示にする。
2. ホーム、契約、見直し、設定を確認する。
3. ダーク表示へ切り替え、同じ画面を確認する。
4. `設定` → `アクセシビリティ` → `画面表示とテキストサイズ` → `コントラストを上げる`を有効にする。
5. 警告、エラー、選択状態が文字・アイコン・形でも区別できることを確認する。
6. 本文と見出しがWeb版と同じゴシック体、大見出しが明朝体、金額が読みやすい等幅寄りの数値書体で表示されることを確認する。
7. `支出の内訳`で`この画面について`を開き、`閉じる`が背景へ沈まず読めることを確認する。
8. `見直し`で説明を閉じて再表示し、説明と再表示ボタンが同時に見えないことを確認する。

> ✅ 文字と操作が読め、色だけに依存した状態表示がない。

## 5-1. Appleサインインボタンの寸法を確認する

○ 小さいiPhoneでも公式ログインボタンが画面を占有しすぎず、押しやすさを維持することを確認する。

□ 操作

1. テスト用状態でAppleサインイン画面を表示する。
2. ボタンが画面中央にあり、横幅が最大320pt、高さが44ptであることをXcodeのView Debuggerまたは画面計測で確認する。
3. 最大Dynamic Typeでもボタンと説明文が重ならないことを確認する。
4. 退会前のApple再認証画面でも同じ寸法になることを確認する。

> ⚠ 実在のAppleアカウント情報や認証画面のスクリーンショットを証跡へ残さない。

> ✅ 2か所のApple公式ボタンが共通寸法で中央表示され、主要操作領域44ptを維持する。

## 6. 最大Dynamic Typeを確認する

○ 最大文字サイズでも主要操作を完了できることを確認する。

□ 操作

1. `設定` → `アクセシビリティ` → `画面表示とテキストサイズ` → `さらに大きな文字`を開く。
2. `さらに大きな文字`を有効にし、スライダーを最大にする。
3. ホームから契約一覧を開く。
4. 契約詳細から編集を開き、各項目と`保存`を確認する。
5. 見直し詳細と設定を開く。
6. 横スクロール、文字の重なり、押せない操作がないことを確認する。

> ✅ 最大文字でも、ホーム、契約、見直し、設定の主要操作を完了できる。

---

## 7. VoiceOverを確認する

○ 画面を見なくても主要情報と操作を理解できることを確認する。

□ 操作

1. 実機の`設定` → `アクセシビリティ` → `VoiceOver`を有効にする。
2. 初回説明を先頭から読み、続行ボタンまで移動する。
3. ホームで年間支出、未完了時だけ表示される次の確認項目、更新、支出、設定の順序を確認する。
4. 契約一覧から合成契約の詳細を開き、編集と削除のラベルを確認する。
5. 見直し詳細で、事実、根拠、分からないこと、選択肢、公式導線の順序を確認する。
6. 設定でデータ、同期、端末とセッション、サインアウトを確認する。
7. 同じ文言の重複読上げ、ラベルのないボタン、意味のない内部ID読上げがないことを確認する。

> ⚠ Appleアカウントや実在契約を読上げた録音・動画を残さない。

> ✅ VoiceOverで主要導線を順番どおり完了できる。

## 8. 操作領域をAccessibility Inspectorで確認する

○ 小さすぎる操作対象とラベル不足を検出する。

□ 操作

2. 対象に実行中のSimulatorまたはiPhoneを選ぶ。
3. `Audit`を実行する。
4. Hit Area、Description、Contrastの指摘を確認する。
5. ホーム、契約、支出、見直し、設定、Appleサインインで重大な指摘が0件であることを確認する。

> ✅ 44ポイント未満の主要操作領域、ラベルなし操作、重大なコントラスト問題が0件である。

## 9. 動きの軽減を確認する

○ アニメーションを減らしても状態変化を理解できることを確認する。

□ 操作

1. 実機の`設定` → `アクセシビリティ` → `動作` → `視差効果を減らす`を有効にする。
2. タブ移動、画面遷移、保存、再読込、エラーからの再試行を行う。
3. 動きが減っても、読込中・完了・失敗を区別できることを確認する。

> ✅ 動きの軽減を有効にしても、情報や操作完了状態が失われない。

## 10. 合成200契約モードを起動する

○ クラウドへ大量登録せず、一覧の安定性を確認する。

□ 操作

1. Xcodeで`Product` → `Scheme` → `Edit Scheme...`を開く。
2. `Run` → `Arguments`を開く。
3. `Arguments Passed On Launch`へ`-subbuddy-quality-200`を追加して有効にする。
4. DebugビルドでSimulatorまたは実機へ起動する。
5. ホームの件数が200件であることを確認する。
6. `契約`タブを開き、先頭から末尾までスクロールする。
7. 検索へ`合成契約 200`を入力し、対象1件へ絞られることを確認する。
8. 確認後、起動引数を無効にする。

> ⚠ この起動引数はDebugビルド専用で、クラウドへの保存・同期を行わない。Releaseの確認には使わない。

> ✅ 合成200件でクラッシュ、行の飛び、重複、検索誤動作がない。

## 11. 200契約の性能を測る

○ 初期表示と操作応答の目標を確認する。

□ 操作

### 11-1. ProfileをDebug構成にする

○ 合成200契約モードをInstrumentsでも有効にする。

□ 操作

1. Xcodeで`Product` → `Scheme` → `Edit Scheme...`を開く。
2. 左側の`Profile`を選ぶ。
3. `Info`の`Build Configuration`を一時的に`Debug`へ変更する。
4. `Arguments`を開く。
5. `Arguments Passed On Launch`の`＋`を押す。
6. `-subbuddy-quality-200`を追加し、左側のチェックを有効にする。
7. `Close`を押す。

> ⚠ `Run`側の起動引数は`Profile`へ自動では引き継がれない。Profileは通常Release構成のため、この設定をしないと合成200契約モードにならない。

> ✅ Profileで起動したホームに`200件`と表示される。

### 11-2. Time ProfilerでCPU処理を確認する

○ 起動・一覧表示・検索でCPUを長く使う処理を確認する。

□ 操作

1. Xcode上部で`SubBuddyApp`と確認用のSimulatorまたは実機を選ぶ。
2. `Product` → `Profile`を選ぶ。ショートカットは`⌘I`。
3. Instrumentsのテンプレート選択画面で`Time Profiler`を検索する。
4. `Time Profiler`を選び、`Choose`を押す。
5. 上部の対象が確認用端末の`SubBuddyApp`であることを確認する。
6. 左上の赤い録画ボタンを押す。アプリが自動で起動する。
7. ホームに`200件`と表示されるまで待つ。
8. `契約`タブを開く。
9. 一覧を先頭から末尾まで1回スクロールする。
10. 検索欄へ`合成契約 200`と入力する。
11. 検索結果を開き、契約詳細が表示されたら戻る。
12. Instrumentsへ戻り、左上の停止ボタンを押す。
13. タイムライン上で調べたい時間帯をドラッグして選択する。
14. 下部の`Call Tree`を開き、`Weight`または実行時間が大きい順に確認する。
15. 必要な場合は`Hide System Libraries`を有効にし、SubBuddyAppの処理を絞り込む。

> ⚠ Time Profilerは、どの処理がCPU時間を使ったかを調べる道具であり、タップから表示までの時間を自動判定する試験ではない。明確な待ち時間があった場合に、その区間と重いSubBuddyAppの処理を確認する。

> ✅ 起動から200件表示までがおおむね2秒以内で、タブ・検索・詳細表示に知覚できる停止がない。停止がある場合は、操作名、時間、最も重いSubBuddyAppの処理だけを記録する。

### 11-3. Animation Hitchesでスクロールの引っ掛かりを確認する

○ 画面更新が間に合わず、スクロールや画面遷移が引っ掛かる箇所を確認する。

□ 操作

1. Time Profilerの記録画面を閉じる。保存確認が出た場合、調査を続けない記録は`Don't Save`を選ぶ。
2. Xcodeへ戻り、もう一度`Product` → `Profile`を選ぶ。
3. テンプレート選択画面で`Animation Hitches`を検索する。
4. `Animation Hitches`を選び、`Choose`を押す。
5. 左上の赤い録画ボタンを押す。
6. ホームに`200件`と表示されたら`契約`タブを開く。
7. 一覧をゆっくり1往復し、次に速く1往復する。
8. 検索欄へ`合成契約 200`と入力し、結果を開いて戻る。
9. Instrumentsへ戻り、停止ボタンを押す。
10. タイムラインの`Hitches`トラックに警告マーカーがないか確認する。
11. マーカーがある場合は選択し、発生した操作と`Hitch Duration`を確認する。

> Hitch rateが表示される場合、Appleの目安では`10 ms/s以下`は良好、`25 ms/s以下`は警告、`50 ms/s以下`は重大、それを超える場合は早急な確認対象となる。画面にHitch rateがない場合は、警告マーカーの件数と最長時間を確認すればよい。[Apple公式: Understanding hitches in your app](https://developer.apple.com/documentation/xcode/understanding-hitches-in-your-app)

> ✅ 目で分かる引っ掛かりと繰り返し発生する警告マーカーがない。Hitch rateが表示される場合は10 ms/s以下である。超えた場合は、操作名、表示された指標、最長のHitch Durationだけを記録する。

### 11-4. 計測設定を元へ戻す

○ 通常のProfile設定へ戻す。

□ 操作

1. Xcodeで`Product` → `Scheme` → `Edit Scheme...`を開く。
2. `Profile` → `Arguments`で`-subbuddy-quality-200`のチェックを外す。
3. `Profile` → `Info`の`Build Configuration`を`Release`へ戻す。
4. `Close`を押す。

> ⚠ Instrumentsの`.trace`には画面上の文字や実行状況が含まれる場合がある。リポジトリへコミットせず、必要な場合もMac内だけに保存する。

> ✅ ProfileがReleaseへ戻り、合成200契約の起動引数が無効になっている。
