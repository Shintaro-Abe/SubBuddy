# iPhone Screen Time 7日連続計測の実機確認手順

## 目的

開発用権限を持つiPhoneで、Screen Timeの計測、端末内保存、Renderへの同期、見直し反映が7日連続で成立することを確認する。期間中にSubBuddyの強制終了、iPhone再起動、通信切断、同日再送、日付境界を組み込み、欠損・重複・別日への混入がないことも確認する。

7日間は圧縮しない。1日でも計測条件を満たせなかった場合は、その日を未確認として記録し、次の有効日から新しい7日間を開始する。

## 前提

- [1日計測・同期の実機確認](./ios-screen-time-measurement-sync-real-device-check.md)が合格している。
- Renderへ確認対象の`main`がデプロイされている。
- Family Controlsの開発用権限を持つSubBuddyをXcodeからiPhoneへインストール済みである。
- 同じ合成確認用アカウントでSubBuddyへサインイン済みである。
- 7日間、毎日16分以上使える確認用アプリを1つ用意できる。
- 確認用アプリは、各計測日の開始前に当日まだ使っていない。

> ⚠ 実在の契約、普段の利用履歴、アプリ名、Appleアカウント、端末ID、認証情報、スクリーンショットを証跡へ残さない。契約名と記録値は、この手順の合成値だけを使う。

---

## 1. Macで修正版の自動確認を行う

○ 7日計測を始める前に、Simulator buildとiOS全テストを完了させる。

□ 操作

MacのSubBuddyリポジトリ直下で実行する。

```bash
apps/ios/scripts/verify-main-ui.sh
```

> ⚠ `** TEST SUCCEEDED **`になるまで7日計測を開始しない。Simulator確認だけならAPI接続先の`export`は不要。

> ✅ buildとiOS全テストが失敗0件で完了する。

---

## 2. テスト開始を監査台帳とWBSへ記録する

○ 圧縮できない7日間の開始日と未完了範囲を正本へ残す。

□ 操作

1. `.audit/test-status.md`を開く。
2. `iPhone Screen Time 7日連続計測`の行を追加する。
3. 開始日、合成契約による実機確認、7日連続・再起動・通信切断・日付境界を確認することを書き、結果を`実施中`にする。
4. `wbs/wbs.yml`の7日連続計測開始タスクへ実際の開始日を反映する。
5. 実在のアプリ名、利用履歴、アカウント、識別子は書かない。

> ✅ テストを開始した日だけが記録され、完了予定を実績として扱っていない。

---

## 3. 7日間専用の合成契約を作る

○ 他の契約や利用記録と混ざらない確認対象を用意する。

□ 操作

1. SubBuddyの`契約`タブを開く。
2. `追加`を押す。
3. 次の合成値を入力する。

| 項目 | 合成値 |
|---|---|
| サービス名 | `合成7日計測` |
| カタログ候補 | `手動入力` |
| カテゴリ | `other` |
| 日本円の請求額 | `700` |
| 請求周期 | `月払い` |
| 契約状態 | `利用中` |

4. `保存`を押す。
5. `合成7日計測`の詳細から`利用状況を計測`を開く。
6. Screen Timeを許可し、確認用アプリを1つだけ選ぶ。
7. 追加操作なしで`選択中のアプリ 1件`と`計測中です。結果が育つまで日数が必要です。`が表示されることを確認する。

> ⚠ 同名の合成契約がある場合は重複登録しない。7日間は契約削除、紐付け解除、SubBuddyの再インストールを行わない。

> ✅ 内部契約IDを入力せず、合成契約1件と確認用アプリ1件を対応付けられる。

---

## 4. 7日間の確認予定を決める

○ 毎日の基本確認と、壊れやすい条件を分散して実施する。

□ 操作

次の表の順序で確認する。日付境界は6日目から7日目へ移る夜に行う。

| 日 | 確認用アプリの利用 | 追加条件 | 期待する最近30日の利用日 |
|---:|---:|---|---:|
| 1日目 | 16分以上 | 通常の計測と同期 | 1日 |
| 2日目 | 31分以上 | SubBuddyを強制終了した状態で計測 | 2日 |
| 3日目 | 16分以上 | iPhone再起動後に計測 | 3日 |
| 4日目 | 16分以上 | 通信切断中の同期失敗と復旧後再送 | 4日 |
| 5日目 | 31分以上 | 同じ日の記録を2回同期 | 5日 |
| 6日目 | 23:40ごろから16分以上 | 日付境界前の記録 | 6日 |
| 7日目 | 0:00以降に16分以上 | 日付境界後の記録と最終確認 | 7日 |

> ⚠ 開始時刻は目安。6日目分を日付が変わる前に16分以上、7日目分を日付が変わった後に16分以上使い、2日分を同じ日にまとめない。

> ✅ 7つの異なる現地日付で確認する予定になっている。

---

## 5. 毎日の基本確認を行う

○ その日の記録が端末からRenderへ届き、利用日数が1日ずつ増えることを確認する。

□ 操作

各日、追加条件を行った後に次を実施する。

1. その日の予定時間だけ確認用アプリを使う。
2. SubBuddyを開き、`合成7日計測`の`利用状況を計測`を開く。
3. `選択中のアプリ 1件`が残っていることを確認する。
4. 計測中でない場合は、Screen Time認可と通信状態を確認する。手動開始は行わず、画面を開き直しても自動再開しなければ原因を記録して不合格とする。
5. ホーム右上の`設定`から`同期`を開く。
6. `自動同期`が`オン`で、`最終同期確認`がSubBuddyを開いた後の日時へ更新されたことを確認する。
7. `今すぐ同期`を押していないことを確認する。日時が更新されない場合だけ、通信・サインイン状態を確認し、`今すぐ同期`で原因を切り分ける。
8. `見直し`タブで`再計算`を押す。現行では同期成功だけでは見直し結果を自動再計算しない。
9. `合成7日計測`を開く。
10. `最近30日の利用日`が、その日の期待日数と一致することを確認する。
11. `利用時間の目安`が当日の利用に対応する段階以上で、前日より低い値へ戻っていないことを確認する。
12. 下の記録表へ、合成値だけを記入する。

| 日 | 現地日付 | 予定分 | iPhoneのScreen Time範囲 | SubBuddyの利用時間目安 | 同期 | 累計利用日 | 結果 |
|---:|---|---:|---|---|---|---:|---|
| 1 | | 16分以上 | | | 成功/失敗 | | 合格/不合格 |
| 2 | | 31分以上 | | | 成功/失敗 | | 合格/不合格 |
| 3 | | 16分以上 | | | 成功/失敗 | | 合格/不合格 |
| 4 | | 16分以上 | | | 成功/失敗 | | 合格/不合格 |
| 5 | | 31分以上 | | | 成功/失敗 | | 合格/不合格 |
| 6 | | 16分以上 | | | 成功/失敗 | | 合格/不合格 |
| 7 | | 16分以上 | | | 成功/失敗 | | 合格/不合格 |

> ⚠ 設定アプリのScreen Timeでは確認用アプリの当日値だけを見る。アプリ名や他アプリの利用履歴を記録・撮影しない。

> ✅ 7つの異なる日付で、累計利用日が1日ずつ増える。

---

## 6. 2日目にSubBuddyの強制終了を確認する

○ SubBuddy本体が終了していてもScreen Time監視が続くことを確認する。

□ 操作

1. 2日目の利用前にSubBuddyをAppスイッチャーから上へ払って終了する。
2. SubBuddyを起動せず、確認用アプリを31分以上使う。
3. 利用後にSubBuddyを起動する。
4. `合成7日計測`の`利用状況を計測`を開く。
5. 選択1件が復元され、端末内記録が存在することを確認する。
6. 「毎日の基本確認」に従って同期し、累計利用日が2日になることを確認する。

> ✅ SubBuddy本体の強制終了中も記録が作られ、起動後に同期できる。

---

## 7. 3日目にiPhone再起動を確認する

○ iPhone再起動後も保存済み選択を復元し、計測を継続または安全に再開できることを確認する。

□ 操作

1. 3日目の利用前にiPhoneを再起動する。
2. 再起動後、最初にパスコードでロックを解除する。
3. SubBuddyを開き、`合成7日計測`の`利用状況を計測`を開く。
4. `選択中のアプリ 1件`が復元されることを確認する。
5. 追加操作なしで計測中へ自動復帰することを確認する。復帰しない場合は原因を記録して不合格とする。
6. 確認用アプリを16分以上使う。
7. 「毎日の基本確認」に従って同期し、累計利用日が3日になることを確認する。

> ⚠ 再起動直後の初回ロック解除前は、保護された端末内データを読めない場合がある。必ずパスコードで解除してから確認する。

> ✅ 再起動後も選択を失わず、当日の記録を同期できる。

---

## 8. 4日目に通信切断と再送を確認する

○ 通信できない間も端末内記録を保持し、復旧後に送信できることを確認する。

□ 操作

1. 確認用アプリを16分以上使う。
2. iPhoneを機内モードにし、Wi-Fiも無効になっていることを確認する。
3. SubBuddyを開き、起動・復帰時の自動同期を発生させる。
4. `設定`から`同期`を開く。
5. 同期失敗が表示され、アプリが終了しないことを確認する。`今すぐ同期`は押さない。
6. 同期画面を開き直し、`端末内の未送信記録`が失われていないことを確認する。
7. 機内モードを解除し、通信が戻ったことを確認する。
8. SubBuddyを一度バックグラウンドへ移し、前面へ戻して自動再試行を発生させる。
9. `最終同期確認`が更新され、同期成功したことを確認する。
10. 見直しを再計算し、累計利用日が4日になることを確認する。

> ✅ 通信失敗で記録を失わず、復旧後の再送で4日目が反映される。

---

## 9. 5日目に同日再送を確認する

○ 同じ日の記録を送り直しても、日数と利用時間を重複させないことを確認する。

□ 操作

1. 確認用アプリを31分以上使う。
2. SubBuddyを前面へ戻し、自動同期の成功を確認する。
3. 見直しを再計算し、累計利用日が5日になることを確認する。
4. 確認・復旧用の`今すぐ同期`を1回押し、同じ当日分を再送する。
5. 見直しをもう一度再計算する。
6. 累計利用日が5日のままであることを確認する。
7. 利用時間の目安が不自然に倍増せず、短い段階へ戻っていないことを確認する。

> ✅ 同日再送後も利用日は5日で、利用時間が重複加算されない。

---

## 10. 6日目から7日目の日付境界を確認する

○ 2日分の記録が、それぞれ正しい現地日付へ保存されることを確認する。

□ 操作

1. 6日目の23:40ごろから、日付が変わる前に確認用アプリを16分以上使う。
2. SubBuddyを前面へ戻して自動同期し、見直しを再計算して累計利用日が6日になることを確認する。
3. 日付が変わるまで確認用アプリを使わず待つ。
4. 7日目の0:00以降に、確認用アプリを新たに16分以上使う。
5. SubBuddyを前面へ戻して自動同期し、見直しを再計算する。
6. 累計利用日が7日になることを確認する。
7. 6日目と7日目が同じ1日へまとめられず、利用日が1日だけ増えていることを確認する。

> ⚠ iPhoneの時刻やタイムゾーンを手動変更しない。日付境界はiPhoneの現在地の時刻で確認する。

> ✅ 日付境界の前後が別日として保存され、累計利用日が7日になる。

---

## 11. 7日目の最終状態を確認する

○ 7日間の欠損・重複・監視停止がないことを判定する。

□ 操作

1. `合成7日計測`の見直し詳細を開く。
2. `最近30日の利用日`が`7日`であることを確認する。
3. 7日分の記録表がすべて合格であることを確認する。
4. 強制終了、再起動、通信切断、同日再送、日付境界の各確認が合格であることを確認する。
5. 不明な欠測、利用日の重複、利用時間の後退、主要クラッシュがないことを確認する。

> ⚠ 1日でも条件を満たさなかった場合は7日連続の合格にしない。原因と確認済み範囲を台帳へ残す。

> ✅ 7日連続で計測・同期・見直し反映が成立し、追加条件もすべて合格している。

---

## 12. 合成契約を直接削除する

○ 計測期間の終了後に合成データと計測設定を片付ける。

□ 操作

1. `契約`タブで`合成7日計測`を開く。
2. `アプリとの紐付けを解除`は押さない。
3. 契約詳細へ戻り、`この契約を削除`を押す。
4. 削除範囲と他契約への非影響を確認する。
5. `契約を削除`を押す。
6. 契約一覧を再読込し、`合成7日計測`が消えたことを確認する。
7. Web版を使う場合は契約一覧を再読込し、同じ合成契約が消えたことを確認する。

> ✅ 合成契約、計測対象との紐付け、端末内記録、同期済み利用集計、見直し結果が片付く。

---

## 13. 監査台帳とWBSを完了させる

○ 実際に確認した範囲だけを正本へ反映する。

□ 操作

1. `.audit/test-status.md`の`iPhone Screen Time 7日連続計測`を開く。
2. すべて合格した場合は結果を`合格`にする。
3. 一部が未確認または失敗した場合は`一部合格`または`不合格`にし、確認済み範囲と残作業を書く。
4. 7日分の累計利用日、強制終了、再起動、通信切断からの再送、同日再送、日付境界、後片付けの結果を記録する。
5. `wbs/wbs.yml`の7日連続計測開始・完了タスクへ実績を反映する。
6. 生ログ、実在のアプリ名・利用履歴、アカウント、識別子、画像は記録しない。

> ✅ 合否と安全な要約だけが監査台帳とWBSへ記録されている。
