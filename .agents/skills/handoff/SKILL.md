# 引き継ぎ書（HANDOFF.md）作成

新規セッションが**この1枚を読むだけで再開できる**引き継ぎ書を作る。文章は AGENTS.md の応答スタイル（平易・簡潔・事務的）に従う。

---

## 0. 不変条件（最優先）

- **反映するのは「今このセッションでやり取りした内容」だけ。** これがこのスキルの核。
  - 含めてよい：今回の会話で行った作業・決定・調査・レビュー・修正、未解決の論点、次の一手、申し送り。
  - 含めない：過去セッションの作業、git log 全体、AGENTS.md/docs に既にある恒久情報、今回触れていない機能や経緯。
  - 例外（補強のためだけ）：今回言及した事実の**裏取り**に限り、リポジトリを確認してよい（SHA・ファイルパス・テスト件数の確認など）。新情報の発掘ではなく、会話で出た内容の確認に用途を限定する。
- **PII・機微データ・シークレットを書かない**（AGENTS.md PII方針）。実メール・資格情報・実利用ログ・本番データは引き継ぎ書に載せない。必要なら参照先（ファイル名）だけ記す。
- **勝手に確定情報として断定しない。** 未確認・未検証・要判断は「要確認」と明記する。相対日付は現在日付（ユーザー提示）を使って絶対日付に直す。推測で日付を作らない。
- HANDOFF は**内部の作業継続メモ**なので、内部項番（P1 等）の使用は可。ただし共有用ドキュメントへ転記するときは [[no-internal-codes-in-docs]] に従い内容語へ。
- このスキルは**引き継ぎ書の作成のみ**。コミット・push・PR・実装変更は行わない（ユーザーが別途指示したときだけ）。

---

## 1. スコープ確定（最初に1回だけ）

セッションの会話を見返し、引き継ぐべき範囲を自分で要約してから、不確かな点だけユーザーに確認する。

- **保存先**：既定は `.steering/[YYYYMMDD]-[作業タイトル]/HANDOFF.md`。
  - 今回の作業に対応するステアリングディレクトリが既にあればそこに置く（あれば上書き更新でなく追記/改訂可否を確認）。
  - 無ければ作業内容から `[YYYYMMDD]-[作業タイトル]` を提案し、ユーザー承認後に作成。`[YYYYMMDD]` は現在日付。
  - ステアリングに紐づかない雑多なセッションなら、保存先をユーザーに確認する。
- **タイトル**：今回のセッションの主題を短く。複数テーマが混在していたら、引き継ぐ対象をユーザーに確認して絞る。

迷ったら「何を引き継ぎ、何は引き継がないか」を一言で示して合意を取る。

---

## 2. 本文の構成（既定テンプレート）

既存の `.steering/*/HANDOFF.md` に倣う。不要な節は削ってよい。各節は短文・箇条書き。

```markdown
# 引き継ぎ書（<セッションの主題>）

> 作成：<YYYY-MM-DD> / 次セッションはまずこの1枚を読めば再開できる。
> ブランチ：`<current branch>`
> 前提：<関連する既存HANDOFFや設計ファイルがあれば。無ければ省略>

---

## 1. 現在地（ひとことで）
- いま何がどこまで終わっていて、何が途中かを2〜4行で。
- 品質ゲートの状態（typecheck / lint / test / 動作確認）があれば一行で。

## 2. このセッションでやったこと
- 作業・決定・調査・レビュー・修正を、結果ベースで箇条書き。
- コード変更は「コミット済み（SHA）」か「未コミット（working tree）」かを必ず区別。
- 変更ファイルは関心ごとにまとめて列挙（パスを正確に）。
- 判断・設計の結論は「なぜそうしたか」を一行添える。

## 3. 状態・検証結果（あれば）
- テスト件数・動作確認の結果。FAIL や未確認はそのまま正直に書く。
- 食い違い・保留中の論点があれば明記。

## 4. 再開時の最初の一手
- 次セッションが最初に打つコマンド／確認手順を具体的に（番号付き）。
- 前提（DB起動など環境手順）があれば添える。

## 5. 残・別スコープ（今回やらないこと）
- 持ち越しタスク。着手条件があれば書く。

## 6. 申し送り（小）
- working tree の未追跡ファイル、別プロセスの稼働、既知の落とし穴など。
```

---

## 3. 品質チェック（書き上げたら自己点検）

- 「このセッションで出ていない情報」を混ぜていないか。混入を見つけたら削る。
- コード変更が**コミット済み/未コミット**で区別されているか。SHA・パス・件数は実物と一致するか（裏取り済みか）。
- PII・シークレットが無いか。
- 相対日付が絶対日付になっているか。
- 次セッションが**この1枚だけ**で再開できるか（外部文脈に暗黙依存していないか）。依存するならその参照先を「前提」に明記。

---

## 4. 完了時

- 保存先パスを提示する。
- 「今セッションの内容のみ反映した」旨と、含めなかった範囲（あれば）を一言添える。
- コミットはしない（必要ならユーザーに確認）。pre-commit-secret-scan はコミット時の別スキル。
